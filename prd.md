# PRD (완성)

PRD가 뭐예요? Product Requirements Document의 약자로, 제품 요구사항 문서라고 불러요.

<hl>한마디로 '우리가 뭘 만들건지' 정리한 문서죠.</hl>

## PRD에는 뭐가 들어가나요?

음식점 메뉴판처럼 정해진 형식은 없어요.

하지만 보통 이런 것들이 들어가죠:

- 문제 정의: 왜 이걸 만들어야 하는지
- 목표: 뭘 달성하고 싶은지
- 타겟 유저: 누구를 위한 건지
- 주요 기능: 뭘 할 수 있는지
- 성공 지표: 어떻게 성공을 측정할 건지

## 좋은 PRD vs 나쁜 PRD

**나쁜 PRD:** "인스타그램 같은 SNS를 만들어요"

이렇게 쓰면 개발자가 물어볼 거예요. "인스타의 뭘 따라할 건가요?"

**좋은 PRD:** "20대 여성을 위한 일상 공유 SNS를 만들어요."

- 사진은 정사각형만 가능
- 24시간 후 자동 삭제
- 친구 10명까지만 공유 가능

차이가 느껴지시나요?

## PRD 작성 꿀팁

**1. 숫자로 말해요**
"빠른 로딩" → "3초 이내 로딩"

**2. 우선순위를 정해요**
- P0: 없으면 출시 불가
- P1: 있으면 좋은데 없어도 OK
- P2: 나중에 추가해도 됨

**3. 안 할 것도 명시해요**
"이번 버전에서는 결제 기능 제외"

## PRD는 살아있는 문서

한 번 쓰고 끝? 아니에요!

유저 피드백 받으면 수정하고, 기술적 제약 발견하면 또 수정해요.

실제로 에어비앤비의 첫 PRD는 단 3페이지였대요.
지금은? 수백 페이지가 넘죠.

## Episode 1-11. PRD 없이 개발하면 생기는 참사들

만약 PRD 없이 "대충 페이스북처럼 만들어보자"라고 시작한다면? 결과는 참담할 수 있어요:

- 개발자는 단순 텍스트 포스팅만 구현
- 디자이너는 이미지, 동영상, 링크 공유까지 포함한 디자인 제작
- 기획자는 태그, 위치 정보, 공개 범위 설정까지 기대

결국 모든 작업을 엎고 다시 시작하게 됩니다. PRD가 없으면 팀원들이 각각 다른 것을 상상하며 작업하게 되거든요.

### PRD를 작성해야 하는 진짜 이유

처음에 PRD 작성이 시간 낭비처럼 느껴질 수 있어요. 하지만 실제로 PRD를 진행해보면, 오히려 시간을 절약해준다는 것을 깨닫게 되요:

- 반복되는 커뮤니케이션 비용 감소
- 재작업으로 인한 개발 시간 단축
- 명확한 목표로 인한 집중도 향상
- 팀원 간 갈등 최소화

무엇보다 PRD는 제품에 대한 깊은 고민의 결과물이에요. PRD를 작성하면서 미처 생각하지 못했던 부분들을 발견하게 되고, 더 나은 제품을 만들 수 있는 아이디어가 떠오르기도 합니다.

## Episode 1-12. PRD는 왜 1페이지를 넘으면 안 될까?

실무에서 PRD를 작성할 때 흔히 받는 피드백이 있어요: "이 PRD 너무 길어요. 핵심만 간단히 정리해주세요."

### 1. 집중력의 한계

사람의 집중력은 한계가 있어요. 특히 바쁜 스타트업 환경에서는 더욱 그렇죠. 10페이지짜리 PRD를 받은 개발자나 디자이너가 과연 끝까지 꼼꼼히 읽을까요? 대부분은 대충 훑어보고 중요한 부분을 놨치게 됩니다.

### 2. 명확한 의사결정

<hl>긴 문서는 핵심을 흘립니다. 정말 중요한 것이 무엇인지 파악하기 어려워지죠. 1페이지로 제한하면 PM은 자연스럽게 핵심만 추려내게 됩니다.</hl>

## 효과적인 1페이지 PRD 작성법

### 1. 문제 정의 (Problem Statement)

첫 문장으로 해결하려는 문제를 명확히 정의하세요. 예를 들어:

"사용자들이 주문 내역을 확인하기 위해 평균 5번의 클릭을 해야 하고, 이로 인해 30%의 사용자가 탈퇴하고 있다."

### 2. 해결책 제시 (Solution)

어떻게 해결할 것인지 한 문장으로 설명하세요:

"메인 화면에 '주문 내역' 탭을 추가하여 한 번의 클릭으로 주문 정보를 확인할 수 있도록 한다."

### 3. 주요 기능 (Key Features)

불릿 포인트로 3-5개의 핵심 기능을 나열하세요:

- 주문 내역 탭: 메인 화면 하단에 위치
- 주문 상태 필터: 진행 중/완료/취소 분류
- 주문 상세 보기: 택배 추적, 상품 정보 포함

## PRD 작성 시 피해야 할 실수들

### 1. 기술적 세부사항 나열하기

PRD는 '무엇을'에 집중해야 해요. '어떻게'는 개발자와 디자이너의 영역입니다. API 설계나 데이터베이스 스키마 같은 것은 포함하지 마세요.

❌ "MySQL 데이터베이스에 user_notification 테이블을 생성하고..."
✅ "사용자는 다양한 유형의 알림을 받을 수 있어야 한다"

### 2. 모호한 표현 사용하기

"빠르게", "효율적으로", "사용자 친화적으로" 같은 표현은 사람마다 다르게 해석될 수 있어요.

❌ "검색이 빠르게 작동해야 한다"
✅ "검색 결과는 입력 후 2초 이내에 표시되어야 한다"

### 3. 범위를 명확히 하지 않기

무엇을 포함하고 무엇을 제외할지 명시하지 않으면, 개발 중에 계속 기능이 추가됩니다.

## PRD 작성 도구와 템플릿

**추천 도구들:**

- **Notion**: 협업이 쉽고 템플릿이 다양함
- **Confluence**: 대규모 조직에서 문서 관리가 용이
- **Google Docs**: 실시간 협업과 코멘트 기능이 강력
- **Figma**: 디자인과 함께 PRD를 작성할 때 유용

## PRD 리뷰 체크리스트

PRD 작성을 완료했다면, 다음 체크리스트로 검토해보세요:

**명확성**
- ☐ 누가 읽어도 같은 이해를 할 수 있는가?
- ☐ 모호한 표현은 없는가?
- ☐ 구체적인 수치나 기준이 명시되어 있는가?

**완성도**
- ☐ 문제 정의가 명확한가?
- ☐ 해결책이 구체적인가?
- ☐ 성공 기준이 측정 가능한가?

**실행 가능성**
- ☐ 개발 리소스를 고려했는가?
- ☐ 일정이 현실적인가?
- ☐ 의존성 관계가 정리되어 있는가?
