# 내가 원한 UX와 다른 게 버그는 아니다

## "이거 버그 아닌가요?"

QA 하다 보면 "이게 내가 생각한 거랑 다른데?"라는 순간이 있어요. 근데 그게 다 버그일까요?

## 버그와 기획 누락의 차이

### 버그란?

==버그는 "기획대로 동작하지 않는 것"이에요.==

**버그 예시:**
- 기획서: "버튼 클릭 시 다음 화면으로 이동"
- 실제: 버튼 클릭해도 아무 반응 없음
- → **버그 맞음**

### 기획 누락이란?

기획서에 없었던 케이스가 발생한 것이에요.

**기획 누락 예시:**
- 기획서: 버튼 클릭 동작만 정의
- 실제: 버튼 더블클릭하면 두 번 요청됨
- → **버그 아님, 기획 누락**

## 흔한 착각들

### 1. "이게 사용성이 안 좋은데요"

```
디자이너: "이 플로우가 불편해요. 버그 아닌가요?"
개발자: "기획대로 만들었는데요?"
```

**결론:** 버그가 아니라 기획 개선 필요

### 2. "디자인이랑 달라요"

```
디자이너: "여백이 24px인데 20px로 되어있어요"
```

**이것은:**
- 개발자가 실수로 잘못 넣음 → **버그**
- 디자인 가이드가 전달 안 됨 → **커뮤니케이션 문제**
- 기술적으로 불가능 → **제약사항**

### 3. "이 경우는 생각 안 했는데"

```
사용자가 이름에 이모지를 넣음 😀
    ↓
시스템 에러 발생
```

**결론:** 버그가 아니라 예외 케이스 미고려

## 버그 vs 기획 누락 구분법

| 항목 | 버그 | 기획 누락 |
|------|------|----------|
| 기획서에 명시됨 | O | X |
| 개발자 잘못 | O | X |
| 재작업 필요 | 수정 | 기획부터 |
| 우선순위 | 높음 | 논의 필요 |

## 왜 구분이 중요할까?

### 1. 담당자가 다릅니다

- **버그**: 개발자가 수정
- **기획 누락**: PM이 기획 보완 → 개발자가 구현

### 2. 일정 영향이 다릅니다

- **버그**: 즉시 수정 가능
- **기획 누락**: 기획 → 디자인 → 개발 순서로 진행

### 3. 책임 소재가 다릅니다

버그를 기획 누락 탓으로 돌리거나, 기획 누락을 버그 취급하면 팀 갈등이 생겨요.

## 디자이너가 할 수 있는 것

### QA 전에

1. **엣지 케이스 미리 정의**
   - 데이터가 없을 때
   - 데이터가 너무 많을 때
   - 특수문자, 이모지 입력
   - 긴 텍스트

2. **에러 상황 디자인**
   - 로딩 실패
   - 네트워크 오류
   - 권한 없음

### QA 중에

1. **발견한 문제를 정확히 기록**
   - 기획서에 있던 내용인지 확인
   - 스크린샷 첨부

2. **버그/기획 누락 구분해서 리포트**
   - "기획서 3페이지 내용대로 안 됨" → 버그
   - "이 케이스는 기획서에 없었는데" → 기획 누락

## 대화 예시

### 버그인 경우

```
디자이너: "버튼 클릭이 안 돼요"
개발자: "기획서 확인해볼게요... 아, 제가 실수했네요. 수정할게요"
```

### 기획 누락인 경우

```
디자이너: "더블클릭하면 두 번 결제돼요"
개발자: "기획서에 더블클릭 방지는 없었는데요?"
PM: "아, 그 케이스를 빠뜨렸네요. 기획 추가할게요"
```

## 정리

:::callout[important]
"내가 원한 UX와 다르다"고 해서 다 버그는 아니에요.
기획서에 명시된 내용과 다르면 버그, 명시되지 않았다면 기획 누락입니다.
:::

**핵심:**
- 버그 = 기획대로 동작하지 않는 것
- 기획 누락 = 기획에 없던 케이스
- 구분해야 올바른 해결 방법을 찾을 수 있음
- QA 전에 엣지 케이스를 미리 정의하면 좋음
