# 버그 리포팅 잘하는 사람 특징

## 개발자가 좋아하는 버그 리포트

같은 버그를 발견해도 어떤 리포트는 30분 만에 해결되고, 어떤 리포트는 3일이 걸려요. 차이가 뭘까요?

## 잘하는 사람의 특징

### 1. 재현 방법을 상세히 씁니다

**안 좋은 예:**
```
로그인이 안 돼요
```

**좋은 예:**
```
1. 로그인 화면 진입
2. 이메일: test@test.com 입력
3. 비밀번호: 1234 입력
4. 로그인 버튼 클릭
5. 에러 메시지 "서버 오류" 표시

* 위 순서대로 하면 100% 재현됩니다
```

### 2. 환경 정보를 포함합니다

==개발자가 같은 환경에서 테스트해야 버그를 찾을 수 있어요.==

**포함할 정보:**
- 브라우저/앱 버전
- OS 버전
- 기기 종류
- 네트워크 상태

### 3. 스크린샷/녹화를 첨부합니다

:::callout[tip]
"백 마디 설명보다 한 장의 스크린샷"
"스크린샷보다 10초 녹화"
:::

**추천 도구:**
- 스크린샷: 기본 캡처 (Cmd+Shift+4 / Win+Shift+S)
- 녹화: QuickTime (Mac) / 게임 바 (Win)
- GIF: LICEcap, Kap

### 4. 기대 결과와 실제 결과를 구분합니다

```
[기대 결과]
로그인 버튼 클릭 시 메인 화면으로 이동

[실제 결과]
로그인 버튼 클릭 시 에러 메시지 표시 후 로그인 화면 유지
```

### 5. 추가 정보를 제공합니다

- 에러 코드/메시지 전체
- 언제부터 발생했는지
- 특정 계정에서만 발생하는지
- 우회 방법이 있는지

## 버그 리포트 템플릿

```markdown
## 제목
[화면명] 간단한 버그 설명

## 환경
- 브라우저: Chrome 120
- OS: macOS Sonoma
- 기기: MacBook Pro
- 계정: test@test.com

## 재현 방법
1.
2.
3.

## 기대 결과
(어떻게 동작해야 하는지)

## 실제 결과
(실제로 어떻게 동작하는지)

## 첨부
(스크린샷, 녹화, 에러 메시지)

## 비고
(추가 정보, 우회 방법 등)
```

## 피해야 할 것들

### 1. 감정적인 표현

```
❌ "이거 또 안 돼요!!! 진짜 답답하네요"
✅ "결제 버튼이 응답하지 않습니다"
```

### 2. 모호한 설명

```
❌ "이상해요"
✅ "버튼 색상이 #333 대신 #000으로 표시됩니다"
```

### 3. 여러 버그를 하나로 묶기

```
❌ "로그인도 안 되고 버튼도 이상하고 색도 달라요"
✅ 버그 1: 로그인 버튼 미작동
   버그 2: 결제 버튼 색상 오류
   (따로 리포트)
```

### 4. 해결책 강요

```
❌ "이거 이렇게 고치세요"
✅ "이런 현상이 발생합니다" (해결은 개발자 판단)
```

## 긴급도 표시

| 긴급도 | 기준 | 예시 |
|--------|------|------|
| Critical | 서비스 불가 | 결제 불가, 전체 장애 |
| High | 핵심 기능 장애 | 로그인 불가, 데이터 손실 |
| Medium | 기능 제한 | 일부 버튼 미작동 |
| Low | 사소한 문제 | 오타, 미세한 UI 차이 |

## 정리

:::callout[important]
좋은 버그 리포트는 개발자가 바로 버그를 재현하고 수정할 수 있게 해줘요.
상세한 재현 방법, 환경 정보, 스크린샷이 핵심입니다.
:::
