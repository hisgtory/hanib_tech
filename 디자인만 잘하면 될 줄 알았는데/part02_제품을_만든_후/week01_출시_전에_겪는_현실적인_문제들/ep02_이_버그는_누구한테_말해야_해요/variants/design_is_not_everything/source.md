## 🚨 Part 2: 실제 갈등 상황과 해결법

### Case 1: "이건 제 영역이 아니에요" 전쟁

#### 🔥 상황
```
Frontend: "API가 이상한 데이터를 주고 있어요"
Backend: "프론트에서 잘못 처리한 것 같은데요"
Frontend: "아니에요, 제가 로그 찍어봤는데..."
Backend: "그럼 제가 다시 확인해볼게요..."
[2시간 후]
Backend: "역시 프론트 문제였네요"
Frontend: "아니라니까요!" 😤
```

#### ✅ 현명한 해결법

**Step 1: 중재자 투입**
```
Tech Lead: "함께 화면 공유하면서 확인해봅시다"
```

**Step 2: 객관적 데이터로 판단**
```
1. Network 탭에서 Request/Response 확인
2. 실제 데이터와 기대값 비교
3. 코드 함께 추적
```

**Step 3: No Blame Culture**
```
"누구 잘못이 아니라, 시스템 개선 기회입니다"
```

### Case 2: 애매한 경계의 버그

#### 🤔 예시: 버튼이 두 번 클릭되는 문제

**Frontend 주장:**
```
"저는 한 번만 호출했어요. 
서버에서 중복 체크를 해야죠"
```

**Backend 주장:**
```
"클라이언트에서 버튼을 disable 
처리해야 하는 거 아닌가요?"
```

#### ✅ 솔로몬의 해결책
```markdown
## 공동 책임 원칙

### Frontend 담당:
- 버튼 즉시 disable 처리
- 로딩 상태 표시
- Debounce 적용

### Backend 담당:
- Idempotency key로 중복 방지
- 트랜잭션 처리
- 에러 응답 명확히

### 함께 하기:
- 테스트 시나리오 작성
- 모니터링 설정
```

### Case 3: 우선순위 배틀로얄

#### ⚔️ 상황
```
PM: "이 버그 당장 고쳐야 해요!"
Dev: "지금 다른 Critical 버그 처리 중인데..."
QA: "이것도 Critical인데요?"
Designer: "사용자 경험이 최악이에요!"
```

#### ✅ 우선순위 매트릭스

```
        긴급도 높음 ↑
        ┌─────────┬─────────┐
        │   P0    │   P1    │
        │ 즉시처리│ 오늘내  │
영향도  ├─────────┼─────────┤
높음 →  │   P2    │   P3    │
        │ 이번주  │ 다음주  │
        └─────────┴─────────┘

P0: 서비스 다운, 결제 오류
P1: 주요 기능 불가, 데이터 손실
P2: 부가 기능 오류, UX 불편
P3: 마이너 이슈, 개선 사항
```

## 🛠 Part 4: 버그 관리 도구 마스터하기

### Jira 버그 티켓 작성법

```yaml
Title: [결제] 카드 결제 시 중복 결제 발생
Type: Bug
Priority: P0 - Critical
Assignee: (일단 비워두거나 Tech Lead 지정)

Description:
  Summary: |
    특정 조건에서 결제가 두 번 처리되는 문제
  
  Environment:
    - Device: iPhone 13 Pro
    - OS: iOS 17.1
    - App Version: 2.3.1
    - Server: Production
    - Time: 2024-03-15 14:23 KST
  
  Steps to Reproduce:
    1. 상품 선택 후 결제 페이지 진입
    2. 카드 정보 입력
    3. 결제 버튼 빠르게 두 번 탭
    4. 결제 완료 확인
  
  Expected Result: |
    결제가 한 번만 처리됨
  
  Actual Result: |
    결제가 두 번 처리되어 두 배 금액 청구
  
  Attachments:
    - screenshot_001.png
    - console_error.txt
    - payment_log.json
  
  Additional Context:
    - 발생 빈도: 10회 중 3회
    - 영향 사용자: 약 100명 추정
    - 임시 조치: 결제 버튼 disable 처리 배포
```

## 🎓 Part 6: 실패에서 배운 교훈들

### 실패 사례 1: 대형 결제 장애

#### 📅 사건 개요
```
2023년 블랙프라이데이
- 11:00 - 첫 버그 리포트 접수
- 11:30 - "별거 아닌 것 같다"고 판단
- 14:00 - 고객 불만 폭증
- 15:00 - 긴급 점검 공지
- 18:00 - 해결 및 보상 진행
손실: 5000만원 + 브랜드 이미지 타격
```

#### 🔍 Root Cause
1. 버그 심각도 과소평가
2. 담당자 불명확 (Frontend ↔ Backend 핑퐁)
3. 에스컬레이션 지연

#### 💡 교훈과 개선
```markdown
## Post-Mortem Actions
1. P0 버그 자동 에스컬레이션 시스템
2. On-call 엔지니어 로테이션 도입
3. 15분 룰: 15분 내 담당자 미확정 시 자동 에스컬레이션
4. War Room 프로토콜 수립
```

## 🚀 Part 7: 프로 레벨 버그 헌터 되기

### Level Up Your Bug Reporting

#### 🥉 Bronze Level: 기본기
```
- 버그 발견 즉시 리포트
- 스크린샷 첨부
- 재현 방법 작성
```

#### 🥈 Silver Level: 분석력
```
- 패턴 파악 (언제/어디서 자주 발생?)
- 연관 버그 링크
- 비즈니스 임팩트 계산
```

#### 🥇 Gold Level: 해결사
```
- Root cause 가설 제시
- 해결 방안 제안
- 테스트 시나리오 작성
```

#### 💎 Diamond Level: 리더
```
- 버그 예방 시스템 구축
- 팀 교육 및 멘토링
- 프로세스 개선 주도
```

### 버그와 함께 성장하기

```markdown
## Monthly Bug Champion 🏆

### 선정 기준
- 가장 많은 Critical 버그 발견
- 가장 상세한 버그 리포트
- 가장 빠른 버그 재현

### 보상
- 팀 내 인정
- 커피 쿠폰
- 프로필에 뱃지
```
