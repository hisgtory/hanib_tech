# EP02 평가: 이 버그는 누구한테 말해야 해요

## 📊 종합 평가

| 항목 | 점수 | 코멘트 |
|------|------|--------|
| **실용성** | ⭐⭐⭐⭐⭐ | 즉시 적용 가능한 가이드 |
| **구조** | ⭐⭐⭐⭐⭐ | 논리적 흐름, 명확한 분류 |
| **톤앤매너** | ⭐⭐⭐⭐ | 친근하고 이해하기 쉬움 |

## ✅ 잘된 점

### 1. 명확한 분류 체계
버그 유형을 4가지로 깔끔하게 분류:
- 화면 이상 → 프론트엔드
- 데이터 이상 → 백엔드
- 기능 이상 → PM/기획자
- 앱 전체 문제 → 개발팀 전체

→ 초보자도 바로 판단할 수 있는 명확한 기준 제시

### 2. 실전 버그 리포트 예시
❌/✅ 비교 예시가 매우 효과적:
```
❌ "로그인이 안 돼요"
✅ [현상] [재현 방법] [환경] [비고] 구조화
```
→ 실무에서 바로 복사해서 쓸 수 있는 템플릿

### 3. 긴급도 판단 가이드
Critical/High/Medium-Low 구분과 대응 방법까지 제시
→ 신입이 가장 헤매는 "이거 얼마나 급한 거지?" 질문에 답변

### 4. 핵심 하이라이트
> "백 마디 말보다 한 장의 스크린샷이 낫습니다."

→ 기억에 남는 핵심 메시지

---

## ⚠️ 개선 제안

### 1. 대화형 도입부 추가 가능
현재 바로 설명으로 시작하는데, `conversation.md`와의 연결을 위해 도입부에 짧은 상황 설정 추가 고려

### 2. 실제 도구 스크린샷 언급
Jira, Linear 등 도구 표만 있는데, 실제 화면 예시나 사용 팁이 있으면 더 좋음

### 3. "모르겠으면" 경로 강조
가장 중요한 메시지인 "모르겠으면 QA팀/PM에게"가 중간에 위치
→ 맨 앞이나 정리 부분에서 더 강조할 수 있음

### 4. 버그 vs 피드백 구분
"이건 버그인가 피드백인가?" 구분 가이드가 있으면 더 완성도 있음

실무에서 바로 적용 가능한 **실용성 높은 콘텐츠**입니다.
버그 유형별 담당자 분류와 버그 리포트 작성법이 특히 유용합니다.
