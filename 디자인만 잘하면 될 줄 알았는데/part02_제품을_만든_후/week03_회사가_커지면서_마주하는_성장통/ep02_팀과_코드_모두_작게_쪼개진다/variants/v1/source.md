# 팀과 코드 모두 작게 쪼개진다

## 스토리텔링 버전

:::conversation
**N**: 한입님, 저희 회사 개발팀이 갑자기 여러 팀으로 쪼개졌어요. 왜 그런 거예요?

**한입**: 회사가 성장하고 있다는 좋은 신호예요! 피자로 설명해드릴게요.

**N**: 피자요?

**한입**: 네, 제프 베조스가 말한 "피자 두 판 규칙"이 있어요.
:::

### 피자 두 판 규칙

:::conversation
**한입**: 팀 크기가 피자 두 판으로 식사할 수 있는 인원(보통 6~8명)을 넘으면 비효율이 생겨요.

**N**: 왜요?

**한입**: 생각해보세요. 3명이 회의하면?

**N**: 금방 끝나요.

**한입**: 15명이 회의하면?

**N**: 아... 발언하려면 순서 기다려야 하고, 다 듣는 데만 해도 시간이...

**한입**: 맞아요! 커뮤니케이션 비용이 기하급수적으로 늘어나요.
:::

### 모놀리식 vs 마이크로서비스

:::conversation
**한입**: 코드도 마찬가지예요. 하나의 큰 코드베이스를 "모놀리식"이라고 해요.

**모놀리식:**
```
하나의 큰 건물
- 모든 부서가 한 건물에
- 한 곳 수리하면 전체 영향
- 처음엔 편함
```

**N**: 그게 왜 문제가 돼요?

**한입**: 사람이 많아지면 문제가 생겨요.
- A팀이 고친 코드가 B팀 기능을 망가뜨림
- 배포할 때 모든 팀이 다 기다려야 함
- 새 사람이 전체를 이해하기 어려움
:::

**마이크로서비스:**
```
여러 개의 작은 건물들
- 결제 서비스 / 회원 서비스 / 배송 서비스
- 각자 독립적으로 운영
- 한 건물 수리해도 다른 건물 영향 없음
```

### 왜 처음부터 쪼개지 않나요?

:::conversation
**N**: 그럼 처음부터 마이크로서비스로 하면 되지 않나요?

**한입**: 좋은 질문이에요. 하지만 처음엔 오히려 손해예요.

**초기 스타트업:**
- 개발자 3명
- 뭘 만들지도 자주 바뀜
- 빠르게 실험해야 함

이때 마이크로서비스는 배보다 배꼽이 커요. 관리할 게 너무 많아지거든요.

**N**: 그럼 언제 쪼개요?

**한입**: 보통 이런 신호가 보일 때요.
- 배포가 무서워질 때 ("이거 배포하면 저쪽이 터지지 않을까?")
- 한 팀의 변경이 다른 팀 작업을 막을 때
- 새 개발자가 코드 파악에 한 달 이상 걸릴 때
:::

### 팀이 쪼개지면 생기는 변화

**좋은 변화:**
- 각 팀이 자기 영역을 책임짐
- 독립적으로 빠르게 배포 가능
- 담당 범위가 명확해짐

**새로운 과제:**
- 팀 간 API 약속이 필요함
- 커뮤니케이션 방식이 달라짐
- 전체 그림을 그리는 사람이 필요함

:::callout[important]
팀과 코드가 쪼개지면, 기획자/디자이너도 담당 팀이 명확해져요.
"우리 팀 범위"를 잘 알아야 협업이 수월해져요!
:::

### 비개발자가 알아두면 좋은 것

:::conversation
**N**: 저도 뭔가 달라진 걸 느끼거든요. 예전엔 아무 개발자한테나 물어봤는데, 이제 "그건 저희 팀이 아니에요"라는 말을 들어요.

**한입**: 네, 그게 자연스러운 거예요! 이제는:

1. **담당 팀 파악** - 이 기능은 어느 팀 소관인지
2. **API 이해** - 팀 간에 데이터가 어떻게 오가는지
3. **배포 일정** - 각 팀의 배포 주기가 다를 수 있음

이걸 알면 협업이 훨씬 수월해져요!
:::
