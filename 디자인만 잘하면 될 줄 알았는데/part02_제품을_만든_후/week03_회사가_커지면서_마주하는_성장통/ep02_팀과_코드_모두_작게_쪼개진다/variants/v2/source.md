# 팀과 코드 모두 작게 쪼개진다 - 실무 체크리스트 버전

<!-- variant: v2, type: 실무 체크리스트 버전 -->

## 핵심 요약

조직이 커지면 팀과 코드 모두 작은 단위로 나눠야 효율적으로 움직일 수 있다. 피자 두 판 룰과 마이크로서비스 개념을 이해하면 조직 변화를 예측할 수 있다.

---

## 핵심 개념 정리

| 개념 | 의미 | 적용 대상 |
|------|------|----------|
| 피자 두 판 룰 | 한 팀이 피자 2판으로 식사 가능한 규모 (5-8명) | 팀 구성 |
| 스쿼드 | 하나의 미션을 담당하는 작은 팀 | 조직 구조 |
| 마이크로서비스 | 큰 시스템을 작은 서비스로 분리 | 기술 아키텍처 |
| API | 서비스 간 소통 규격 | 기술 연결 |

---

## 팀 분리 시점 체크리스트

### 분리가 필요한 신호
- [ ] 팀 미팅에 10명 이상 참석
- [ ] 한 사람이 여러 프로젝트 동시 담당
- [ ] 의사결정이 느려짐
- [ ] 서로 뭘 하는지 모름
- [ ] 코드 충돌이 잦음

### 분리 전 확인
- [ ] 분리 후 각 팀의 미션이 명확한가?
- [ ] 팀 간 의존성이 낮은가?
- [ ] 각 팀이 독립적으로 배포 가능한가?
- [ ] 팀장/리드가 확보되었는가?

---

## 팀 구성 체크리스트

### 이상적인 팀 구성 (5-8명)
- [ ] PM/PO 1명
- [ ] 디자이너 1명
- [ ] 프론트엔드 개발자 1-2명
- [ ] 백엔드 개발자 1-2명
- [ ] QA 1명 (선택)

### 팀 자율성 확보
- [ ] 자체적으로 우선순위 결정 가능
- [ ] 독립적으로 배포 가능
- [ ] 명확한 성공 지표 보유
- [ ] 다른 팀 승인 없이 실행 가능

---

## 코드 분리 (마이크로서비스) 체크리스트

### 분리 시점 판단
- [ ] 코드베이스가 너무 커서 이해하기 어려움
- [ ] 빌드/배포 시간이 너무 오래 걸림
- [ ] 한 부분 수정이 다른 곳에 영향
- [ ] 팀 간 코드 충돌이 잦음
- [ ] 서비스별 스케일링이 필요함

### 분리 원칙
- [ ] 도메인 기준으로 분리 (사용자, 결제, 주문 등)
- [ ] 팀 단위와 일치시키기
- [ ] API로 명확하게 연결
- [ ] 데이터베이스도 분리 고려

---

## PM/디자이너가 알아야 할 것

### 팀 분리 시 변화
| Before | After |
|--------|-------|
| 한 팀에서 모두 개발 | 여러 팀에 요청 필요 |
| 한 번에 배포 | 서비스별 배포 일정 다름 |
| 디자이너 1명이 전체 담당 | 팀별 디자이너 또는 공유 |
| 하나의 Figma 파일 | 팀별 Figma 파일 |

### 협업 방식 변화
- [ ] 다른 팀 기능 필요 시 API 요청
- [ ] 팀 간 일정 조율 필요
- [ ] 디자인 일관성 관리 (디자인 시스템)
- [ ] 팀 간 회의 정례화

---

## 팀 간 협업 체크리스트

### 의존성 관리
- [ ] 다른 팀에 필요한 것 사전 공유
- [ ] 요청 → 합의 → 개발 프로세스
- [ ] 일정 여유 확보 (다른 팀 일정 고려)
- [ ] 대안 계획 준비

### 커뮤니케이션
- [ ] 팀 간 정기 싱크 미팅
- [ ] 공유 채널 운영 (Slack 등)
- [ ] 문서화 철저히
- [ ] API 변경 사전 공지

---

## 조직 성장 단계별 체크리스트

### Stage 1: 스타트업 초기 (5-10명)
- [ ] 한 팀으로 운영
- [ ] 모놀리식 코드베이스
- [ ] 빠른 의사결정
- [ ] 유연한 역할

### Stage 2: 성장기 (10-30명)
- [ ] 팀 분리 시작
- [ ] 역할 전문화
- [ ] 프로세스 도입
- [ ] 코드 분리 검토

### Stage 3: 확장기 (30-100명)
- [ ] 여러 스쿼드 운영
- [ ] 마이크로서비스 도입
- [ ] 플랫폼 팀 구성
- [ ] 거버넌스 수립

---

## 분리 후 관리 체크리스트

### 일관성 유지
- [ ] 디자인 시스템 공유
- [ ] 코딩 컨벤션 통일
- [ ] 공통 라이브러리 관리
- [ ] 정기적인 전체 싱크

### 문제 예방
- [ ] 팀 간 중복 개발 방지
- [ ] 사일로 현상 경계
- [ ] 지식 공유 세션
- [ ] 크로스 팀 리뷰

## 실무 팁

:::callout[tip]
**"우리 팀 일 아니에요"는 위험 신호**

팀을 나눈 후 자주 듣는 말이에요.
이런 상황을 방지하려면:
- 명확한 팀 미션 정의
- 그레이존 처리 원칙 합의
- 팀 간 정기 미팅
- 상위 레벨 조율자 지정
:::

:::callout[warning]
**너무 작게 쪼개면 오히려 비효율**

마이크로서비스의 함정:
- 서비스 간 통신 복잡도 증가
- 트랜잭션 관리 어려움
- 모니터링/디버깅 복잡
- 팀 간 조율 비용 증가

→ 적정 수준을 찾는 것이 핵심
:::
