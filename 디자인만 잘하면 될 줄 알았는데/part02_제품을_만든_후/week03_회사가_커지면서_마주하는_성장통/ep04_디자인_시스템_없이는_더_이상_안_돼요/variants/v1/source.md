# 디자인 시스템 없이는 더 이상 안 돼요

## 스토리텔링 버전

:::conversation
**N**: 한입님, 저희 서비스 화면이 50개가 넘어갔는데요... 버튼 스타일이 다 달라요. 이게 같은 서비스 맞나 싶어요.

**한입**: 아, 디자인 시스템이 필요한 시점이 온 거네요!

**N**: 디자인 시스템이요?

**한입**: 레고 블록 아시죠? 그거랑 비슷해요.
:::

### 디자인 시스템 = UI 레고 블록

:::conversation
**한입**: 레고로 뭘 만들든 같은 블록을 조합하잖아요. 성을 만들든 자동차를 만들든요.

**N**: 네, 블록 종류는 정해져 있고 조합만 다르죠.

**한입**: 디자인 시스템도 그래요!

**블록 = 컴포넌트:**
- 버튼 (Primary, Secondary, Ghost)
- 인풋 필드
- 카드
- 모달, 토스트...

이 "블록들"을 조합해서 모든 화면을 만드는 거예요.
:::

### 왜 필요해졌을까요?

**사람이 늘었어요:**
```
디자이너 1명: 머릿속에 다 있음
디자이너 5명: 가끔 물어보면 됨
디자이너 20명: 문서 없으면 다 다르게 만듦
```

**화면이 늘었어요:**
```
화면 10개: 하나씩 디자인 가능
화면 100개: 일관성 유지 불가능
```

**속도가 필요해요:**
```
❌ 버튼 디자인부터 매번 새로
✅ 미리 만들어둔 버튼 가져다 쓰기
```

:::callout[important]
디자인 시스템 = 일관성 + 속도 + 협업
세 마리 토끼를 다 잡는 거예요!
:::

### 디자인 시스템에 뭐가 들어가요?

:::conversation
**N**: 구체적으로 뭘 정리하는 거예요?

**한입**: 크게 세 가지 층이 있어요.

**1. 기초 (Foundation)**
- 색상: Primary #3B82F6, Error #EF4444...
- 타이포: 제목 24px Bold, 본문 16px Regular...
- 간격: 4px, 8px, 16px, 24px...
- 모서리: 4px, 8px, 전체 둥글게...

**2. 컴포넌트 (Components)**
- 버튼들
- 인풋 필드들
- 카드들
- 모달, 토스트...

**3. 패턴 (Patterns)**
- 폼은 이렇게 배치해요
- 리스트는 이렇게 구성해요
- 에러는 이렇게 표시해요

**N**: 아, 레벨이 있군요!
:::

### 모두에게 좋은 이유

**디자이너에게:**
- 기본 컴포넌트 매번 새로 안 그려도 됨
- 핵심 UX에 집중할 시간 확보
- 새 디자이너 온보딩 쉬움

**개발자에게:**
- 컴포넌트 재사용으로 속도 향상
- "이 버튼 어떻게 생겼어요?" 안 물어도 됨
- 한 곳 수정하면 전체 반영

**사용자에게:**
- 익숙한 UI = 학습 비용 감소
- 예측 가능한 경험
- 버그/불일치가 적음

### 흔한 실수

**1. 너무 일찍 만들기**
```
❌ 서비스 출시도 전에 디자인 시스템부터
✅ 어느 정도 패턴이 보인 후 정리
```

**2. 너무 상세하게 만들기**
```
❌ 모든 케이스를 다 정의하려고 함
✅ 80%를 커버하는 핵심만 먼저
```

**3. 만들고 안 쓰기**
```
❌ 문서만 있고 실제론 각자 만듦
✅ 실제로 쓰이는지 모니터링
```
