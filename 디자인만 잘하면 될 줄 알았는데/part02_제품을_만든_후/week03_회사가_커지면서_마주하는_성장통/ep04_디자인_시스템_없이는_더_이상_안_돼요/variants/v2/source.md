# 디자인 시스템 없이는 더 이상 안 돼요 - 실무 체크리스트 버전

<!-- variant: v2, type: 실무 체크리스트 버전 -->

## 핵심 요약

디자인 시스템은 UI 컴포넌트, 스타일 가이드, 사용 원칙을 정리한 체계다. 제품이 커지고 팀이 늘어날수록 일관성과 효율성을 위해 필수가 된다.

---

## 디자인 시스템 구성 요소

| 구성 요소 | 설명 | 예시 |
|----------|------|------|
| 토큰 (Tokens) | 가장 작은 단위의 스타일 값 | 색상, 폰트, 간격 |
| 컴포넌트 | 재사용 가능한 UI 요소 | 버튼, 인풋, 카드 |
| 패턴 | 컴포넌트 조합 방식 | 폼, 모달, 네비게이션 |
| 원칙 | 사용 가이드라인 | 언제 어떤 컴포넌트를 쓸지 |

## 디자인 시스템 도입 시점 체크리스트

### 도입이 필요한 신호
- [ ] 같은 컴포넌트가 화면마다 다르게 보임
- [ ] 디자이너마다 버튼 스타일이 다름
- [ ] 개발자가 매번 새로 만듦
- [ ] 디자인 리뷰에서 "일관성" 피드백 반복
- [ ] 신규 화면 제작 시간이 오래 걸림

### 도입 전 확인
- [ ] 담당자/오너가 정해졌는가?
- [ ] 시간과 리소스가 확보되었는가?
- [ ] 현재 UI 인벤토리가 정리되었는가?
- [ ] 팀의 공감대가 형성되었는가?

- [ ] 마진/패딩 체계
- [ ] 컴포넌트 간 간격

## 실무 팁

:::callout[tip]
**작게 시작하세요**

처음부터 완벽한 시스템을 만들려 하지 마세요.
- 가장 많이 쓰이는 5개 컴포넌트부터
- 핵심 색상과 폰트만 먼저
- 쓰면서 개선하기

→ 80/20 법칙: 20%가 80% 커버
:::

:::callout[warning]
**디자인 시스템 ≠ 컴포넌트 라이브러리**

흔한 오해:
- 컴포넌트만 있으면 끝? X
- 사용 원칙과 가이드 필수
- 문서화 없으면 아무도 안 씀
- 거버넌스 없으면 파편화

→ 시스템은 컴포넌트 + 원칙 + 프로세스
:::
