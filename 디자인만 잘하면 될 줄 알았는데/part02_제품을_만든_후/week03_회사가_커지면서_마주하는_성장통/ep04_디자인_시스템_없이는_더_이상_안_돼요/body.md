# 디자인 시스템 없이는 더 이상 안 돼요

## "이 버튼, 우리 버튼 맞아요?"

서비스가 커지면서 화면이 수십 개가 됐어요. 그런데 보니까 버튼 스타일이 화면마다 달라요. 색도 다르고, 크기도 다르고, 모서리도 다르고... 같은 서비스 맞아요?

## 디자인 시스템이란?

### UI 부품의 레고 블록

```
레고: 블록 조합으로 다양한 것을 만듦
디자인 시스템: UI 컴포넌트 조합으로 다양한 화면을 만듦
```

==디자인 시스템은 반복 사용되는 UI 요소들을 모아놓은 것이에요.==

### 무엇이 포함될까요?

```
1. 기초 (Foundation)
   - 색상 (Primary, Secondary, Error...)
   - 타이포그래피 (제목, 본문, 캡션...)
   - 간격 (4px, 8px, 16px...)
   - 그림자, 모서리 둥글기...

2. 컴포넌트 (Components)
   - 버튼 (Primary, Secondary, Ghost...)
   - 인풋 필드
   - 카드
   - 모달, 토스트...

3. 패턴 (Patterns)
   - 폼 레이아웃
   - 리스트 구조
   - 네비게이션
```

## 왜 필요해졌을까요?

### 1. 사람이 늘어났어요

```
1명: 머릿속에 다 있음
5명: 가끔 물어보면 됨
20명: 문서 없으면 다 다르게 만듦
```

### 2. 화면이 늘어났어요

```
화면 10개: 하나씩 디자인 가능
화면 100개: 일관성 유지 불가능
```

### 3. 속도가 필요해요

```
❌ 버튼 디자인부터 매번 새로
✅ 미리 만들어둔 버튼 가져다 쓰기
```

:::callout[important]
디자인 시스템은 일관성 + 속도 + 협업을 위한 것이에요.
:::

## 디자인 시스템의 장점

### 디자이너 입장

```
✓ 기본 컴포넌트를 매번 새로 만들 필요 없음
✓ 핵심 UX에 집중할 시간 확보
✓ 일관된 디자인 유지 가능
✓ 새 디자이너 온보딩이 쉬움
```

### 개발자 입장

```
✓ 컴포넌트 재사용으로 개발 속도 향상
✓ "이 버튼 어떻게 생겼어요?" 질문 안 해도 됨
✓ 디자인 변경 시 한 곳만 수정하면 전체 반영
```

### 사용자 입장

```
✓ 익숙한 UI로 학습 비용 감소
✓ 예측 가능한 경험
✓ 버그/불일치가 적음
```

## 디자인 시스템 만들기

### 1. 현재 상태 파악

```
기존 화면들을 캡처해서
- 버튼이 몇 종류인지
- 색상이 몇 가지인지
- 폰트 스타일이 몇 개인지
정리해보기
```

### 2. 정리하고 줄이기

```
버튼이 15종류?
→ 비슷한 것 합치기
→ 꼭 필요한 5종류로 정리
```

### 3. 규칙 만들기

```
이 버튼은 언제 써요?
- Primary: 페이지의 주요 행동
- Secondary: 보조 행동
- Ghost: 덜 중요한 행동
```

### 4. 문서화하기

```
컴포넌트 이름, 용도, 사용 예시
Do & Don't 예시
```

### 5. 코드와 연결하기

```
디자인 파일의 컴포넌트
= 개발 코드의 컴포넌트

같은 이름, 같은 속성
```

## 실제 도구들

### 디자인 도구

```
Figma: 컴포넌트, 변수, 스타일
Zeplin, Storybook: 디자인-개발 연결
```

### 개발 도구

```
React Component Library
Vue Component Library
Design Tokens
```

:::callout[tip]
디자인 토큰: 색상, 간격 등을 코드와 디자인에서 같이 쓰는 값
예: --color-primary: #3B82F6
:::

## 디자인 시스템 운영

### 처음부터 완벽할 필요 없어요

```
1단계: 색상, 타이포 정리
2단계: 자주 쓰는 컴포넌트 5개
3단계: 점진적으로 확장
```

### 담당자가 필요해요

```
❌ 아무도 안 관리하면 → 금방 어질러짐
✅ 담당자/팀이 있으면 → 지속적으로 발전
```

### 커뮤니케이션이 핵심

```
새 컴포넌트 추가됐어요!
이 컴포넌트 쓰는 법 바뀌었어요!

→ 팀에 공유 필수
```

## 흔한 실수

### 1. 너무 일찍 만들기

```
❌ 서비스 출시도 전에 디자인 시스템부터
✅ 어느 정도 패턴이 보인 후 정리
```

### 2. 너무 상세하게 만들기

```
❌ 모든 케이스를 다 정의하려고 함
✅ 80%를 커버하는 핵심만 먼저
```

### 3. 만들고 안 쓰기

```
❌ 문서만 있고 실제론 각자 만듦
✅ 실제로 쓰이는지 모니터링
```

## 정리

:::callout[important]
디자인 시스템은 조직이 커지면 필수가 돼요.
처음부터 완벽하게 말고, 점진적으로 만들어가세요.
:::

**핵심:**
- 디자인 시스템 = UI 컴포넌트의 레고 블록
- 일관성, 속도, 협업을 위해 필요해요
- 기초(색상, 타이포) → 컴포넌트 → 패턴 순서로
- 담당자/팀이 있어야 지속 가능해요
- 점진적으로 만들고, 실제로 쓰이게 하세요
