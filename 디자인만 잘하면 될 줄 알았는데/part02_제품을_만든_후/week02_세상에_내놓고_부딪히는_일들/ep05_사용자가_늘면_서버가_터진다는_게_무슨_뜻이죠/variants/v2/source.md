# 사용자가 늘면 서버가 터진다는 게 무슨 뜻이죠 - 실무 체크리스트 버전

<!-- variant: v2, type: 실무 체크리스트 버전 -->

## 핵심 요약

서버에는 동시 처리 용량 한계가 있다. 트래픽이 급증하면 서버가 느려지거나 다운될 수 있으므로, 사전 대비와 모니터링이 필수다.

---

## 서버 관련 용어 정리

| 용어 | 의미 | 비유 |
|------|------|------|
| 트래픽 | 서버로 들어오는 요청량 | 식당에 오는 손님 수 |
| 서버 | 요청을 처리하는 컴퓨터 | 주방 |
| 스케일링 | 용량 늘리기 | 주방 확장 |
| 로드밸런서 | 트래픽 분산 | 대기표 시스템 |
| 캐시 | 자주 쓰는 데이터 저장 | 미리 만들어둔 음식 |

---

## 트래픽 급증 상황 체크리스트

### 예상 가능한 상황
- [ ] 마케팅 캠페인 시작
- [ ] 언론 보도 / 바이럴
- [ ] 시즌 이벤트 (연말, 할인 행사)
- [ ] 새 기능 출시
- [ ] TV 광고 송출

### 예상 불가능한 상황
- [ ] 갑작스러운 바이럴 (SNS, 커뮤니티)
- [ ] 경쟁사 서비스 장애
- [ ] 인플루언서 언급
- [ ] 뉴스 보도

---

## 출시 전 서버 점검 체크리스트

### 개발팀 확인 사항
- [ ] 예상 동시 접속자 수 산정
- [ ] 현재 서버 용량으로 감당 가능한지 확인
- [ ] 스케일링 계획 수립 (수동/자동)
- [ ] 부하 테스트 완료
- [ ] 장애 대응 플레이북 준비

### PM/기획자 확인 사항
- [ ] 마케팅 일정과 서버 증설 일정 동기화
- [ ] 예상 트래픽 규모 개발팀에 공유
- [ ] 장애 발생 시 대응 시나리오 준비
- [ ] 고객 공지 문구 사전 작성

---

## 모니터링 지표 체크리스트

### 필수 모니터링 지표
- [ ] CPU 사용률 (80% 이상 주의)
- [ ] 메모리 사용률
- [ ] 응답 시간 (Latency)
- [ ] 에러율 (Error Rate)
- [ ] 동시 접속자 수

### 알림 설정
- [ ] CPU 80% 초과 시 알림
- [ ] 에러율 1% 초과 시 알림
- [ ] 응답 시간 2초 초과 시 알림
- [ ] 서버 다운 시 즉시 알림

---

## 장애 단계별 대응 체크리스트

### Level 1: 느려짐
증상: 페이지 로딩 5초 이상

- [ ] 서버 상태 확인
- [ ] 불필요한 요청 줄이기
- [ ] 캐시 활성화 확인
- [ ] CDN 상태 확인

### Level 2: 간헐적 에러
증상: 일부 요청 실패, 새로고침 시 정상

- [ ] 에러 로그 확인
- [ ] 서버 스케일 아웃 검토
- [ ] DB 병목 확인
- [ ] 외부 API 상태 확인

### Level 3: 서비스 다운
증상: 모든 요청 실패

- [ ] 즉시 비상 연락망 가동
- [ ] 서버 재시작/롤백 검토
- [ ] 사용자 공지 발송
- [ ] 장애 상황 실시간 공유

---

## 장애 대응 커뮤니케이션 체크리스트

### 내부 커뮤니케이션
- [ ] 장애 인지 즉시 슬랙/카톡 공유
- [ ] 담당자 호출 (개발, 인프라)
- [ ] 5분마다 상황 업데이트
- [ ] 복구 완료 시 전체 공유

### 외부 커뮤니케이션
- [ ] 장애 공지 페이지 준비
- [ ] SNS 안내 (접속 불가 사과)
- [ ] 고객센터 스크립트 공유
- [ ] 복구 완료 안내

---

### 아키텍처 관점
- [ ] 단일 장애점(SPOF) 제거
- [ ] 오토 스케일링 설정

- [ ] 인시던트 리뷰 정례화

## 실무 팁

:::callout[tip]
**마케팅 일정은 개발팀과 공유하세요**

"다음 주에 푸시 보내요" 한 마디가 서버를 살립니다.
- TV 광고 → 최소 1주 전 공유
- 대규모 이벤트 → 최소 2주 전 공유
- 갑작스러운 바이럴 예상 → 실시간 공유
:::

:::callout[warning]
**"서버 늘리면 되죠?"는 간단한 문제가 아닙니다**

서버 증설에는:
- 비용 증가 (월 수백만원~)
- 설정 시간 필요 (수 시간~수일)
- 아키텍처 변경 필요할 수도

→ 사전 계획이 중요합니다
:::
