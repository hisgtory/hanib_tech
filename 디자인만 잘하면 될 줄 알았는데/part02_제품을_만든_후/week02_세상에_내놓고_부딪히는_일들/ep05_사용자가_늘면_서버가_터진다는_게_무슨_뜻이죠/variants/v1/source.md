# 사용자가 늘면 서버가 터진다는 게 무슨 뜻이죠

## 스토리텔링 버전

:::conversation
**N**: 한입님, 뉴스 보셨어요? 그 앱 대박 나서 서버 터졌대요. 근데 서버가 터진다는 게 뭐예요? 진짜 폭발해요?

**한입**: ㅎㅎ 진짜 폭발하는 건 아니고요, 비유적인 표현이에요. 식당으로 비유해볼게요.

**N**: 식당이요?

**한입**: 네! 서버는 식당의 주방이라고 생각하면 돼요.
:::

### 서버 = 주방

:::conversation
**한입**: 평소에 손님 10명 오는 작은 식당이 있어요. 요리사 1명이면 충분하죠.

그런데 TV에 나와서 갑자기 손님 1,000명이 몰리면?

**N**: 아... 요리사 한 명으로는 못 감당하겠네요.

**한입**: 맞아요! 그게 "서버가 터지는" 거예요.
- 주문이 밀리고 (요청 대기)
- 음식이 안 나오고 (응답 지연)
- 결국 손님들이 나가버려요 (타임아웃)
:::

### 왜 처음부터 크게 안 만들어요?

:::conversation
**N**: 그럼 처음부터 요리사 100명 쓰면 안 돼요?

**한입**: 좋은 질문이에요! 근데 문제가 있어요.

1. **비용**: 요리사 100명 월급 감당 가능해요?
2. **불확실성**: 진짜 손님 1,000명 올지 몰라요
3. **낭비**: 안 오면 99명은 놀고 있는 거예요

그래서 보통 "필요할 때 늘리는" 방식을 써요. 이걸 "스케일링"이라고 해요.
:::

### 스케일링의 두 가지 방법

**1. 스케일 업 (Scale Up) - 더 좋은 장비로**

```
요리사 1명 → 슈퍼 요리사 1명
(더 빠르고 능력 좋은 사람으로 교체)

서버 1대 → 더 좋은 서버 1대
(CPU, 메모리 업그레이드)
```

장점: 단순함
단점: 한계가 있음 (세상에서 제일 좋은 요리사도 한계가 있어요)

**2. 스케일 아웃 (Scale Out) - 더 많은 장비로**

```
요리사 1명 → 요리사 10명
(같은 수준 요리사를 더 고용)

서버 1대 → 서버 10대
(같은 서버를 더 추가)
```

장점: 무한 확장 가능
단점: 복잡함 (10명이 어떻게 협력할지 정해야 해요)

:::callout[tip]
요즘은 "오토 스케일링"이라고 해서, 손님이 많아지면 자동으로 요리사(서버)를 더 부르는 시스템이 있어요. 클라우드의 장점이죠!
:::

### 서버가 터지기 전 조짐

:::conversation
**N**: 터지기 전에 알 수 있는 방법이 있어요?

**한입**: 네! 모니터링을 통해 미리 알 수 있어요.

**위험 신호들:**
- CPU 사용률 80% 이상
- 메모리 부족 경고
- 응답 시간이 점점 느려짐
- 에러율 증가

**N**: 그럼 그 신호 보이면 어떻게 해요?

**한입**: 스케일 아웃하거나, 캐싱을 추가하거나, 코드를 최적화하거나... 방법은 여러 가지예요.
:::

갑자기 터지면 늘릴 시간도 없어요!
