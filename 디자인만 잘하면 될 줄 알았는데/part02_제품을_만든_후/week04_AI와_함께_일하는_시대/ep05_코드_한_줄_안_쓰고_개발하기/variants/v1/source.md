# 코드 한 줄 안 쓰고 개발하기 - 스토리텔링 중심 버전

<!-- variant: v1, type: 스토리텔링 중심 버전 -->

:::conversation
N: 한입님, "노코드로 앱 만들었어요"라는 글을 봤는데... 진짜예요?

한입: 네, 가능해요. 요즘은 Bubble, Webflow, Framer 같은 도구로 실제 서비스를 만들 수 있어요.

N: 그러면 개발자 없이도 앱을 만들 수 있는 건가요?!

한입: 어느 정도는요. 근데 **만능은 아니에요**. 정확히 뭘 할 수 있고 뭘 못 하는지 알아야 해요.
:::

## 노코드로 할 수 있는 것들

:::conversation
한입: 노코드로 실제로 만들 수 있는 것들을 보여드릴게요.

N: 네, 궁금해요!

한입: **랜딩 페이지**는 기본이고요. Webflow나 Framer로 진짜 예쁘게 만들 수 있어요.

N: 그건 홈페이지 빌더랑 비슷한 거 아니에요?

한입: 그것보다 훨씬 자유로워요. 디자이너가 원하는 대로 **픽셀 단위** 커스터마이징이 가능해요.

N: 오... 앱은요?

한입: Bubble로는 **웹 앱**을 만들 수 있어요. 회원가입, 로그인, 데이터 저장... 다 돼요.

N: 진짜요?
:::

:::callout[tip]
노코드로 만들 수 있는 것들:
- 랜딩 페이지, 마케팅 웹사이트
- 간단한 웹 앱 (CRUD 기반)
- 내부용 관리자 페이지
- 프로토타입, MVP
- 자동화 워크플로우 (Zapier, Make)
:::

## 노코드의 한계

:::conversation
N: 그러면 왜 개발자가 필요한 거예요? 다 노코드로 하면 되잖아요.

한입: 여기서 현실적인 얘기를 해야 해요.

N: 뭐가 안 되는 건데요?

한입: 첫째, **네이티브 앱**은 힘들어요. 앱스토어에 올라가는 그런 앱이요.

N: 아... 웹 앱만 되는 거군요.

한입: 둘째, **복잡한 로직**도 한계가 있어요. 예를 들어 실시간 채팅, 결제 시스템, 복잡한 알고리즘... 이런 건 노코드만으로 어려워요.

N: 그리고요?

한입: 셋째, **확장성** 문제가 있어요. 사용자 100명일 땐 괜찮은데, 10만 명이 되면 느려지거나 비용이 폭증할 수 있어요.
:::

:::callout[warning]
노코드의 현실적 한계:
- 네이티브 앱 개발 불가 (웹 앱만 가능)
- 복잡한 비즈니스 로직 구현 어려움
- 확장성(Scalability) 제한
- 플랫폼 종속 (이사하기 어려움)
- 미세한 커스터마이징 한계
:::

## 그래서 어떻게 활용해야 할까?

:::conversation
한입: 노코드는 **시작점**으로 쓰는 게 좋아요.

N: 시작점이요?

한입: 아이디어가 있을 때, 바로 개발팀에 "이거 만들어주세요" 하면 시간도 오래 걸리고 비용도 많이 들잖아요.

N: 그렇죠...

한입: 대신 노코드로 **빠르게 MVP**를 만들어서 시장 반응을 먼저 보는 거예요.

N: 아, 린 스타트업에서 배운 거랑 연결되네요!

한입: 정확해요! **시장 검증용**으로 노코드를 쓰고, 반응이 좋으면 그때 제대로 개발하는 거죠.
:::

## PM/디자이너가 노코드를 알아야 하는 이유

:::conversation
N: 저는 개발자가 아닌데, 노코드를 배워야 하나요?

한입: 배울 필요까지는 없지만, **뭘 할 수 있는지**는 알아두면 좋아요.

N: 왜요?

한입: "이 아이디어, 노코드로 빠르게 검증해보면 어떨까요?"라고 제안할 수 있잖아요.

N: 아, 개발 리소스 안 쓰고도 뭔가 해볼 수 있다는 걸 알면요?

한입: 맞아요. 특히 **내부용 도구**는 노코드로 충분한 경우가 많아요. 어드민 페이지라든지, 팀 대시보드라든지...

N: 그런 걸 개발팀에 매번 부탁하면 미안하기도 하고...

한입: 그래서 요즘은 PM이나 디자이너가 직접 Notion DB나 Airtable로 간단한 도구를 만들기도 해요.
:::

:::callout[tip]
PM/디자이너가 노코드로 할 수 있는 것들:
- 간단한 프로토타입 (Framer, Figma 프로토타입)
- 팀 내부용 대시보드 (Notion, Airtable)
- 설문/피드백 수집 도구 (Typeform, Tally)
- 자동화 워크플로우 (Zapier, Make)
:::

## 노코드 vs 로우코드 vs 코드

:::conversation
N: 로우코드라는 것도 있던데, 노코드랑 뭐가 다른 거예요?

한입: 쉽게 말하면 이래요.

N: 네?

한입: **노코드**는 코드 한 줄도 안 써요. 완전 비개발자용이에요.

N: 그럼 로우코드는요?

한입: **로우코드**는 기본은 드래그앤드롭인데, 필요하면 코드도 좀 써요. 개발자가 더 빠르게 만들 때 쓰죠.

N: 그리고 **풀 코드**는 처음부터 다 코드로 짜는 거고요?

한입: 맞아요! 자유도가 높아질수록 코드가 많아지는 거예요.
:::

| 구분 | 코드 필요 | 자유도 | 학습 난이도 | 적합한 상황 |
|------|----------|--------|------------|------------|
| 노코드 | X | 낮음 | 쉬움 | MVP, 빠른 검증 |
| 로우코드 | 약간 | 중간 | 보통 | 내부 도구, 빠른 개발 |
| 풀 코드 | 전부 | 높음 | 어려움 | 복잡한 서비스 |

:::yama
노코드는 "개발자 대체"가 아니라
"검증 속도 향상" 도구예요.

아이디어를 빠르게 시장에 던져보고,
반응을 보고, 판단하고, 다시 수정하고.

완벽한 제품을 만들기 전에
"이거 진짜 필요한가?"를 먼저 확인하세요.
노코드는 그 질문에 답하는 가장 빠른 방법이에요.
:::
