# QA도 AI가 대신해줄 수 있을까 - 스토리텔링 중심 버전

<!-- variant: v1, type: 스토리텔링 중심 버전 -->

:::conversation
N: 한입님, AI가 버그도 찾아준다는 게 사실이에요?

한입: 어느 정도는 사실이에요. 근데 **완전 자동화**는 아니에요.

N: QA 담당자가 필요 없어지는 건 아니라는 거죠?

한입: 네, 오히려 QA 담당자가 **더 효율적으로** 일할 수 있게 도와주는 거예요.
:::

## AI가 QA에서 할 수 있는 것

:::conversation
한입: 먼저 AI가 실제로 도움을 주는 영역을 알려드릴게요.

N: 네, 어떤 게 있어요?

한입: 첫째, **코드 리뷰** 단계에서 잠재적 버그를 잡아줘요. GitHub Copilot이나 다른 AI 도구들이 코드를 보고 "여기 null 체크 안 했네요" 같은 걸 알려줘요.

N: 오, 그건 유용하겠네요!

한입: 둘째, **테스트 케이스 생성**도 도와줘요. "이 함수에 대한 테스트 케이스 10개 만들어줘"라고 하면 만들어줘요.

N: 그러면 QA가 훨씬 빨라지겠어요.

한입: 셋째, **로그 분석**이에요. 수천 줄의 에러 로그 중에서 "이 패턴이 문제인 것 같아요"라고 요약해줄 수 있어요.
:::

:::callout[tip]
AI QA 도구들이 잘하는 것:
- 정적 코드 분석 (코드만 보고 잠재적 문제 발견)
- 테스트 케이스 자동 생성
- 에러 로그 패턴 분석
- 반복적인 회귀 테스트
- 코드 커버리지 분석
:::

## AI가 못하는 QA 영역

:::conversation
N: 그러면 AI만으로 QA가 완성되는 건가요?

한입: 아뇨, 절대 아니에요. AI가 **못하는 영역**이 있어요.

N: 어떤 거요?

한입: 첫째, **사용자 경험** 테스트예요. "이 버튼이 헷갈린다", "이 흐름이 어색하다"... 이런 건 AI가 판단 못 해요.

N: 아, 기능은 되는데 UX가 별로인 경우?

한입: 맞아요. 둘째, **맥락 이해**가 필요한 테스트요. "한국 사용자는 이 결제 방식을 싫어해"... 이런 건 문화적 맥락이 필요한 거잖아요.

N: 그건 AI가 모르겠네요.

한입: 셋째, **창의적인 테스트**예요. "만약 사용자가 이상하게 행동하면?" 같은 엣지 케이스를 상상하는 건 사람이 더 잘해요.
:::

:::callout[warning]
AI가 놓치는 QA 영역:
- 사용자 경험(UX) 품질 판단
- 문화적/지역적 맥락이 필요한 테스트
- 창의적인 엣지 케이스 발견
- 비즈니스 로직의 타당성 검증
- "느낌"이 중요한 품질 (속도감, 자연스러움)
:::

## 실제 AI QA 활용 사례

:::conversation
N: 실제로 어떻게 쓰면 좋을까요?

한입: 예시를 들어드릴게요. 로그인 기능을 만들었다고 해볼까요?

N: 네.

한입: AI에게 "이 로그인 함수의 테스트 케이스를 만들어줘"라고 하면, 기본적인 케이스들을 만들어줘요.

N: 어떤 케이스요?

한입: 정상 로그인, 비밀번호 틀림, 이메일 형식 오류, 빈 입력... 이런 **일반적인** 케이스요.

N: 그리고요?

한입: 그 다음에 QA 담당자가 **창의적인** 케이스를 추가하는 거예요. "1000자 이메일 입력하면?", "이모지로 비밀번호 만들면?", "동시에 100명이 같은 계정으로 로그인하면?"...

N: 아, AI가 기본을 깔아주고 사람이 특수한 걸 추가하는 거군요!

한입: 정확해요. **반복 작업은 AI**, **창의적 작업은 사람**. 이게 최적의 조합이에요.
:::

## PM/디자이너가 알아야 할 것

:::conversation
N: 저는 QA를 직접 하진 않는데요. 이걸 알아야 하나요?

한입: 네, 이유가 있어요.

N: 왜요?

한입: QA 과정에서 발견된 버그를 **어떻게 우선순위 정할지** 알아야 해요.

N: 아, 모든 버그를 다 고칠 순 없으니까요?

한입: 맞아요. "이건 AI가 찾은 잠재적 버그인데 실제로 발생 가능성 낮아요" vs "이건 실제 사용자가 겪은 치명적 버그예요"... 이 차이를 알아야 판단할 수 있어요.

N: 오, 그렇군요.

한입: 또 하나, AI 도구가 "위험해요"라고 하면 무조건 따를 게 아니라, **맥락을 보고 판단**해야 해요. false positive(잘못된 경고)도 많거든요.
:::

:::callout[tip]
PM/디자이너가 AI QA에 대해 알아야 할 것:
- AI가 찾은 문제 = 무조건 고쳐야 하는 문제 아님
- 실제 사용자 영향도로 우선순위 판단
- false positive(잘못된 경고) 가능성 인지
- UX 품질은 AI가 못 잡음 → 수동 테스트 필요
:::

## QA의 미래

:::conversation
N: 그러면 앞으로 QA는 어떻게 변할까요?

한입: 두 가지 방향으로 예상해요.

N: 어떤 거요?

한입: 첫째, **반복적인 테스트**는 거의 자동화될 거예요. 회귀 테스트, 기본 기능 테스트... 이런 건 AI가 다 할 거예요.

N: 그러면 QA 담당자는 뭘 하게 되나요?

한입: 둘째, QA 담당자는 **더 창의적인 일**에 집중하게 될 거예요. 사용자 시나리오 설계, 엣지 케이스 발굴, UX 품질 검증...

N: 오히려 더 재밌는 일만 남는 건가요?

한입: 어떻게 보면 그래요. **단순 반복은 AI한테**, **판단이 필요한 건 사람한테**. 이게 미래의 QA예요.
:::

:::yama
AI는 "버그 찾는 기계"가 아니라
"버그 찾는 사람을 돕는 도구"예요.

반복적인 테스트는 AI에게 맡기고,
사람은 "진짜 사용자가 어떻게 느낄까?"에 집중하세요.

결국 품질이란 "사용자 만족"이고,
그건 아직 사람만이 제대로 판단할 수 있어요.
:::
