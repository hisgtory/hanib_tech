# 웹과 앱의 UX가 달라야 하는 이유

## 같은 서비스인데 왜 다르게 디자인해야 할까?

"우리 웹 디자인 그대로 앱으로 옮기면 안 돼요?" 많이 듣는 질문이에요. 하지만 답은 "그러면 안 돼요"입니다.

## 사용 환경이 다르다

**웹 사용 환경:**
- 책상에 앉아서 마우스와 키보드로
- 큰 화면에서 여러 정보를 동시에 볼 수 있음
- 집중해서 오래 사용하는 경향

**앱 사용 환경:**
- 이동 중에 손가락 하나로
- 작은 화면에서 한 번에 하나씩
- 짧게 여러 번 사용하는 경향

:::callout[important]
지하철에서 한 손으로 폰 들고 다른 손으로 손잡이 잡은 채 사용한다고 상상해보세요. 웹처럼 복잡한 UI가 가능할까요?
:::

## 인터랙션 방식이 다르다

**웹 인터랙션:**
- 클릭 (정확한 포인터)
- 호버 (마우스 올려서 미리보기)
- 오른쪽 클릭 (컨텍스트 메뉴)
- 스크롤 (마우스 휠)

**앱 인터랙션:**
- 탭 (손가락이라 부정확)
- 스와이프 (밀어서 넘기기)
- 롱프레스 (꾹 누르기)
- 핀치 줌 (두 손가락으로 확대/축소)
- 쉐이크 (흔들기)

==호버가 없다는 건 매우 큰 차이예요. 웹에서 마우스 올리면 나오던 툴팁이 앱에서는 불가능하거든요.==

## 플랫폼별 디자인 가이드라인

**iOS - Human Interface Guidelines:**
- 깔끔하고 직관적인 디자인
- 탭바(하단 네비게이션) 선호
- 뒤로 가기: 왼쪽 상단 또는 스와이프

**Android - Material Design:**
- 레이어와 그림자 강조
- 햄버거 메뉴(사이드 네비게이션) 허용
- 뒤로 가기: 하드웨어 버튼 또는 앱바

**공통적으로:**
- 터치 타겟 최소 44pt (iOS) / 48dp (Android)
- 엄지손가락이 닿기 쉬운 영역에 주요 기능 배치

## 실제 사례: 네비게이션의 차이

**웹:**
- 상단에 수평 메뉴
- 드롭다운으로 하위 메뉴 표시
- 로고 클릭 시 홈으로

**앱:**
- 하단에 탭바 (주요 기능 3-5개)
- 상단은 현재 화면 제목과 액션 버튼
- 스와이프로 뒤로 가기

## 디자이너가 주의해야 할 점

**웹 디자인을 앱으로 옮길 때 흔한 실수:**
1. 호버 효과를 그대로 사용 → 앱에서는 불가능
2. 작은 클릭 영역 → 손가락으로 누르기 어려움
3. 복잡한 폼 → 모바일 키보드로 입력하기 불편
4. 마우스 우클릭 메뉴 → 롱프레스로 대체 필요

**앱 디자인 시 고려할 점:**
- 한 화면에 하나의 목적
- 최소한의 입력, 최대한의 선택
- 오프라인 상태 고려
- 푸시 알림 디자인

## 반응형은 해결책이 아니에요

"반응형으로 만들면 되지 않나요?"

반응형 웹은 **화면 크기만 조절**할 뿐, 인터랙션 방식은 바꾸지 않아요. 호버도 그대로고, 터치 최적화도 안 되어 있죠.

진짜 좋은 모바일 경험을 위해서는 **처음부터 모바일 환경을 고려한 설계**가 필요해요.

## 정리

| 구분 | 웹 | 앱 |
|------|-----|-----|
| 사용 환경 | 책상, 집중 | 이동 중, 짧게 |
| 입력 방식 | 마우스, 키보드 | 손가락 |
| 호버 | 가능 | 불가능 |
| 네비게이션 | 상단 메뉴 | 하단 탭바 |
| 터치 영역 | 작아도 됨 | 최소 44pt |

:::callout[tip]
"같은 서비스니까 같은 디자인"이 아니라, "같은 서비스지만 다른 경험"으로 접근하세요!
:::
