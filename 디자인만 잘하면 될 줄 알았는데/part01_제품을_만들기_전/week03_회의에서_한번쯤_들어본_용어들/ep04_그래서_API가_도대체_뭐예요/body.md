# 그래서 API가 도대체 뭐예요

## API? 들어는 봤는데...

API. Application Programming Interface. 개발자들이 입에 달고 사는 단어죠. 근데 도대체 뭘까요?

## 레스토랑으로 이해하기

여러분이 레스토랑에 갔다고 상상해보세요.

1. 손님(사용자)이 메뉴판을 봅니다
2. 웨이터에게 "불고기 정식 하나요"라고 주문합니다
3. 웨이터가 주방에 주문을 전달합니다
4. 주방에서 요리를 만듭니다
5. 웨이터가 요리를 손님에게 가져다줍니다

[illust: ID="P1-W3-E04-01" DESC="API를 레스토랑 웨이터로 비유한 흐름도 - 손님(프론트엔드) → 주문(요청) → 웨이터(API) → 주방(백엔드) → 음식(응답) → 웨이터(API) → 손님"]

여기서 **웨이터가 바로 API**에요!

- 손님(프론트엔드)이 주방(백엔드)에 직접 갈 수 없어요
- 그래서 웨이터(API)를 통해 요청하고 응답을 받아요
- 메뉴판은 API 문서와 같아요 (주문 가능한 것들의 목록)

:::callout[quote]
API = "정해진 규칙대로 요청하면, 정해진 형태로 응답해줄게"
:::

## 실제 예시로 이해하기

**날씨 앱을 만든다고 가정해볼게요:**

1. 날씨 앱(프론트엔드)이 "서울 날씨 알려줘"라고 요청
2. 기상청 API가 요청을 받음
3. 기상청 서버(백엔드)가 데이터를 찾음
4. API가 "서울, 맑음, 15도"라고 응답
5. 날씨 앱이 화면에 표시

**일상에서 이미 쓰고 있는 API:**
- 카카오로 로그인 → 카카오 로그인 API
- 네이버 지도 표시 → 네이버 지도 API
- 카카오페이 결제 → 카카오페이 API
- 인스타그램 피드 불러오기 → 인스타그램 API

## API 요청의 기본 구조

**요청(Request):**
- 어디로? (URL/엔드포인트)
- 어떻게? (GET, POST, PUT, DELETE)
- 뭘 보낼까? (데이터)

**응답(Response):**
- 성공했나? (상태 코드: 200, 404, 500 등)
- 결과는? (JSON 형태의 데이터)

:::callout[tip]
200 = 성공, 404 = 못 찾음, 500 = 서버 에러
이 세 가지만 알아도 절반은 이해한 거예요!
:::

## 디자이너가 API를 알아야 하는 이유

==API를 이해하면 "왜 이 화면이 필요한지" 설명할 수 있어요.==

**디자인에 영향을 미치는 API 특성:**

1. **로딩 시간**
   - API 응답이 느리면 로딩 화면이 필요해요
   - 스켈레톤 UI를 디자인해야 할 수도

2. **에러 상태**
   - API가 실패하면 어떤 화면을 보여줄까요?
   - 네트워크 오류, 서버 오류, 데이터 없음...

3. **데이터 형태**
   - API가 어떤 정보를 주는지에 따라 UI가 달라져요
   - "프로필 사진이 없으면 어떻게 표시할까요?"

4. **실시간 업데이트**
   - 채팅처럼 실시간이면 새로고침 없이 업데이트
   - 일반 API면 새로고침해야 볼 수 있음

## 개발자와 대화하는 법

**Before (API 모를 때):**
"여기 데이터 좀 넣어주세요"

**After (API 알 때):**
"이 화면에서 API 응답이 느리면 스켈레톤 UI 넣고, 에러 나면 재시도 버튼 보여주는 게 어때요?"

## 정리

| 비유 | 실제 |
|------|------|
| 레스토랑 | 서비스 |
| 손님 | 프론트엔드 (화면) |
| 웨이터 | API |
| 주방 | 백엔드 (서버) |
| 메뉴판 | API 문서 |
| 주문 | API 요청 |
| 음식 | API 응답 (데이터) |

:::callout[important]
API는 프론트엔드와 백엔드를 연결하는 약속된 규칙이에요.
웨이터처럼 중간에서 요청을 전달하고 결과를 가져다주죠.
:::
