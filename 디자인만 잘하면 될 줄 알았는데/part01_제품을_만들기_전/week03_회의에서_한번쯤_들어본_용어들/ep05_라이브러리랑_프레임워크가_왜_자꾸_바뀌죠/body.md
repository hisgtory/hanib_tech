# 라이브러리랑 프레임워크가 왜 자꾸 바뀌죠

## "또 바꾼다고요?"

"우리 React에서 Vue로 바꾸기로 했어요." "이제 Flutter 쓴대요." 개발팀에서 이런 말 들어본 적 있나요? 왜 자꾸 바뀌는 걸까요?

## 라이브러리 vs 프레임워크, 뭐가 다를까?

### 라이브러리 = 도구 상자

필요할 때 꺼내 쓰는 도구들이에요. 망치가 필요하면 망치를, 드라이버가 필요하면 드라이버를 꺼내 쓰듯이요.

**예시:**
- Lodash (자주 쓰는 기능 모음)
- Moment.js (날짜 다루기)
- Axios (API 호출)

### 프레임워크 = 설계도가 있는 집

이미 구조가 정해져 있어요. 그 안에서 규칙대로 작업해야 해요.

**예시:**
- React, Vue, Angular (프론트엔드)
- Django, Spring (백엔드)
- Flutter, React Native (앱)

:::callout[quote]
"라이브러리는 내가 호출하고, 프레임워크는 나를 호출한다"
:::

## 왜 자꾸 바뀔까?

### 1. 기술의 빠른 발전

5년 전에 최고였던 기술이 지금은 구식이 되기도 해요. 더 좋은 방법이 계속 나오거든요.

**실제 역사:**
- 2013년: AngularJS가 대세
- 2015년: React가 등장하며 역전
- 2018년: Vue가 급부상
- 2020년 이후: Next.js, Svelte 등 새로운 선택지

### 2. 개발 생산성 향상

새 프레임워크는 보통 더 적은 코드로 더 많은 일을 할 수 있게 해줘요.

**예시:**
```
jQuery 시절: 100줄
React 시절: 30줄
최신 프레임워크: 10줄
```

### 3. 커뮤니티와 지원

인기 있는 프레임워크는:
- 문제가 생겨도 검색하면 답이 나옴
- 새로운 기능이 계속 추가됨
- 취업 시장에서 유리함

인기가 없어지면 반대 상황이 되기 때문에 갈아타게 돼요.

### 4. 성능 개선

새로운 기술은 대체로:
- 더 빠른 로딩 속도
- 더 적은 메모리 사용
- 더 나은 사용자 경험

## 개발자들의 고충

==프레임워크가 바뀌면 개발자도 다시 공부해야 해요.==

**실제 상황:**
- 기존 코드 전부 다시 작성해야 할 수도
- 새 문법, 새 규칙 학습 필요
- 마이그레이션 기간 동안 생산성 저하

그래서 개발자들은 "이번엔 진짜 오래 갈까?"를 항상 고민해요.

## 디자이너가 알아야 할 점

**프레임워크 변경이 디자인에 미치는 영향:**

1. **가능한 것과 불가능한 것이 달라질 수 있어요**
   - "전에는 됐는데 이제 안 돼요"
   - "이제 이것도 가능해요!"

2. **컴포넌트 라이브러리가 바뀔 수 있어요**
   - Material Design → Ant Design
   - 버튼, 폼 스타일이 달라질 수 있음

3. **일정이 늘어날 수 있어요**
   - 새 기술 학습 시간 필요
   - 예상보다 오래 걸릴 수 있음

## 흔히 듣는 프레임워크들

**프론트엔드:**
| 이름 | 특징 |
|------|------|
| React | 페이스북이 만듦, 가장 인기 |
| Vue | 배우기 쉬움, 아시아에서 인기 |
| Angular | 구글이 만듦, 대규모 프로젝트용 |
| Svelte | 가벼움, 신생 |

**앱 개발:**
| 이름 | 특징 |
|------|------|
| Flutter | 구글, 빠른 개발 |
| React Native | 페이스북, 웹 개발자 친화적 |
| Swift/Kotlin | 네이티브, 최고 성능 |

## 정리

:::callout[important]
기술은 계속 발전하고, 개발자들은 더 좋은 도구를 찾아 이동해요.
"또 바꾼다고요?"가 아니라 "더 좋은 게 나왔구나"로 이해해주세요.
:::

**핵심:**
- 라이브러리 = 필요할 때 꺼내 쓰는 도구
- 프레임워크 = 규칙이 있는 작업 환경
- 바뀌는 이유 = 더 좋은 기술이 나오니까
- 디자이너 영향 = 가능한 것, 컴포넌트, 일정이 달라질 수 있음
