# 프론트엔드와 백엔드가 나뉜 이유

## 왜 둘로 나눠졌을까?

레스토랑에 가면 홀과 주방이 나눠져 있죠. 손님이 직접 주방에 들어가 요리사에게 주문하지 않잖아요. 서빙 직원이 주문을 받고, 주방에서 요리사가 음식을 만들어요. 웹/앱 개발도 마찬가지입니다.

[illust: ID="P1-W3-E01-01" DESC="레스토랑 비유 다이어그램 - 왼쪽: 홀(프론트엔드)에서 손님이 웨이터에게 주문, 오른쪽: 주방(백엔드)에서 요리사가 조리, 중간에 웨이터(API)가 연결"]

- **프론트엔드 (홀)**: 사용자가 보고 만지는 모든 것
- **백엔드 (주방)**: 데이터를 저장하고 처리하는 보이지 않는 곳

## 현대 개발이 복잡해진 이유

예전에는 한 사람이 다 했어요. 그런데 지금은?

**프론트엔드만 해도 산더미:**
- React, Vue, Angular 같은 프레임워크
- 상태 관리, 번들링, 테스팅
- 성능 최적화, 접근성
- 브라우저 호환성

**백엔드도 만만치 않아요:**
- 데이터베이스 설계
- API 아키텍처
- 보안, 캐싱
- 로드 밸런싱

한 사람이 이 모든 것을 깊이 있게 다루기가 거의 불가능해졌어요.

## 전문화의 힘

:::callout[quote]
"넓게 아는 것보다 깊게 아는 것이 낫다"
:::

네이버 같은 대형 IT 기업을 보면 명확히 분리되어 있어요:
- **프론트엔드팀**: UI/UX, 사용자 경험에 특화
- **백엔드팀**: API, 데이터베이스, 서버에 특화
- **DevOps팀**: 인프라, 배포, 모니터링에 특화

각 팀이 자신의 전문 영역에 집중하면서 유기적으로 협업하는 구조예요.

## API가 만드는 경계선

둘 사이를 연결하는 건 **API**입니다. 웨이터가 홀과 주방 사이를 오가듯이요.

프론트엔드 개발자가 알아야 할 것:
- API 주소 (어디로 요청을 보낼지)
- 요청/응답 형식
- 에러 처리 방법

프론트엔드 개발자가 몰라도 되는 것:
- 데이터베이스 구조
- SQL 쿼리 최적화
- 서버 아키텍처

## 병렬 개발이 가능해져요

분리되면 좋은 점이 있어요. **동시에 작업**할 수 있다는 거예요!

백엔드팀이 API를 개발하는 동안, 프론트엔드팀은 가짜 데이터(Mock)로 화면을 미리 만들 수 있어요. 결과적으로 개발 속도가 빨라지죠.

## 디자이너가 알아야 할 포인트

==프론트엔드와 백엔드의 분리를 이해하면, 개발자와의 소통이 훨씬 쉬워져요.==

**실무 팁:**
- "이 데이터 어디서 와요?" → 백엔드에서 API로 받아옵니다
- "로딩 중일 때 뭘 보여줄까요?" → API 응답을 기다리는 동안의 상태
- "에러 화면도 필요한가요?" → API 호출이 실패할 수 있으니 필요해요

## 정리

| 구분 | 프론트엔드 | 백엔드 |
|------|-----------|--------|
| 비유 | 레스토랑 홀 | 레스토랑 주방 |
| 역할 | 사용자가 보는 화면 | 데이터 처리 |
| 사용 기술 | React, Vue, HTML/CSS | Node.js, Python, DB |
| 담당자 | 프론트엔드 개발자 | 백엔드 개발자 |

:::callout[tip]
회의에서 "프론트에서 처리할까요, 백에서 처리할까요?"라는 말이 나오면, 이제 무슨 뜻인지 알 수 있겠죠?
:::
