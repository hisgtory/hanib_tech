## 일 잘하는 디자이너의 3가지 실무 역량

### 1. 시스템 사고 (System Thinking)

화면 하나하나를 개별적으로 디자인하는 게 아니라,
**전체 서비스를 하나의 시스템으로 바라봐요.**

| 화면 단위 사고 | 시스템 사고 |
|---------------|------------|
| "이 버튼 색깔 뭘로 할까?" | "버튼 계층 구조는 어떻게 가져갈까?" |
| "이 페이지 레이아웃은?" | "일관된 그리드 시스템은?" |
| "여기 폰트 사이즈는?" | "타이포그래피 스케일은?" |
| "로딩 화면 어떻게?" | "모든 상태(로딩/에러/빈값)의 패턴은?" |

:::callout[tip]
"이 화면"이 아니라 "이런 유형의 화면들"을 생각하세요.
한 번 결정하면 수십 개 화면에 적용됩니다.
:::

### 2. 프로토타이핑 활용

일 잘하는 디자이너는 **말로 설명하지 않아요.**
움직이는 걸로 보여줍니다.

**프로토타입의 힘:**
- 기획자: "아, 이런 느낌이었구나!"
- 개발자: "이렇게 움직이는 거군요"
- 클라이언트: "오, 벌써 만들었어요?"

**단계별 프로토타입:**

```
Low-fi (종이/와이어프레임)
↓ 구조 검증
Mid-fi (피그마 기본 인터랙션)
↓ 플로우 검증
Hi-fi (실제 애니메이션)
↓ 감성 검증
```

:::callout[important]
프로토타입은 "완성품 미리보기"가 아니라
"아이디어 검증 도구"예요.
빠르게 만들고, 빠르게 피드백 받으세요.
:::

### 3. 디자인 시스템 구축 & 활용

**컴포넌트화의 마법:**

```
Before (비효율)
├── 메인_버튼_빨강.png
├── 메인_버튼_파랑.png
├── 서브_버튼_빨강.png
└── ... (무한 반복)

After (효율)
├── Button (컴포넌트)
│   ├── Variant: Primary / Secondary / Ghost
│   ├── Size: Large / Medium / Small
│   └── State: Default / Hover / Disabled
```

**디자인 시스템의 효과:**

| Before | After |
|--------|-------|
| 버튼 하나 바꾸면 50개 화면 수정 | 마스터만 바꾸면 전체 반영 |
| "이 색 정확히 뭐였지?" | 토큰으로 관리 |
| 개발자: "이거 어디 기준이에요?" | 문서화된 스펙 전달 |

## 실무에서 바로 적용하기

### 체크리스트

**시스템 사고:**
- [ ] 새 화면 그리기 전, 기존 패턴 먼저 확인했나?
- [ ] 예외 케이스(에러, 로딩, 빈 상태)도 고려했나?
- [ ] 이 결정이 다른 화면에 미칠 영향을 생각했나?

**프로토타이핑:**
- [ ] 복잡한 인터랙션은 움직이는 걸로 보여주고 있나?
- [ ] 피드백 받기 전에 프로토타입 먼저 만들었나?
- [ ] 개발 핸드오프에 인터랙션 스펙이 포함되어 있나?

**디자인 시스템:**
- [ ] 반복되는 요소를 컴포넌트로 만들었나?
- [ ] 컬러/타이포/스페이싱에 일관된 토큰을 쓰고 있나?
- [ ] 개발팀과 공유되는 디자인 시스템 문서가 있나?
