## 커뮤니케이션 잘하는 디자이너의 특징

### 1. 피드백을 선물로 받아들인다

**안 좋은 반응:**
- "제 디자인 의도는요..."
- "그건 UX 원칙상..."
- (속으로) '저 사람 디자인 뭘 알아?'

**좋은 반응:**
- "어떤 부분이 불편하셨어요?"
- "다른 대안도 준비해볼게요"
- "그 관점은 생각 못 했네요, 감사해요"

:::callout[quote]
"피드백에 방어적인 디자이너는
결국 자기 한계 안에 갇힙니다."
:::

**피드백 수용 공식:**

```
듣기 → 질문하기 → 이해했음을 표현 → 반영 여부 공유

❌ "네, 근데요..."
✅ "말씀하신 부분이 [이것]이죠? 
    제가 [이렇게] 수정해볼게요."
```

### 2. 디자인을 "설명"이 아니라 "설득"한다

**설명하는 디자이너:**
> "여기 여백을 24px로 한 이유는 8의 배수 그리드를 
> 사용하고 있어서이고, 이 색상은 WCAG 2.1 기준 
> 4.5:1 명도 대비를 충족하며..."

😴 (상대방 이미 졸고 있음)

**설득하는 디자이너:**
> "사용자가 결제 버튼을 더 빨리 찾을 수 있어요.
> 전환율이 올라갈 거예요."

👀 (상대방 눈이 반짝)

| 설명 | 설득 |
|-----|-----|
| How (어떻게 만들었는지) | Why (왜 이게 좋은지) |
| 디자인 원칙 나열 | 비즈니스 임팩트 연결 |
| "제가 이렇게 했어요" | "사용자/회사에 이런 효과가 있어요" |

### 3. "안 돼요" 대신 "이렇게 하면요"

기획자가 이상한(?) 요청을 했을 때:

**🚫 안 좋은 반응:**
- "그건 UX적으로 안 돼요"
- "디자인 가이드에 어긋나요"
- "그렇게 하면 이상해요"

**✅ 좋은 반응:**
- "그 목적이라면, 이 방법은 어떨까요?"
- "비슷한 효과를 내면서 더 나은 방법이 있어요"
- "A안과 B안 중에 어떤 게 좋으실까요?"

:::callout[tip]
거절할 때도 **대안을 함께** 제시하세요.
"No"만 하는 사람은 결국 배제됩니다.
:::

## 상황별 커뮤니케이션 가이드

### 기획자와 소통할 때

```
❌ "요구사항이 명확하지 않아요"
✅ "이 부분을 이렇게 이해했는데, 맞나요?"

❌ "이건 디자인으로 해결할 문제가 아니에요"
✅ "이 문제의 근본 원인을 같이 파악해볼까요?"

❌ "일정이 너무 촉박해요"
✅ "이 일정 안에 가능한 범위를 정리해볼게요"
```

### 개발자와 소통할 때

```
❌ "왜 디자인대로 안 나왔어요?"
✅ "의도한 것과 다른 부분이 있어서 확인하고 싶어요"

❌ "이것도 구현 못 해요?"
✅ "기술적 제약이 있다면 대안을 찾아볼게요"

❌ "디자인 스펙은 전달했잖아요"
✅ "제가 전달한 스펙에서 불명확한 부분 있으면 알려주세요"
```

### 이해관계자와 소통할 때

```
❌ "이게 더 예뻐요"
✅ "이 디자인이 목표 달성에 더 효과적이에요"

❌ "트렌드가 이래요"
✅ "경쟁사 대비 이런 차별점이 생겨요"

❌ "제 디자인 철학은..."
✅ "사용자 데이터를 보면..."
```

결국 디자인도 사람이 하고, 사람을 위해 하니까요.
:::
