# 팀에 전파하기

## "기획 다 했는데, 이제 어떻게 공유하죠?"

기획을 열심히 했어요. 그런데 이걸 팀원들에게 어떻게 전달해야 할까요? PPT 수백 장을 만들어야 할까요?

## 시대가 바뀌었어요

### 과거: 두꺼운 기획서

```
워터폴 시대의 기획 전달:

1. 기획서.pptx (200-300페이지) 작성
2. 와이어프레임 전체 제작
3. 킥오프 미팅에서 2시간 발표
4. "질문 있으신가요?"
5. 정해진 대로 진행

→ 기획자가 모든 것을 결정
→ 팀은 "수동적으로 받아들이는" 역할
```

### 현재: PRD 한 장

```
애자일 시대의 기획 전달:

1. PRD 5-20페이지 작성
2. 팀 미팅에서 30분 논의
3. 함께 방향 결정
4. 만들면서 계속 업데이트

→ 기획자가 방향을 제시
→ 팀이 "함께 완성하는" 역할
```

:::callout[tip]
완벽한 기획서보다 빠른 공유와 피드백이 더 중요해요.
:::

## PRD란?

### Product Requirements Document

```
PRD에 들어가는 내용:

1. Problem (문제)
   - 왜 이걸 만들어야 하나요?
   - 데이터와 유저 피드백

2. Goal (목표)
   - 어떤 변화를 만들고 싶은가

3. Solution (해결책)
   - 핵심 아이디어
   - 참고 사례

4. Requirements (요구사항)
   - Must have (필수)
   - Nice to have (있으면 좋음)

5. Success Metrics (성공 지표)
   - 어떻게 성공을 측정할 것인가
```

### 실제 PRD 예시

```
[소셜 로그인 추가 PRD]

Problem:
- 회원가입 전환율 40%
- "이메일 입력이 번거롭다" 피드백 반복

Goal:
- 전환율 40% → 70% 달성

Solution:
- 카카오/구글 소셜 로그인 추가

Requirements:
- Must have: 카카오, 구글
- Nice to have: 애플, 네이버

Success Metrics:
- 소셜 로그인 선택률
- 전체 전환율
```

## 살아있는 PRD

### 한 번 쓰고 끝이 아니에요

```
Week 1: PRD 초안 5페이지
        ↓ 팀 피드백
Week 2: 수정 후 6페이지
        ↓ 디자인하면서 발견된 이슈
Week 3: 업데이트 7페이지
        ↓ 개발하면서 추가된 정책
Week 4: 최종 8페이지

→ PRD는 계속 진화해요
```

:::callout[important]
PRD는 "완벽한 문서"가 아니라 "팀이 함께 만들어가는 과정"이에요.
:::

## 역할의 변화

### 과거

```
기획자: 모든 것 결정 (기획서 + IA + 와이어프레임)
        ↓
디자이너: 색 입히기
        ↓
개발자: 구현
```

### 현재

```
PM: Why & What (왜, 무엇을)
    ↓
PD: How (어떻게) - 와이어프레임, 인터랙션, 정책
    ↓
함께: 세부 정책 완성
    ↓
개발자: 구현
```

==디자이너(PD)의 역할이 확장됐어요. 단순히 예쁘게 만드는 게 아니라, 사용자 경험 전체를 설계해요.==

## 팀에 맞는 방식 찾기

### 상황별 선택

```
대규모 조직 / 레거시 서비스:
- 여러 팀 협업
- 규제/법률 검토 필요
- 상세한 문서화 중요
→ 전통적 기획서가 유효

스타트업 / 신규 서비스:
- 빠른 의사결정
- 적은 인원
- 실험 중심
→ 간결한 PRD가 적합
```

:::callout[quote]
"완벽한 기획서는 없습니다. 우리 팀이 가장 빠르게, 가장 명확하게 협업할 수 있는 방식이 정답이에요."
:::

## 정리

**핵심:**
- 두꺼운 기획서 시대는 지났어요
- PRD로 핵심만 공유하고 함께 만들어가세요
- 문서는 살아있어야 해요, 계속 업데이트하세요
- 팀 상황에 맞는 방식을 선택하세요
