# 우리 팀이 제품을 만드는 방법: 애자일 

N: 와... 린 스타트업 덕분에 이렇게 잘 살아남게 된 거네요!

한입: 음... 반만 성공했다고 해야 할까요?

N: 반만이요? 무슨 뜻이에요?

한입: '무엇을 만들까'는 알아냈는데, '어떻게 만들까'는 여전히 잘
몰랐거든요.

N: 어? 그게 다른 거예요?

한입: 완전히 다른 문제죠. 린 스타트업은 시장 검증의 문제고, 이건 팀
협업의 문제니까요.

N: 아...

한입: 바로, 제품을 만드는 프로세스의 문제였어요.

N: 그래도 다들 열심히 빠르게 일하면 되는 거 아닌가요?

한입: 그렇게 생각하기 쉽죠. 근데요, 각자가 아무리 열심히 해도 협업
방식이 잘못되면 결과가 안 나와요.

N: 그게 무슨...

## **제품을 출시하지 못하는 팀**

팀이 모여서 함께 제품 아이디어를 골랐습니다. 2주 뒤 기획자가
와이어프레임과 기획서를 디자이너에게 공유했습니다. 디자이너는 아쉬운
UX와 누락된 화면들을 발견하지만 팀이 약속한 일정을 지키기 위해 최소한의
커뮤니케이션만으로 디자인을 완성합니다.

4주차, 기획서와 디자인을 개발자에게 소개하는 킥오프 미팅을 갖습니다.

개발자들이 느끼기에 초기 아이디어보다 기획이 많이 커지고 달라진 것
같습니다.

구현이 불가능한 부분들도 보이고 도저히 일정을 맞추기 힘들 것 같습니다.

급하게 개발하다보니 서버 개발자가 작성한 서버 기능이 앱의 UX와는 조금
맞지 않아 앱 개발자는 서버 개발자에 수정을 요청합니다.

일정은 밀리고 팀은 지쳐갑니다.

돌이켜보니 지난 프로젝트도 이렇게 끝을 맺었던 것 같습니다.

왜 이런 일이 반복될까요?

각자 맡은 일을 열심히 했던 것 같지만 팀은 명확한 단기 목표가 없었고,
팀원간 소통이 부족해 업무 동기화에 실패했습니다.

## **팀이 더욱 깊게 몰입하는 법**

그럼 성공하는 팀들은 뭐가 다를까요?

'잘 돼가요?' '네, 잘 되고 있어요'와 같은 소통이 아닌 **제대로
소통하면서** 팀이 모두 깊게 몰입하는 환경이 필요합니다.

**스프린트 : 단거리 전력 질주**

매주 월요일, 팀원들이 다같이 이번 주에 끝낼 수 있는 일만 약속합니다.

**핵심은 '이번 주'라는 명확한 데드라인이에요.** 마라톤이 아니라 단거리
전력질주, 그래서 '스프린트'라고 부릅니다.

정해진 기간 동안은 이 일에만 집중해요. "이것도 해야 하는데..."
"저것도 중요한데..." 중간에 끼어드는 일들은 다 **백로그**에
적어둡니다.\
다음 스프린트에 하면 되니까요.

이렇게 해야 팀이 흔들리지 않고 목표에 집중할 수 있어요.

그리고 스프린트의 마지막 날, 팀이 모여 회고를 합니다. 잘한 점, 아쉬운
점, 개선할 점을 각각 나누면서 이야기해요. 이렇게 매주 조금씩 일하는
방식을 개선해나가는 거죠.

스프린트의 기간은 팀에 따라 1주나 2주 단위로 진행합니다.

**백로그와 칸반보드**

새로운 스프린트가 시작되면 백로그에서 이번 스프린트에 할 업무를
고릅니다.

그리고 업무별 진행 상태를 할 일, 진행 중, 리뷰 중, 완료와 같이 구분해서
관리하는데요.

업무를 상태별로 묶어서 시각화한 것을 칸반 보드라고 합니다.

한번쯤 봐보셨을 Notion의 Board가 바로 그것입니다.

백로그에 쌓아둔 할 일들 중에서 이번 스프린트에 할 것들을 골라요. 그리고
**칸반보드**에서 각 일의 상태를 추적합니다.

월요일엔 모든 카드가 '할 일'에 있지만, 작업이 진행되면서 오른쪽으로
이동하고 금요일엔 대부분 '완료'로 옮겨져 있어야 합니다. 누가 뭘 하고
있는지, 어디서 막혔는지 한눈에 보이고 프로젝트별로 보드를 만들 경우
런칭까지 남은 이슈들을 명확하게 파악할 수 있습니다.

이 시각적 진척도가 팀에 엄청난 동기부여를 넘어 실용적인 프로젝트 관리도
가능해지죠.

## **집단지성의 힘을 극대화 하는 법**

스프린트면 몰입과 소통을 모두 해결했다고 할 수 있을까요?

아닙니다. 실제로 일을 하다보면 계획했을 때는 몰랐던 문제들이 튀어나오고,
상황이 달라지기 일쑤죠.

그래서 성공하는 팀들은 매일 아침 15분씩 모여 이야기합니다. 짧은 시간
안에 끝내는 게 핵심이에요.

이런 짧은 일일 미팅을 '데일리 스크럼'이라고 부르고, 서서 진행하는
경우가 많아서 '스탠드업 미팅'이라고도 해요. 서서 하면 회의가 길어지지
않거든요.

"어제 뭐 했고, 오늘 뭐 할 거고, 막히는 게 있나요?"

앱 개발자: "로그인 구현 완료해서 오늘부터 홈 피드 개발 시작하려고
해요" 서버 개발자: "아, 그럼 지금 하던 탐색탭 API 잠시 두고 오늘 홈
피드 API부터 챙길게요"\
디자이너: "맞다, 홈 피드에 디자인 변경사항 있는데 오전 중으로
반영해둘게요!"\
기획자: "그럼 저는 오늘 홈 피드 테스트 시나리오 작성할게요"

이런 식으로 팀이 더욱 효율적으로 일하기 위해 서로의 우선순위가
실시간으로 조정되기도 합니다. 서로에게 뭐가 필요한지 바로바로 조율하기도
하고 서로의 지식을 공유하기도 합니다.

앱 개발자: "이 애니메이션 구현하는데 3일은 걸릴 것 같아요"\
디자이너: "아, 그럼 Lottie로 바꿀까요? 그럼 하루면 되지
않나요?"\
서버 개발자: "맞아요, 우리 이전 프로젝트에서 Lottie 썼을 때 좋았어요"
이런식으로 방향을 바꾸기도 하죠.

때로는 형식적인 것처럼 느껴질 수도 있지만 스크럼은 많은 장점이 있습니다.

1.  **하루 단위 목표 수립으로 더욱 몰입할 수 있음**\
    "오늘 안에 끝내야지"라는 마음가짐이 생겨요

2.  **팀원들의 업무 상황을 서로 잘 알 수 있음**\
    누가 뭘 하고 있는지, 어디서 막혔는지 바로 파악할 수 있어요

3.  **팀원을 서포트하기 위해 본인의 업무 우선순위를 조정할 수 있음**
    "내가 먼저 이거 해주면 팀이 더 빨라지겠네"

4.  **업무에 대해 피드백, 노하우 공유 등을 토대로 서로 더 효율적으로
    일할 수 있게 도움**\
    "그거 저번에 제가 해봤는데, 이렇게 하면 빨라요"

## **애자일**

지금까지 소개한 스프린트, 백로그, 칸반보드, 데일리 스크럼은 모두
애자일(Agile)이라는 하나의 철학에서 나온 거예요.

우리가 2장에서 본 실패하는 팀, 기억나시나요?\
기획 → 디자인 → 개발이 순서대로 진행되는 '폭포수Waterfall' 방식이었죠.

이 방식은 실패를 만나기까지 너무 오래 걸립니다.

6개월 후에 완성된 제품이 시장에서 외면받아도 이미 늦었죠.

애자일은 다릅니다. 짧은 주기로 작은 제품을 만들고, 피드백을 받고,
개선하고, 다시 만들어요. 실패해도 1-2주만 잃으면 되니까 부담이 없죠.

결국 애자일은 '빠르게 실패하고, 빠르게 배우는 것'입니다. 완벽한 계획
대신 끊임없는 개선을 선택하는 거예요.

3개월 전에 세운 100페이지짜리 기획을 고집하는 대신, 시장이 바뀌면 우리도
바뀝니다.

지라나 노션 같은 도구가 중요한 게 아니에요. 사람과 사람 사이의 소통이
핵심이죠.

"**계획을 완벽하게 세우는 것보다, 변화에 빠르게 대응하는 게 중요하다."**
