# 서버 개발자와 DevOps는 뭐가 달라요

## "다 같은 개발자 아니에요?"

"서버 개발자한테 물어보세요." "그건 DevOps팀 담당이에요." 둘 다 서버 관련인 것 같은데, 왜 다른 사람한테 물어보래요?

## 레스토랑으로 비유해볼게요

### 서버 개발자 = 요리사

주방에서 음식을 만드는 사람이에요.
- 레시피(로직)를 개발해요
- 재료(데이터)를 어떻게 조합할지 결정해요
- 맛있는 음식(기능)을 만들어내요

**서버 개발자가 하는 일:**
- 회원가입 로직 만들기
- 결제 처리 시스템 구현
- 데이터베이스 설계
- API 개발

### DevOps = 주방 시설 관리자

주방이 잘 돌아가도록 환경을 관리하는 사람이에요.
- 가스레인지, 냉장고가 잘 작동하는지 확인
- 주방 온도를 적절하게 유지
- 손님이 많아지면 주방을 확장

**DevOps가 하는 일:**
- 서버 인프라 구축 및 관리
- 배포 자동화 (새 기능을 서비스에 반영)
- 모니터링 (서버 상태 감시)
- 장애 대응

## 왜 분리되어 있을까?

### 1. 전문성이 다릅니다

| 서버 개발자 | DevOps |
|------------|--------|
| 프로그래밍 언어 | 인프라 도구 |
| 비즈니스 로직 | 시스템 운영 |
| 기능 개발 | 안정성 확보 |
| "이 기능 어떻게 만들지?" | "이 서버 어떻게 띄울지?" |

### 2. 관심사가 다릅니다

**서버 개발자의 고민:**
- 이 데이터를 어떻게 처리할까?
- 코드를 어떻게 더 깔끔하게 짤까?
- 새 기능을 어떻게 구현할까?

**DevOps의 고민:**
- 서버가 죽으면 어떻게 복구하지?
- 사용자가 갑자기 늘어나면 어떻게 하지?
- 배포를 더 빠르고 안전하게 할 순 없을까?

## 실제 협업 시나리오

**새 기능 출시할 때:**

```
서버 개발자: "회원가입 기능 완성했어요!"
DevOps: "테스트 서버에 먼저 배포할게요."
DevOps: "문제없네요. 실서버에 반영합니다."
```

**장애 발생했을 때:**

```
DevOps: "서버 메모리가 부족해요. 일단 증설했습니다."
서버 개발자: "코드에서 메모리 누수 찾아서 수정할게요."
```

## 디자이너가 알아야 할 점

### 누구한테 물어봐야 할까?

| 상황 | 담당 |
|------|------|
| "이 기능 언제 나와요?" | 서버 개발자 |
| "왜 서버가 느려요?" | DevOps (또는 둘 다) |
| "이 데이터 어디서 와요?" | 서버 개발자 |
| "배포 언제 돼요?" | DevOps |
| "API 스펙 주세요" | 서버 개발자 |

### 일정에 미치는 영향

==새 기능 개발 = 서버 개발 시간 + 배포 준비 시간==

단순히 "코드 짜는 시간"만 계산하면 안 돼요.
- DevOps가 배포 환경 준비하는 시간
- 테스트 서버에서 검증하는 시간
- 실서버 반영 후 모니터링 시간

이 모든 게 포함되어야 해요.

## 요즘 트렌드

:::callout[tip]
요즘은 "풀스택 개발자"처럼 "개발자가 DevOps도 하는" 경우가 많아요.
하지만 규모가 커지면 결국 분리됩니다.
:::

**작은 스타트업:** 개발자가 DevOps도 겸임
**중간 규모:** DevOps 담당자 1-2명
**대기업:** DevOps팀이 따로 존재

## 정리

:::callout[important]
서버 개발자는 '무엇을' 만들고, DevOps는 '어떻게' 운영할지 담당해요.
요리사와 주방 관리자처럼, 둘 다 있어야 레스토랑이 잘 돌아갑니다.
:::

**핵심:**
- 서버 개발자 = 기능을 만드는 사람
- DevOps = 서비스가 잘 돌아가게 하는 사람
- 질문할 때 누구에게 물어볼지 구분하면 소통이 빨라져요
