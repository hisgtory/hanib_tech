# 일 잘하는 개발자 특징

## 스토리텔링 버전

:::conversation
**N**: 한입님, 저희 팀에 개발자가 두 분 계신데요. 실력은 비슷한 것 같은데 한 분이랑 일하면 훨씬 편해요. 뭐가 다른 걸까요?

**한입**: 오, 재밌는 관찰이네요! 어떤 점이 다르던가요?

**N**: 한 분은 "안 돼요"라고 자주 하시고, 다른 분은 질문을 많이 하시는데... 결과적으로 질문 많이 하시는 분이랑 일하면 결과물이 더 좋아요.

**한입**: 아, 제가 아는 "일 잘하는 개발자"의 특징이네요!
:::

### 특징 1: "왜?"를 많이 물어봐요

:::conversation
**한입**: 일 잘하는 개발자는 요구사항을 그냥 받지 않아요.

**N**: 어떻게요?

**한입**: 예를 들어 "버튼 색을 빨간색으로 바꿔주세요"라고 하면,

**보통 개발자:** "네, 바꿔드릴게요."

**일 잘하는 개발자:** "어떤 이유로 빨간색이 필요하세요? 사용자에게 경고를 주려는 거라면 아이콘을 추가하는 게 더 효과적일 수도 있어요."

**N**: 아! 문제의 본질을 파악하려는 거군요?

**한입**: 맞아요. 그래서 결과적으로 더 좋은 해결책이 나오죠.
:::

### 특징 2: "안 돼요" 대신 대안을 제시해요

:::conversation
**N**: 아까 말씀드린 "안 돼요" 자주 하시는 분이 계시거든요...

**한입**: 물론 기술적으로 진짜 안 되는 경우도 있어요. 하지만 일 잘하는 개발자는 다르게 말해요.

**"안 돼요" 대신:**
- "이 방식은 어렵지만, 이렇게 하면 비슷한 효과를 낼 수 있어요"
- "지금은 어렵지만, 다음 스프린트에 기반 작업을 하면 가능해요"
- "완전히는 어렵고, 80% 정도로 구현하면 일주일이면 돼요"

**N**: 같은 "안 돼요"인데 받아들이는 느낌이 다르네요!
:::

### 특징 3: 모르는 걸 모른다고 말해요

```
❌ "음... 될 것 같은데요" (확신 없이)
✅ "이 부분은 제가 경험이 없어서 확인해보고 말씀드릴게요"
```

:::callout[tip]
모르는 걸 아는 척하면 나중에 더 큰 문제가 생겨요.
일 잘하는 개발자는 솔직하게 모른다고 하고, 빠르게 파악해와요.
:::

### 특징 4: 비개발자 눈높이로 설명해요

:::conversation
**N**: 가끔 개발자분들이 어려운 용어를 쓰시면 뭔 말인지 모르겠어요.

**한입**: 일 잘하는 개발자는 상대방에 맞춰 설명해요.

**기획자에게:** "이 기능은 서버에서 2초 정도 걸려요. 사용자가 기다려야 하니까 로딩 화면이 필요해요."

**경영진에게:** "이 작업은 3일 걸리는데, 하면 앱 속도가 30% 빨라져서 이탈률이 줄어들 거예요."

**개발자에게:** "Redis 캐시 레이어 추가하고 인덱스 최적화하면 p99 레이턴시 200ms 이하로 떨어뜨릴 수 있어요."

**N**: 같은 내용인데 듣는 사람에 따라 다르게 말하는 거네요!
:::

### 특징 5: 일정을 보수적으로 잡아요

```
일 못하는 개발자:
"3일이면 돼요" → 실제로 2주 걸림

일 잘하는 개발자:
"일주일 잡아주세요. 빨리 끝나면 다른 것도 볼게요" → 5일 만에 완료
```

:::callout[important]
예상보다 빨리 끝나면 영웅이 되고,
예상보다 늦어지면 민폐가 돼요.
일 잘하는 개발자는 버퍼를 남겨둬요.
:::

### 기획자/디자이너가 도울 수 있는 것

:::conversation
**N**: 그럼 저도 개발자분들이 일 잘할 수 있게 도울 수 있는 게 있을까요?

**한입**: 물론이죠!

1. **"왜"를 먼저 설명해주세요** - 요구사항의 배경을 알면 더 좋은 해결책이 나와요
2. **우선순위를 명확히 해주세요** - "다 급해요"는 아무것도 급하지 않은 거예요
3. **변경 사항은 빨리 공유해주세요** - 나중에 알면 재작업이 커져요
4. **기술적 어려움을 이해해주세요** - "그냥 빨리 해주세요"는 신뢰를 깨요
:::

---

:::yama
일 잘하는 개발자는 "왜?"를 묻고, 대안을 제시하고, 눈높이에 맞게 설명해요.
그리고 좋은 협업자를 만나면 더 빛나요!
:::
