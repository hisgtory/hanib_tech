# 일 잘하는 개발자 특징 - 실무 체크리스트 버전

<!-- variant: v2, type: 실무 체크리스트 버전 -->

## 핵심 요약

일 잘하는 개발자는 코딩 실력만 좋은 게 아니라, 커뮤니케이션과 문제 해결 방식이 다르다. PM/디자이너가 알아두면 협업이 수월해진다.

---

## 일 잘하는 개발자 특징 체크리스트

### 커뮤니케이션
- [ ] "안 돼요" 대신 "이렇게 하면 가능해요" 제안
- [ ] 모르는 것은 솔직하게 인정
- [ ] 기술 용어를 쉽게 풀어서 설명
- [ ] 일정 지연 시 미리 공유
- [ ] 질문에 정확하고 간결하게 답변

### 업무 방식
- [ ] 요구사항을 명확히 확인하고 시작
- [ ] 예상 소요 시간을 정확하게 추정
- [ ] 작업을 작은 단위로 나눠서 진행
- [ ] 중간 진행 상황을 주기적으로 공유
- [ ] 완료 후 테스트까지 스스로 진행

### 문제 해결
- [ ] 문제 상황을 정확히 파악
- [ ] 여러 해결책을 비교 분석
- [ ] 장단점을 설명하고 추천안 제시
- [ ] 결정 후 빠르게 실행
- [ ] 결과를 측정하고 개선

---

## 개발자와 소통할 때 체크리스트

### 요청 전 준비
- [ ] 요청 목적이 명확한가?
- [ ] 기대 결과가 구체적인가?
- [ ] 우선순위가 정해져 있는가?
- [ ] 마감 기한이 현실적인가?
- [ ] 참고 자료가 준비되어 있는가?

### 요청 시 포함할 것
- [ ] What: 무엇을 해야 하는지
- [ ] Why: 왜 필요한지
- [ ] When: 언제까지 필요한지
- [ ] How: 제약 조건이 있는지
- [ ] Priority: 얼마나 급한지

### 요청 후 할 것
- [ ] 질문에 신속하게 답변
- [ ] 변경 사항 즉시 공유
- [ ] 중간 점검 일정 잡기
- [ ] 완료 후 피드백 제공

## 개발자에게 하면 좋은 말 vs 피해야 할 말

### 좋은 말 ✓
| 상황 | 예시 |
|------|------|
| 일정 문의 | "이 기능 구현에 얼마나 걸릴 것 같아요?" |
| 가능성 확인 | "기술적으로 가능한지 검토해주실 수 있나요?" |
| 대안 요청 | "시간이 부족하면 어떤 범위까지 가능할까요?" |
| 우선순위 조정 | "이것만 먼저 해주시고 나머지는 다음에 할게요" |

### 피해야 할 말 ✗
| 말 | 문제점 |
|-----|--------|
| "간단한 건데..." | 간단한지 아닌지는 개발자가 판단 |
| "그냥 빨리 해주세요" | 구체적인 기한 제시 필요 |
| "다 되셨죠?" | 진행 상황을 먼저 확인 |
| "왜 이것도 안 돼요?" | 기술적 제약을 이해하려는 자세 필요 |

## 일정 추정 관련 체크리스트

### 개발자의 추정치를 받았을 때
- [ ] 포함된 범위 확인 (기획/디자인/개발/테스트/배포)
- [ ] 버퍼 포함 여부 확인
- [ ] 병렬 작업 가능 여부 확인
- [ ] 의존성 있는 작업 확인
- [ ] 리스크 요소 확인

### 일정이 맞지 않을 때
- [ ] 범위 축소 가능한지 협의
- [ ] 단계별 출시 가능한지 확인
- [ ] 추가 리소스 투입 가능한지 확인
- [ ] 마감 연기 가능한지 확인

- [ ] 수정 기한 합의

## 실무 팁

:::callout[tip]
**"기술 부채"를 이해하세요**

개발자가 "나중에 정리해야 해요"라고 하면:
- 급하게 만들어서 코드가 지저분해졌다는 의미
- 지금은 되지만, 나중에 문제 될 수 있음
- 시간을 들여 정리해야 함

→ 기능 추가만 요청하지 말고, 정리 시간도 배려해주세요
:::

:::callout[warning]
**"쉬운 거 아니에요?"는 금물**

눈에 보이는 건 빙산의 일각입니다:
- 버튼 하나 추가 = UI 변경 + 로직 추가 + API 연동 + 테스트
- 색상 변경 = 디자인 시스템 전체 영향도 확인 필요
- 텍스트 수정 = 다국어 대응, 길이 대응 확인 필요
:::
