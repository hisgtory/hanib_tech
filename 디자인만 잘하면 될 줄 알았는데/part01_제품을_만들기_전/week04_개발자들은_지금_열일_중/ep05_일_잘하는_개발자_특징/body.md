# 일 잘하는 개발자 특징

## 코드만 잘 짜면 될까?

"그 개발자 진짜 일 잘해." 이 말의 의미가 뭘까요? 코딩을 빨리 한다? 버그가 없다? 사실 그것만으론 부족해요.

## 일 잘하는 개발자의 특징

### 1. 질문을 잘 합니다

**안 좋은 경우:**
```
PM: "로그인 기능 만들어주세요"
개발자: "네" (그냥 만듦)
결과: 예상과 다른 결과물
```

**좋은 경우:**
```
PM: "로그인 기능 만들어주세요"
개발자: "소셜 로그인도 필요한가요?
       비밀번호 찾기도 포함인가요?
       자동 로그인 유지는요?"
결과: 기대에 맞는 결과물
```

==일 잘하는 개발자는 먼저 물어봅니다.==

### 2. "안 돼요" 대신 대안을 제시합니다

**안 좋은 경우:**
```
디자이너: "이 애니메이션 넣어주세요"
개발자: "안 돼요"
```

**좋은 경우:**
```
디자이너: "이 애니메이션 넣어주세요"
개발자: "이 방식은 성능 문제가 있어요.
       대신 이렇게 하면 비슷한 효과를 낼 수 있는데,
       이건 어떨까요?"
```

### 3. 예상 일정과 실제가 비슷합니다

**안 좋은 경우:**
- 3일이라고 했는데 2주 걸림
- 항상 "거의 다 됐어요"
- 마감 직전에 갑자기 "안 될 것 같아요"

**좋은 경우:**
- 3일이라고 하면 3일 내외로 완료
- 일정이 밀릴 것 같으면 미리 공유
- 변수가 있으면 미리 말해줌

### 4. 코드를 남이 읽을 수 있게 짭니다

코드는 나 혼자만 보는 게 아니에요.

**안 좋은 코드:**
- 나만 이해할 수 있음
- 6개월 후엔 나도 이해 못 함
- 다른 사람이 수정하기 어려움

**좋은 코드:**
- 변수 이름만 봐도 뭔지 알겠음
- 적절한 주석
- 다른 사람도 쉽게 수정 가능

### 5. 문서화를 합니다

**안 좋은 경우:**
```
새 개발자: "이 기능 어떻게 동작해요?"
기존 개발자: "음... 제가 다 머릿속에 있어서..."
```

**좋은 경우:**
```
새 개발자: "이 기능 어떻게 동작해요?"
기존 개발자: "여기 문서 있어요. 읽어보시고 질문 있으면 말씀해주세요."
```

### 6. 테스트를 합니다

"되는 것 같아요"가 아니라 "확인했어요"라고 말할 수 있어야 해요.

- 기본 기능 테스트
- 예외 상황 테스트
- 다양한 환경에서 테스트

### 7. 커뮤니케이션을 잘 합니다

:::callout[important]
기술력이 뛰어나도 소통이 안 되면 일 잘하는 개발자가 아닙니다.
:::

**좋은 커뮤니케이션:**
- 진행 상황을 주기적으로 공유
- 문제가 생기면 빨리 알림
- 전문 용어 없이 설명할 수 있음
- 다른 직군의 의견을 경청

## 협업 관점에서 보면

### 디자이너와 일할 때

**일 잘하는 개발자:**
- 디자인 의도를 물어봄
- 구현 가능 여부를 빨리 알려줌
- 대안을 제시함
- 디자인 시스템을 이해하려 함

### PM/기획자와 일할 때

**일 잘하는 개발자:**
- 요구사항의 배경을 이해하려 함
- 기술적 제약을 쉽게 설명함
- 일정 추정이 정확함
- 우선순위 조율에 협조적

## 디자이너가 개발자와 협업할 때

### 이런 개발자를 만나면 행운이에요

- 디자인 의도를 물어보는 개발자
- "이렇게 하면 어떨까요?"라고 제안하는 개발자
- 구현 전에 확인하는 개발자
- 일정을 지키려 노력하는 개발자

### 협업을 더 잘하려면

**디자이너도:**
- 왜 이렇게 디자인했는지 설명해주기
- 우선순위 알려주기
- 변경 사항 미리 공유하기
- 개발 상황 이해하려 노력하기

## 정리

:::callout[quote]
"좋은 코드는 컴퓨터가 이해하는 코드가 아니라,
사람이 이해하는 코드다."
- 마틴 파울러
:::

**일 잘하는 개발자의 핵심:**
1. 질문을 잘 함 (요구사항 파악)
2. 대안을 제시함 ("안 돼요"로 끝내지 않음)
3. 일정 예측이 정확함
4. 남이 읽을 수 있는 코드
5. 문서화와 테스트
6. 커뮤니케이션 능력

**결론:**
코딩 실력 + 소통 능력 + 책임감 = 일 잘하는 개발자
