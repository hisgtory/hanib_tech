# 개발자들은 서로의 코드를 어떻게 합쳐요

## "같은 파일을 동시에 수정하면 어떻게 돼요?"

개발자 A가 로그인 화면을 고치고, 개발자 B도 로그인 화면을 고치고 있어요. 둘이 동시에 작업하면 누구 코드가 살아남을까요?

## Git이라는 마법의 도구

개발자들은 **Git**이라는 도구를 써서 코드를 관리해요.

### Google Docs 생각해보세요

여러 명이 동시에 문서를 편집할 수 있죠? Git도 비슷해요.
- 누가 언제 뭘 고쳤는지 기록됨
- 이전 버전으로 되돌릴 수 있음
- 동시 작업 가능

### 단, 코드는 자동 병합이 안 돼요

Google Docs는 실시간으로 합쳐지지만, 코드는 그렇지 않아요.
==같은 줄을 동시에 고치면 "충돌(Conflict)"이 발생해요.==

## 브랜치(Branch)라는 개념

### 나뭇가지처럼 뻗어나가요

[illust: ID="P1-W4-E02-01" DESC="Git 브랜치를 나무로 비유한 다이어그램 - 가운데 굵은 줄기(main), 양쪽으로 뻗어나가는 가지들(feature 브랜치), 완성된 가지가 다시 줄기로 합쳐지는 모습"]

```
main (메인 코드)
  │
  ├── feature/login (A가 작업 중)
  │
  └── feature/signup (B가 작업 중)
```

각 개발자가 자기만의 "가지"에서 작업하고, 완성되면 main에 합쳐요.

### 왜 이렇게 할까요?

1. **독립적 작업**: 내 작업이 남의 작업에 영향 안 줌
2. **안전한 실험**: 잘못되면 가지만 버리면 됨
3. **체계적 관리**: 기능별로 구분되어 추적 용이

## Pull Request (PR) - 합쳐달라는 요청

### 코드 합치기 전 검토 단계

```
개발자 A: "로그인 기능 완성했어요. 합쳐주세요!" (PR 생성)
    ↓
다른 개발자들: 코드 리뷰
    ↓
승인되면: main에 병합(Merge)
```

### PR에서 하는 것들

- 코드 리뷰 (버그 없는지, 잘 짰는지 확인)
- 테스트 통과 여부 확인
- 디자인/기획 의도대로인지 확인

## GitHub - 코드의 집

:::callout[tip]
Git은 도구이고, GitHub은 그 도구로 관리하는 코드를 저장하는 곳이에요.
카카오톡이 앱이라면, 카카오 서버가 메시지를 저장하는 것처럼요.
:::

**GitHub에서 볼 수 있는 것들:**
- 전체 코드
- 누가 언제 뭘 고쳤는지 (커밋 히스토리)
- 현재 진행 중인 작업 (PR 목록)
- 버그나 할 일 (Issues)

## 실제 작업 흐름

```
1. main에서 새 브랜치 생성
   → "feature/로그인-버튼-수정"

2. 브랜치에서 작업
   → 코드 수정, 커밋(저장)

3. PR 생성
   → "로그인 버튼 색상 변경했어요"

4. 코드 리뷰
   → 팀원들이 확인, 피드백

5. 승인 후 Merge
   → main에 반영됨

6. 배포
   → DevOps가 서버에 올림
```

## 충돌(Conflict)이 발생하면

### 같은 줄을 고쳤을 때

```
개발자 A: 버튼 색상을 빨간색으로
개발자 B: 버튼 색상을 파란색으로
    ↓
Git: "둘 중 뭘로 할 건데요?" (충돌!)
```

### 해결 방법

개발자가 직접 선택해요:
- A 코드 선택
- B 코드 선택
- 둘 다 살리기
- 새로운 코드로 대체

## 디자이너가 알아야 할 점

### 1. "PR 올렸어요"의 의미

코드는 완성됐지만, 아직 실서버에 반영된 건 아니에요.
- 리뷰 대기 중
- 테스트 중
- 배포 대기 중

### 2. 일정에 영향을 주는 것들

- **코드 리뷰 시간**: PR 올린 후 바로 반영 안 됨
- **충돌 해결 시간**: 다른 사람이 같은 부분 건드렸으면 추가 시간
- **테스트 시간**: 자동화 테스트 돌리는 시간

### 3. 왜 "다른 작업이랑 겹쳐요"라고 하는지

같은 화면을 여러 개발자가 동시에 수정하면:
- 충돌 가능성 높아짐
- 누가 먼저 머지할지 조율 필요
- 추가 작업 발생 가능

## 정리

:::callout[important]
개발자들은 Git으로 코드를 관리하고, GitHub에 저장해요.
각자 브랜치에서 작업한 후, PR을 통해 리뷰받고 합칩니다.
:::

**핵심:**
- Git = 코드 버전 관리 도구
- GitHub = 코드 저장소
- 브랜치 = 독립적인 작업 공간
- PR = 코드 합치기 전 검토 단계
- "PR 올렸어요" ≠ 바로 반영됨
