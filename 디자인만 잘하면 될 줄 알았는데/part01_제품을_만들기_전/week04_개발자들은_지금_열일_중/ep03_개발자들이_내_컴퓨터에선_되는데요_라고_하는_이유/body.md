# 개발자들이 "내 컴퓨터에선 되는데요" 라고 하는 이유

## 개발자들의 국민 대사

"제 컴퓨터에선 되는데요..." 개발자한테 버그 얘기하면 이 말 한 번쯤 들어보셨죠? 왜 내 컴퓨터에선 안 되는 게 개발자 컴퓨터에선 될까요?

## 환경이 다르기 때문이에요

### 같은 요리, 다른 결과

같은 레시피로 요리해도 결과가 다를 수 있어요.
- 불 세기가 다르면
- 재료 상태가 다르면
- 조리 도구가 다르면

코드도 마찬가지예요. 실행 환경이 다르면 결과가 달라질 수 있어요.

## 환경 차이의 종류

### 1. 운영체제 차이

| 개발자 PC | 사용자 환경 |
|-----------|------------|
| Mac | Windows |
| 최신 버전 | 구 버전 |
| 한국어 설정 | 영어 설정 |

### 2. 브라우저 차이

```
개발자: Chrome 최신 버전으로 테스트 완료!
사용자: Safari로 접속... 안 되네요?
```

브라우저마다 동작 방식이 조금씩 달라요.

### 3. 데이터 차이

**개발자의 테스트 데이터:**
- 깔끔하게 정리된 가짜 데이터
- 모든 필드가 채워져 있음
- 특수문자 없음

**실제 사용자 데이터:**
- 이름에 이모지가 있음 👋
- 전화번호에 하이픈이 있음
- 주소가 100자가 넘음

### 4. 네트워크 차이

**개발자 환경:**
- 빠른 사내 와이파이
- 서버가 바로 옆에

**사용자 환경:**
- 느린 LTE
- 서버가 해외에

### 5. 권한 차이

개발자 계정은 보통 모든 권한이 있어요.
일반 사용자 계정으론 안 보이는 기능이 있을 수 있어요.

## 왜 이런 일이 생길까?

### 1. 모든 환경을 테스트하긴 어려워요

- 브라우저 종류 × 버전 조합만 해도 수십 개
- 기기 종류도 수천 가지
- 모든 경우의 수를 테스트하는 건 불가능에 가까워요

### 2. 개발 환경과 실제 환경이 다를 수밖에 없어요

```
개발 환경: 내 맥북, 최신 크롬, 빠른 네트워크
테스트 환경: 테스트 서버, 제한된 테스트
실제 환경: ???
```

## 이 문제를 줄이려면

### 개발팀이 하는 노력

1. **다양한 환경에서 테스트**
   - 크로스 브라우저 테스트
   - 모바일/데스크톱 테스트
   - 느린 네트워크 시뮬레이션

2. **자동화 테스트**
   - 여러 환경에서 자동으로 테스트 실행
   - 배포 전 체크

3. **모니터링**
   - 실사용자 환경에서 발생하는 에러 추적
   - 어떤 환경에서 문제가 생기는지 분석

## 디자이너가 도울 수 있는 것

### 버그 리포트 시 환경 정보 포함하기

==버그 신고할 때 이 정보가 있으면 해결이 빨라져요:==

- **브라우저**: Chrome 120 / Safari 17 등
- **기기**: iPhone 15 / Galaxy S24 / MacBook 등
- **OS**: iOS 17 / Windows 11 등
- **네트워크**: 와이파이 / LTE 등
- **계정 유형**: 관리자 / 일반 사용자 등

### 예시

```
❌ "로그인이 안 돼요"

✅ "Chrome 120, Windows 11에서
   일반 사용자 계정으로 로그인 시
   '비밀번호 오류' 메시지가 뜹니다.
   Safari에서는 정상 작동해요."
```

## 개발자의 이 말이 들리면

:::callout[tip]
"내 컴퓨터에선 되는데요"는 변명이 아니라 단서예요.
"그럼 뭐가 다른 거지?"로 시작하면 해결이 빨라집니다.
:::

**확인할 것들:**
- 어떤 브라우저/기기에서 안 되나요?
- 특정 계정에서만 안 되나요?
- 특정 데이터에서만 안 되나요?
- 와이파이/LTE 차이가 있나요?

## 정리

:::callout[important]
코드는 같아도 실행 환경이 다르면 결과가 달라질 수 있어요.
"내 컴퓨터에선 되는데요"는 환경 차이를 찾아야 한다는 신호입니다.
:::

**핵심:**
- 브라우저, OS, 네트워크, 데이터, 권한 등이 다를 수 있음
- 모든 환경을 테스트하긴 현실적으로 어려움
- 버그 리포트에 환경 정보를 포함하면 해결이 빨라짐
- "뭐가 다른 거지?"가 해결의 시작점
