# 요청자 특성

## 커뮤니케이션 스타일

### 요청 방식
- **간결함**: 핵심만 짧게 전달 ("프론트 서버 켜줘", "커밋 푸시해")
- **한국어 선호**: 기술 용어도 한국어로 ("브랜치 따서")
- **결과 중심**: 과정보다 결과에 관심

### 피드백 방식
- 문제 발생시 콘솔 에러 직접 공유
- 스크린샷이나 구체적 증상 설명
- 빠른 확인 후 다음 단계 진행

### 선호하는 응답
```
좋음: "완료. localhost:3030 접속하세요"
피하기: 긴 설명 후 결과
```

## 작업 패턴

### 반복적 요청
1. **서버 관리**: "서버 켜줘", "재시작해줘"
2. **Git 작업**: "커밋해줘", "브랜치 따서 푸시"
3. **디버깅**: 에러 메시지 공유 → 해결 요청

### 의사결정 스타일
- 선택지 제시 → 빠른 결정
- 세부사항은 위임 ("알아서 해줘" 스타일)
- 큰 방향만 결정, 구현은 신뢰

### 피드백 루프
```
요청 → 구현 → 테스트 → 피드백 → 수정
(빠른 사이클 선호)
```

## 기술적 선호

### 아키텍처
- 단순한 구조 선호
- 로컬 파일 시스템 기반 (DB 없이)
- CLI 도구 활용

### 도구
- Claude CLI 적극 활용
- Go + React 조합
- Git으로 버전 관리

### 자동화
- 반복 작업 자동화 선호
- Makefile 활용
- 스크립트로 해결

## 콘텐츠 관점

### 콘텐츠 철학
- 비개발자를 위한 개발 협업 가이드
- 스토리텔링 (대화체) 중시
- 실무 적용 가능성 강조

### 품질 기준
- 핵심 메시지 명확
- 감정적 공감 포함
- 실제 활용 가능한 예시

### 변형(Variant) 전략
- 같은 내용 다양한 버전 관리
- 원본 보존 (original)
- 버전별 목적 구분 (v2: 체크리스트, v3: 카드뉴스)

## 협업 팁

### 효과적인 소통
```
1. 결과 먼저 보여주기
2. 문제 있으면 바로 말하기
3. 선택이 필요하면 옵션 제시
4. 긴 설명보다 짧은 확인
```

### 신뢰 구축
```
1. 요청한 것 정확히 수행
2. 추가 작업은 물어보고
3. 에러 발생시 빠른 해결
4. 진행 상황 적절히 공유
```

### 피해야 할 것
```
- 불필요한 설명
- 요청하지 않은 개선
- 결정을 미루는 것
- 느린 응답
```
