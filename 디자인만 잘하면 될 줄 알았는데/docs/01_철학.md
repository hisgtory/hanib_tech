# 프로젝트 철학

## 핵심 비전

"디자인만 잘하면 될 줄 알았는데"는 **비개발자(기획자, 디자이너, 마케터, 창업가)를 위한 개발 협업 가이드북**입니다.

### 왜 이 프로젝트가 필요한가?

1. **지식의 비대칭**: 개발자와 비개발자 사이의 언어와 사고방식 차이
2. **협업의 어려움**: "이거 안 돼요"라는 말 뒤에 숨겨진 맥락을 이해하지 못함
3. **불필요한 갈등**: 서로의 제약조건을 모르기 때문에 발생하는 오해

### 콘텐츠 철학

#### 1. 스토리텔링 우선
- 개념을 직접 설명하지 않고, **대화 형식(conversation)**으로 자연스럽게 전달
- 등장인물: **한입**(경험 많은 선배), **N**(궁금증 많은 신입)
- 독자가 N에 감정이입하며 배움

#### 2. 실무 중심
- 이론보다 **실제 상황에서의 적용**
- "이런 상황에서 이렇게 말하세요" 수준의 구체성
- 체크리스트, 템플릿 등 바로 쓸 수 있는 도구 제공

#### 3. 감정 인정
- 기획 리뷰에서 "까인다"는 느낌
- 개발자가 "안 된다"고 할 때의 좌절감
- 이런 감정들을 인정하고 다루는 방법 제시

#### 4. 핵심 메시지 강조 (Yama)
- 각 에피소드마다 **한 줄로 기억할 핵심**
- 복잡한 내용도 결국 하나의 통찰로 귀결

## HSEPC 모델

콘텐츠 구성의 기본 프레임워크:

| 단계 | 설명 | 예시 |
|------|------|------|
| **H**ook | 관심 끌기 | "이 기능 어디 있지?" 사용자가 앱을 삭제합니다 |
| **S**toryline | 이야기 전개 | 한입과 N의 대화로 개념 설명 |
| **E**scalation | 점진적 심화 | 기본 → 실전 → 고급 기법 |
| **P**ayoff | 핵심 통찰 | "화면이 자꾸 바뀌면 IA를 안 그렸기 때문" |
| **C**onsistency | 일관된 톤 | 친근하고 격려하는 말투 |

## 타겟 독자

### Primary
- 스타트업 초기 멤버 (기획자, 디자이너)
- 개발팀과 협업해야 하는 PM/PO
- 1인 창업가

### Secondary
- 주니어 개발자 (비개발자 시각 이해용)
- 팀 리더 (팀 교육 자료로 활용)

## 차별화 포인트

1. **한국어 최적화**: 한국 스타트업 문화에 맞는 예시
2. **실무 대화 예시**: 실제로 쓸 수 있는 말투와 표현
3. **감정적 공감**: 단순한 정보 전달이 아닌 독자 마음 케어
4. **모듈식 구성**: 필요한 부분만 골라 읽을 수 있는 구조
