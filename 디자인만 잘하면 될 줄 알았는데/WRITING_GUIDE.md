# 콘텐츠 작성 가이드

> "디자인만 잘하면 될 줄 알았는데" 콘텐츠 작성자를 위한 가이드

---

## 프로젝트 구조

```
디자인만 잘하면 될 줄 알았는데/
├── part{NN}_{제목}/
│   └── week{NN}_{제목}/
│       ├── intro.md              # 주차 소개
│       └── ep{NN}_{제목}/
│           ├── body.md           # 본문 (상세 설명) ← 실제 사용 버전
│           ├── conversation.md   # 대화형 콘텐츠
│           ├── yama.md           # 핵심 메시지 (3줄 요약)
│           └── variants/         # 대안 버전들
│               ├── v1/
│               │   ├── body.md
│               │   ├── conversation.md
│               │   └── yama.md
│               └── v2/
│                   ├── body.md
│                   ├── conversation.md
│                   └── yama.md
```

---

## 파일별 작성법

### 1. yama.md (핵심 메시지)

에피소드의 핵심을 **3줄 이내**로 요약합니다. 독자가 기억해야 할 가장 중요한 메시지입니다.

```markdown
:::yama
첫 번째 핵심 메시지.
두 번째 핵심 메시지.
세 번째 핵심 메시지.
:::
```

**예시:**
```markdown
:::yama
리텐션이 성장의 핵심이에요.
새 유저를 데려오기 전에, 기존 유저가 머무는지 확인하세요.
Aha moment를 찾고 빨리 도달하게 하세요.
:::
```

**작성 팁:**
- 독자가 이것만 읽어도 핵심을 알 수 있어야 함
- 명령형/권유형으로 작성 ("~하세요", "~해야 해요")
- 추상적 표현보다 구체적 행동 지침

---

### 2. conversation.md (대화형 콘텐츠)

**N**(비개발자)과 **한입**(설명자) 간의 대화 형식입니다.

```markdown
:::conversation
**N**: 질문이나 상황 설명

**한입**: 답변이나 설명

**N**: 후속 질문

**한입**: 추가 설명
:::
```

**예시:**
```markdown
:::conversation
**N**: 마케팅 비용은 쓰는데 유저가 안 늘어. 왜 그래?

**한입**: 들어오는 만큼 나가서 그래. 리텐션이 안 좋은 거야.

**N**: 리텐션?

**한입**: 사용자가 계속 쓰는 비율. 100명 가입하고 10명만 남으면 리텐션 10%야.

**N**: 알겠어. 새 유저 데려오기 전에 기존 유저부터.

**한입**: 맞아. 리텐션 나쁘면 마케팅 비용이 다 새는 거야.
:::
```

**작성 팁:**
- N은 항상 비개발자의 시점에서 질문
- 한입은 친근하고 쉬운 언어로 설명
- 비유와 일상 예시 적극 활용
- 한 턴당 1-2문장 (길어지면 지루해짐)
- 마지막은 핵심 메시지로 마무리

---

### 3. body.md (본문)

가장 상세한 설명을 담는 파일입니다. 마크다운 확장 문법을 활용합니다.

**기본 구조:**
```markdown
# 에피소드 제목

## "문제 상황이나 질문"

도입부 설명

## 핵심 개념

### 세부 주제 1

설명 내용

### 세부 주제 2

설명 내용

## 정리

:::callout[important]
핵심 정리 메시지
:::

**핵심:**
- 포인트 1
- 포인트 2
- 포인트 3
```

---

## 마크다운 확장 문법

### 콜아웃 (Callout)

중요한 내용을 강조할 때 사용합니다.

```markdown
:::callout[tip]
도움이 되는 팁이나 조언
:::

:::callout[warning]
주의해야 할 내용
:::

:::callout[important]
반드시 기억해야 할 핵심 내용
:::
```

**렌더링 예시:**

| 타입 | 용도 |
|------|------|
| `tip` | 유용한 팁, 추천 사항 |
| `warning` | 주의 사항, 실수하기 쉬운 부분 |
| `important` | 핵심 메시지, 반드시 기억할 내용 |

---

### 하이라이트

특별히 강조할 문장에 사용합니다.

```markdown
==이 문장은 하이라이트됩니다.==
```

**사용 예시:**
```markdown
==리텐션은 복리처럼 작동해요. 작은 차이가 큰 격차를 만들어요.==
```

**작성 팁:**
- 에피소드당 1-2개만 사용 (남발하면 효과 감소)
- 독자가 반드시 기억해야 할 한 문장에 사용
- 너무 긴 문장보다는 짧고 임팩트 있게

---

### 코드 블록

개념을 시각적으로 정리할 때 사용합니다. (실제 코드가 아님)

```markdown
```
Day 1 리텐션: 가입 다음 날 접속 비율
Day 7 리텐션: 가입 7일 후 접속 비율
Day 30 리텐션: 가입 30일 후 접속 비율
```
```

**활용 예시:**
- 수치 비교
- 단계별 프로세스
- 전/후 비교
- 공식이나 계산

---

### 일러스트 지시

일러스트레이터에게 그림을 요청할 때 사용합니다.

```markdown
[illust: "일러스트 설명"]
```

**예시:**
```markdown
[illust: "한입이 머리를 쥐어싸고 있는 그림"]

[illust: "N이 노트북 앞에서 고민하는 모습"]

[illust: "밑 빠진 독에 물 붓는 장면 - 리텐션 비유"]
```

**작성 팁:**
- 등장인물: 한입, N
- 상황/동작을 구체적으로 묘사
- 비유를 시각화할 때 어떤 개념인지 주석 추가
- 감정 표현이 필요하면 명시 (당황, 기쁨, 깨달음 등)

**활용 케이스:**
- 개념 설명을 위한 비유 시각화
- 캐릭터의 감정 표현
- 대화 흐름 중간에 상황 묘사
- 체크리스트나 단계별 과정 아이콘

## 톤 & 매너

### 기본 원칙

1. **친근하게**: 존댓말 사용, 딱딱한 표현 피하기
2. **쉽게**: 전문 용어 사용 시 바로 설명 추가
3. **구체적으로**: 추상적 설명보다 예시 중심

### 좋은 예시 vs 피할 예시

| 피하세요 | 이렇게 쓰세요 |
|----------|---------------|
| "리텐션 최적화가 필요합니다" | "기존 유저가 머무는지 확인하세요" |
| "스케일링 이슈가 발생할 수 있습니다" | "사용자가 늘면 서버가 터져요" |
| "UX 개선을 통한 전환율 향상" | "사용하기 쉽게 만들면 더 많은 사람이 가입해요" |

### 비유 활용

복잡한 개념은 일상적인 비유로 설명합니다.

```markdown
**한입**: 밑 빠진 독이야. 아무리 물 부어도 새면 안 차잖아.
```

```markdown
**한입**: 요리할 때 레시피 보고 따라 하듯이, 개발자들도 프레임워크 규칙 따라서 코드 짜는 거야.
```

- **N (Non-developer)**: 디자이너, 기획자, 마케터 등 비개발자
- 개발 배경 지식이 없다고 가정

## variants 폴더 사용법

같은 에피소드를 다른 버전으로 작성할 때 사용합니다.

### 구조

```
ep01_린_스타트업/
├── body.md              # 현재 실제 사용 버전
├── conversation.md
├── yama.md
└── variants/
    ├── v1/              # 버전 1
    │   ├── body.md
    │   ├── conversation.md
    │   └── yama.md
    ├── v2/              # 버전 2
    │   ├── body.md
    │   ├── conversation.md
    │   └── yama.md
    └── v3/
        ├── body.md
        ├── conversation.md
        └── yama.md
```

### 핵심 원칙

1. **상위 폴더 = 실제 사용 버전**: 에피소드 폴더 바로 아래 파일들이 현재 서비스에 반영되는 버전
2. **variants 내부 = 동일 구조**: 각 버전 폴더(v1, v2...)는 상위와 똑같이 body.md, conversation.md, yama.md를 가짐
3. **버전 승격**: 특정 버전을 실제로 쓰려면 해당 variants/vN/ 내용을 상위로 복사

### 워크플로우 예시

```bash
# 새 버전 작성
mkdir variants/v2
# v2 폴더에 body.md, conversation.md, yama.md 작성

# v2를 실제 버전으로 승격
cp variants/v2/* ./
```

### 사용 케이스

- A/B 테스트용 콘텐츠
- 다른 톤앤매너 시도
- 타겟 독자층별 버전
- 이전 버전 백업

## 자주 하는 실수

### 1. 태그 닫기 누락
```markdown
# 잘못된 예
:::yama
핵심 메시지
# :::를 빼먹음

# 올바른 예
:::yama
핵심 메시지
:::
```

### 2. 하이라이트 남발
```markdown
# 잘못된 예
==이것도 중요하고== ==저것도 중요하고== ==다 중요해요==

# 올바른 예
여러 중요한 내용 중에서 ==가장 핵심만 하이라이트==합니다.
```

### 3. 대화가 너무 김
```markdown
# 잘못된 예
**한입**: 프론트엔드는 사용자가 직접 보고 상호작용하는 부분이에요.
버튼을 누르면 반응하고, 스크롤하면 내용이 나오고, 폼에 입력하면
데이터가 전송되는 그런 모든 것들이요. HTML, CSS, JavaScript 같은
기술을 사용하고... (계속)

# 올바른 예
**한입**: 프론트엔드는 눈에 보이는 부분이야. 버튼, 화면, 입력창 같은 거.
```
