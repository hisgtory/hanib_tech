# 디자인 리뷰와 가능성 검토 (완성)

"이 디자인 구현 가능한가요?"
"이건 너무 복잡해서 개발 기간이..."
"네이티브에서는 이렇게 안 돼요."

디자인 리뷰에서 자주 나오는 피드백들이에요.

## 디자인 리뷰는 왜 필요할까요?

### 1. 기술적 제약 사항 파악
- 플랫폼별 한계 확인
- 개발 공수 예측

### 2. 비즈니스 목표 정렬
- 디자인이 제품 목표에 부합하는지
- 사용자 가치 전달 여부

### 3. 예상치 못한 문제 발견
- 접근성 문제
- 성능 이슈
- UX 결함

## 디자인 리뷰의 참여자들

### 1. 개발자
- 기술적 구현 가능성 검토
- 예상 개발 시간 산정

### 2. 기획자
- 비즈니스 요구사항 충족 확인
- 사용자 시나리오 검증

### 3. QA
- 테스트 가능성 검토
- 예상 버그 포인트 파악

### 4. 이해관계자
- 비즈니스 가치 확인
- 일정 및 비용 검토

## 가능성 검토 체크리스트

- 플랫폼 지원 여부 (iOS, Android, Web)
- 네이티브 UI 컴포넌트 사용 가능 여부
- 애니메이션 성능 이슈
- 서드파티 라이브러리 필요 여부
- API 연동 가능 여부
- 접근성 가이드라인 충족

## 효과적인 디자인 리뷰 진행 방법

### 1. 사전 자료 공유
- 디자인 파일 링크
- 화면 플로우 문서
- 기획 의도 설명

### 2. 구조화된 피드백
- 화면별로 순차적 리뷰
- 문제점은 구체적으로 지적
- 대안 제시

### 3. 결과 문서화
- 주요 피드백 정리
- 액션 아이템 도출
- 담당자 및 일정 확정

## 개발자가 "안 돼요"라고 할 때

<hl>정말 안 되는 것인지, 시간이 오래 걸리는 것인지 파악해야 해요.</hl>

### 1. 기술적 불가능
- 플랫폼 제약
- 보안 정책 위배

### 2. 시간/비용 문제
- 커스텀 개발 필요
- 서드파티 라이브러리 미지원

### 3. 대안 찾기
- 디자인 수정
- 단계적 구현
- 다른 기술 활용

## 리뷰 후 디자인 수정 팁

### 1. 우선순위 정하기
- 필수 수정 사항
- 권장 수정 사항
- Nice to have

### 2. 시간 협상
- 수정 범위에 따른 일정 조정
- MVP 범위 축소

### 3. 버전 관리
- 수정 전/후 버전 보관
- 변경 사항 문서화

## 가능성 검토 도구 활용

- **Principle**: 복잡한 인터랙션 프로토타이핑
- **Lottie**: 애니메이션 구현 테스트
- **ProtoPie**: 센서 기반 인터랙션 검증
- **Origami Studio**: Facebook 개발 프로토타이핑 툴

프로토타입으로 미리 검증하면 개발 단계에서의 수정을 줄일 수 있어요.

<callout>
"가능한 것과 불가능한 것의 차이는
단지 두 글자일 뿐이다."
- 나폴레옹 보나파르트
</callout>

처음부터 불가능하다고 포기하지 말고, 함께 가능한 방법을 찾아보세요.

## Episode 2-9. 디자인 리뷰 후에 '가능성 검토'를 안 했을 때 일어나는 참사

디자인 리뷰에서 "완벽해요! 바로 개발 들어가죠!"라고 했는데, 개발 시작 일주일 후...

"아... 이 인터랙션이 iOS에서 안 되네요."
"이 애니메이션 구현하려면 한 달은 더 걸릴 것 같아요."
"Android에서는 이 UI가 권장사항에 어긋나요."

### 가능성 검토란?

디자인이 **기술적으로, 비즈니스적으로, 일정적으로** 실현 가능한지 미리 확인하는 과정입니다.

예쁜 디자인도 중요하지만, 실제로 구현할 수 있는지가 더 중요해요!

### 가능성 검토의 3단계

**1단계: 기술적 가능성**
- 플랫폼별 제약사항 확인
- 성능에 미치는 영향 평가
- 필요한 라이브러리/기술 스택 검토

**2단계: 일정/비용 평가**
- 예상 개발 시간 산정
- 추가 리소스 필요 여부
- 우선순위에 따른 단계별 구현 계획

**3단계: 대안 준비**
- Plan A가 안 될 때의 Plan B
- 기술적 제약 시 디자인 수정안
- 일정 단축을 위한 간소화 버전

### 실제 케이스 스터디

**케이스 1: "이 카드 뒤집기 애니메이션 구현해주세요"**

❌ **가능성 검토 없이:**
- 디자인 확정 → 개발 시작 → "iOS에서 성능 이슈 발견" → 디자인 재작업

✅ **가능성 검토 후:**
- 디자인 단계에서 프로토타입 제작 → 성능 테스트 → iOS용 대안 디자인 준비

**케이스 2: "소셜 로그인 후 자동으로 프로필 정보 가져오기"**

❌ **가능성 검토 없이:**
- 기획 확정 → 개발 착수 → "개인정보 정책상 불가" → 전체 플로우 재설계

✅ **가능성 검토 후:**
- 법무팀 사전 검토 → 개인정보 정책 확인 → 사용자 동의 단계 추가한 플로우 설계

### 가능성 검토 체크리스트

**기술적 측면:**
- [ ] 각 플랫폼(iOS/Android/Web)에서 지원 가능?
- [ ] 성능에 부정적 영향 없음?
- [ ] 필요한 외부 라이브러리가 안정적임?
- [ ] 접근성 가이드라인 준수 가능?

**비즈니스 측면:**
- [ ] 법적/정책적 제약사항 없음?
- [ ] 보안 요구사항 충족 가능?
- [ ] 유지보수 비용이 합리적임?
- [ ] ROI(투자 대비 수익)가 명확함?

**일정 측면:**
- [ ] 예상 개발 기간이 현실적임?
- [ ] 의존성 있는 다른 작업들과 충돌 없음?
- [ ] QA 및 배포 일정까지 고려됨?

### 가능성 검토를 효과적으로 하는 방법

**1. 프로토타입 활용**
복잡한 인터랙션은 Figma의 Smart Animate, ProtoPie, Principle 등으로 미리 검증

**2. 개발팀과의 사전 논의**
"이런 기능을 고려 중인데 어떻게 생각하세요?"라고 미리 물어보기

**3. 단계별 접근**
한 번에 완벽하게 구현하려 하지 말고, 단계별로 발전시킬 계획 세우기

<hl>가능성 검토는 꿈을 깨는 작업이 아니라, 꿈을 현실로 만드는 작업이에요!</hl>

## Episode 2-10. 디자인 리뷰에서 '이건 개발에서 안 돼요'라고 듣지 않는 법

"이건 구현하기 어려워요."
"개발 기간이 너무 길어질 것 같아요."
"기술적으로 불가능해요."

이런 말을 들으면 기분이 어떠세요? 아마 좌절감이 들거나, 개발팀과 갈등이 생길 수 있어요.

### 왜 '안 돼요'라는 말을 듣게 될까?

**1. 기술적 이해 부족**
디자이너나 기획자가 기술적 제약을 모르고 설계

**2. 커뮤니케이션 부족**
개발팀과 사전 논의 없이 디자인 진행

**3. 일정 압박**
촉박한 일정으로 인해 복잡한 구현 기피

**4. 경험 부족**
비슷한 케이스를 구현해본 적이 없어서 어려워함

### 사전에 '안 돼요'를 방지하는 방법

**1. 개발팀과 친해지기**
- 정기적인 1:1 미팅
- 개발 프로세스에 대한 이해
- 각 플랫폼별 특성 학습

**2. 기술 트렌드 파악하기**
- iOS/Android 디자인 가이드라인 숙지
- 새로운 기술 동향 관심
- 경쟁사 앱의 기술적 구현 방식 분석

**3. 단계적 설계**
- MVP → 확장 버전으로 계획
- "1차에서는 이렇게, 2차에서는 이렇게" 접근
- 핵심 기능 우선 구현 후 점진적 개선

### 실전 대화 기법

❌ **잘못된 접근:**
"이 디자인대로 만들어주세요."

✅ **올바른 접근:**
"이런 사용자 경험을 만들고 싶은데, 어떤 방법이 좋을까요?"

❌ **잘못된 질문:**
"이거 언제까지 가능해요?"

✅ **올바른 질문:**
"이 기능의 복잡도는 어느 정도이고, 어떤 부분이 가장 까다로울까요?"

### 개발 제약을 극복하는 창의적 방법들

**케이스 1: 복잡한 애니메이션**
- 제약: 성능 이슈로 부드럽지 않음
- 해결: Lottie 파일 활용 또는 더 단순한 모션으로 대체

**케이스 2: 커스텀 UI 컴포넌트**
- 제약: 개발 시간이 오래 걸림
- 해결: 기존 네이티브 컴포넌트를 창의적으로 활용

**케이스 3: 실시간 동기화**
- 제약: 서버 리소스 부담
- 해결: 사용자가 새로고침하거나 중요한 순간에만 동기화

### '안 돼요' 대신 들을 수 있는 말들

- "이런 방식으로 하면 가능할 것 같아요"
- "1차에서는 이렇게, 2차에서는 더 발전시켜 봅시다"
- "이 부분만 조금 수정하면 구현할 수 있어요"
- "비슷하지만 더 간단한 대안이 있어요"

### 개발팀과 Win-Win 관계 만들기

**개발자 입장 이해하기:**
- 코드 품질과 유지보수성 고려
- 일정과 기술적 안정성 사이의 균형
- 새로운 기술 도입에 따른 리스크

**디자이너/기획자가 할 수 있는 것:**
- 기술적 제약 안에서의 창의적 해결책 찾기
- 사용자 가치와 구현 복잡도 사이의 우선순위 정하기
- 개발팀의 의견을 존중하면서도 사용자 경험 타협하지 않기

<hl>기억하세요. '안 돼요'는 끝이 아니라 더 나은 해결책을 찾기 위한 시작점이에요!</hl>
