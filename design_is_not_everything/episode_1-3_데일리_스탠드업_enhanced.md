# 에피소드 1-3: 데일리 스탠드업에서 할 말 준비하기 - 15분의 마법

## 🎯 이 글을 읽으면 얻게 되는 것
- 데일리 스탠드업의 진짜 목적 이해
- 효과적인 업데이트 방법 습득
- 블로커 공유하는 요령
- 지루한 회의를 생산적으로 바꾸는 방법

## 🌅 매일 아침, 서서 하는 회의

### 왜 서서 할까?
```
🏃 스탠드업의 철학

앉아서 하면:
• 평균 시간: 45분
• 집중도: 낮음
• 잡담 확률: 80%

서서 하면:
• 평균 시간: 15분
• 집중도: 높음
• 핵심만: 100%

"서 있으면 빨리 끝내고 싶어져요" 
- 모든 개발자의 속마음
```

### 데일리 스탠드업의 진짜 목적
```
✅ 올바른 목적
• 팀 동기화 (Sync)
• 블로커 조기 발견
• 도움 요청/제공
• 오늘의 우선순위 정렬

❌ 잘못된 목적
• 상사에게 보고
• 업무 감시
• 성과 자랑
• 문제 해결 (너무 길어짐)
```

## 📝 3가지 질문의 법칙

### 기본 템플릿
```
🎤 15초 스피치 구조

1. 어제 한 일 (Yesterday)
   "어제 로그인 API 개발을 완료했습니다"

2. 오늘 할 일 (Today)
   "오늘은 회원가입 화면 작업할 예정입니다"

3. 블로커 (Blockers)
   "디자인 가이드가 필요한데 언제 받을 수 있을까요?"

시간 배분:
• 1인당: 1-2분
• 10명 팀: 15분 이내
• 질문/답변: 회의 후 따로
```

### 역할별 업데이트 예시

#### 디자이너의 스탠드업
```
💬 디자이너 업데이트 예시

월요일:
"어제: 사용자 인터뷰 정리 완료
오늘: 온보딩 플로우 와이어프레임 작업
블로커: 없음"

화요일:
"어제: 와이어프레임 초안 완성
오늘: 개발팀과 실현 가능성 검토
블로커: A/B 테스트 데이터 필요"

금요일:
"어제: 디자인 QA 진행
오늘: 다음 스프린트 디자인 준비
블로커: 색상 팔레트 최종 결정 필요"
```

#### 개발자의 스탠드업
```
💬 개발자 업데이트 예시

Junior 개발자:
"어제: 로그인 폼 유효성 검사 구현
오늘: 에러 메시지 처리 추가
블로커: CORS 에러 해결 중, 도움 필요"

Senior 개발자:
"어제: 결제 모듈 리팩토링, PR 리뷰 3건
오늘: 배포 파이프라인 설정
블로커: AWS 권한 필요"

풀스택 개발자:
"어제: API 엔드포인트 5개 완성
오늘: 프론트엔드 연동 테스트
블로커: 목업 데이터 형식 확인 필요"
```

#### PM/PO의 스탠드업
```
💬 PM/PO 업데이트 예시

"어제: 스테이크홀더 미팅, 요구사항 정리
오늘: 백로그 우선순위 재조정
블로커: 마케팅팀 피드백 대기 중"

"어제: A/B 테스트 결과 분석
오늘: 다음 스프린트 계획 준비
블로커: 리소스 할당 결정 필요"
```

## 🚫 블로커 공유의 기술

### 좋은 블로커 vs 나쁜 블로커
```
✅ 좋은 블로커 공유
• 구체적: "결제 API 문서가 없어서 진행 불가"
• 해결 가능: "디자인 파일 접근 권한 필요"
• 시간 명시: "오늘 3시까지 필요"
• 대안 제시: "임시로 목업 데이터로 진행 중"

❌ 나쁜 블로커 공유
• 모호함: "뭔가 안 돼요"
• 비난: "XX님이 안 줘서..."
• 변명: "제가 못한 게 아니라..."
• 해결 불가: "서버가 구려서..."
```

### 블로커 카테고리
```
🚧 블로커 유형별 대응

1. 기술적 블로커
   예: "빌드 에러 발생"
   대응: 팀원 도움 요청

2. 의존성 블로커
   예: "API 완성 대기 중"
   대응: 우선순위 조정

3. 정보 블로커
   예: "요구사항 불명확"
   대응: PO와 즉시 확인

4. 리소스 블로커
   예: "테스트 서버 없음"
   대응: 대체 방안 모색

5. 외부 블로커
   예: "클라이언트 승인 대기"
   대응: 다른 작업 진행
```

## 💡 효과적인 스탠드업 팁

### 준비하기
```
📋 스탠드업 전 체크리스트

5분 전 준비:
□ 어제 작업 내용 정리
□ 오늘 할 일 우선순위
□ 블로커 리스트 확인
□ 질문 사항 메모
□ 지라/노션 티켓 확인

Pro Tip:
• 매일 퇴근 전 내일 스탠드업 메모
• 주요 성과 숫자로 표현
• 스크린샷/데모 준비
```

### 말하기 기술
```
🎙️ 임팩트 있는 발표법

숫자로 말하기:
❌ "로그인 기능 개발 중"
✅ "로그인 API 3개 중 2개 완성 (70%)"

결과 중심으로:
❌ "코드 작성했습니다"
✅ "회원가입 속도 2초 → 0.5초 개선"

팀 기여 강조:
❌ "제 일 끝났어요"
✅ "프론트엔드 연동 준비 완료했습니다"

간결하게:
❌ "어제 아침에 출근해서 커피 마시고..."
✅ "결제 모듈 완성, PR 대기 중"
```

## 🌍 원격 스탠드업

### 비대면 스탠드업 도구
```
💻 원격 스탠드업 플랫폼

동기식 (Sync):
• Zoom/Meet: 화상 스탠드업
• Slack Huddle: 음성 전용
• Gather: 가상 오피스

비동기식 (Async):
• Slack 봇: 자동 질문/수집
• Notion: 일일 업데이트 페이지
• Loom: 비디오 메시지
• Stand-up Bot: 자동화 도구

하이브리드:
• 오전: 비동기 텍스트 업데이트
• 필요시: 동기 화상 회의
```

### 시차가 있는 팀
```
🌏 글로벌 팀 스탠드업

문제:
• 서울 오전 10시 = SF 오후 6시
• 실시간 미팅 불가능

해결책:
1. 비동기 스탠드업
   - 각자 시간에 Slack 업데이트
   - 24시간 내 전원 확인

2. 로테이션 시간
   - 월/수: 아시아 친화적
   - 화/목: 미국 친화적

3. 녹화 공유
   - 핵심 내용 녹화
   - 자막 추가
   - 요약본 작성
```

## 📊 스탠드업 안티패턴

### 피해야 할 행동들
```
🚫 스탠드업 킬러

1. 스토리텔러 👎
   "그래서 제가 어제 코드를 보다가..."
   → 핵심만 간단히

2. 문제 해결사 👎
   "그건 이렇게 하면 되는데..."
   → 회의 후 따로 논의

3. 침묵 지키기 👎
   "특별한 거 없습니다"
   → 작은 진전도 공유

4. 변명 대장 👎
   "못한 이유는..."
   → 솔직한 현황 공유

5. 지각 대장 👎
   매번 5분씩 늦음
   → 팀 시간 존중
```

### 건강한 스탠드업 문화
```
✨ 좋은 스탠드업 만들기

심리적 안전감:
• 실패 공유 OK
• 모르는 것 질문 OK
• 도움 요청 OK

포용적 환경:
• 신입도 편하게 발언
• 영어/한국어 병행 OK
• 기술 용어 설명

지속적 개선:
• 월 1회 스탠드업 회고
• 형식 실험 (음악, 게임)
• 피드백 반영
```

   • 😊 컨디션 최고

## 📈 스탠드업 효과 측정

### KPI 설정
```
📊 스탠드업 성과 지표

정량적 지표:
• 평균 소요 시간: 15분 이하
• 참석률: 95% 이상
• 블로커 해결 시간: 4시간 이내
• 액션 아이템 완료율: 80%

정성적 지표:
• 팀 소통 만족도
• 정보 공유 수준
• 협업 개선도
• 문제 조기 발견율

측정 도구:
• 주간 설문조사
• 월간 회고
• 분기별 팀 건강도 체크
```
