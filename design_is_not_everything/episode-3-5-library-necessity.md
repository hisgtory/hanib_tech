# Episode 3-5. 라이브러리 없이 코딩하면 지는 이유

## Hook: 성현이의 일주일

오늘도 커피를 한 모금 마시며 코드를 작성하던 개발자 성현이가 있었습니다. 회사에서 새로운 프로젝트를 시작하게 된 성현이는 "날짜 선택기가 필요해"라는 요구사항을 받게 되었죠. '그까짓거 간단하네!'라고 생각한 성현이는 직접 날짜 선택기를 만들기 시작했습니다.

하지만 막상 만들기 시작하니 생각보다 고려해야 할 것들이 많았습니다. 달력 UI를 어떻게 그릴까? 월말과 월초의 날짜를 어떻게 처리할까? 윤년은 어떻게 계산할까? 사용자가 날짜를 클릭했을 때 어떻게 반응할까? 키보드 입력은 어떻게 처리할까?

하루, 이틀, 삼일... 성현이는 계속해서 날짜 선택기를 만들고 있었습니다. 그런데 옆에서 이를 지켜보던 시니어 개발자 지원이가 다가와 물었습니다.

"성현이, 혹시 날짜 선택기 직접 만들고 있어?"
"네! 거의 다 만들었어요. 윤년 처리만 하면 돼요!"
"이미 잘 만들어진 라이브러리가 있는데 왜 바퀴의 재발명을 하고 있어?"

## 바퀴의 재발명을 하지 않는 이유

지원이는 성현이에게 '바퀴의 재발명(Reinventing the wheel)'이라는 개념을 설명해주었습니다. 이는 이미 잘 만들어진 해결책이 있음에도 불구하고, 처음부터 다시 만드는 비효율적인 행동을 말합니다. 마치 이미 바퀴가 발명되어 사람들이 잘 사용하고 있는데, 또 다시 바퀴를 발명하려고 하는 것과 같은 상황이죠.

개발자들은 '바퀴의 재발명'을 하지 않습니다. 이는 이미 누군가 만들어 놓은 것을 다시 만드는 비효율적인 행동을 의미합니다.

**예를 들어:**
- 날짜 계산 기능이 필요할 때 → moment.js, date-fns 같은 라이브러리 사용
- 차트를 그려야 할 때 → Chart.js, D3.js 같은 라이브러리 사용
- HTTP 요청을 보내야 할 때 → axios, fetch API 사용

## 라이브러리란?

라이브러리(Library)는 자주 사용되는 기능들을 미리 만들어 놓은 코드의 모음입니다. 마치 도서관에서 필요한 책을 빌려 쓰듯이, 개발자들은 필요한 기능을 라이브러리에서 가져다 사용합니다.

## 라이브러리를 사용하는 이유

### 1. 시간 절약
이미 만들어진 코드를 사용하므로 개발 시간이 단축됩니다

### 2. 검증된 코드
수많은 사람들이 사용하고 테스트한 안정적인 코드입니다

### 3. 전문성
해당 분야의 전문가가 만든 최적화된 코드를 사용할 수 있습니다

### 4. 유지보수
라이브러리는 지속적으로 업데이트되고 버그가 수정됩니다

## 프레임워크 vs 라이브러리

라이브러리와 프레임워크는 자주 혼동되는 개념입니다. 그러나 명확한 차이가 있습니다:

**라이브러리:**
- 필요한 기능만 가져다 쓰는 '도구'입니다
- 개발자가 통제권을 가집니다 ("You call library")
- 예: jQuery, Lodash, axios

**프레임워크:**
- 전체적인 구조와 규칙을 제공하는 '틀'입니다
- 프레임워크가 통제권을 가집니다 ("Framework calls you")
- 예: React, Angular, Vue.js, Spring

## 라이브러리 선택 기준

실무에서 라이브러리를 선택할 때 고려해야 할 사항들:

- **인기도와 커뮤니티**: GitHub 스타 수, 다운로드 수 확인
- **문서화**: 공식 문서가 잘 정리되어 있는가?
- **유지보수**: 마지막 업데이트가 언제인가?
- **호환성**: 현재 프로젝트와 잘 맞는가?
- **성능**: 필요한 성능 요구사항을 충족하는가?
- **크기**: 라이브러리의 크기가 프로젝트에 부담이 되지 않는가?
- **라이센스**: 프로젝트에 적합한 라이센스인가?

## 📝 구체적인 개선 방안

### 현재 강점
- 성현이와 지원이의 스토리로 라이브러리 필요성을 자연스럽게 설명
- 바퀴의 재발명 개념을 명확하게 정의
- 라이브러리 vs 프레임워크 차이를 통제권 관점에서 구분

### 보완이 필요한 부분

1. **디자이너/기획자 관점에서의 라이브러리 이해**
   - 현재: 개발자 중심 설명
   - 개선: 비개발자가 라이브러리를 이해해야 하는 이유
   ```
   예시 추가:
   - 디자인 시스템도 일종의 라이브러리 (UI 컴포넌트 재사용)
   - 아이콘 라이브러리 선택이 브랜드 일관성에 미치는 영향
   - 차트 라이브러리에 따른 데이터 시각화 제약사항
   - 폰트 라이브러리와 웹 성능의 관계
   ```

2. **팀 협업 시 발생하는 라이브러리 관련 갈등**
   - 현재: 선택 기준만 제시
   - 개선: 실제 프로젝트에서 겪는 이해관계 충돌
   ```
   갈등 시나리오:
   - 디자이너: "이 UI 컴포넌트가 예쁘네요"
   - 개발자: "그 라이브러리는 용량이 너무 커서..."
   - 기획자: "경쟁사는 이런 차트를 쓰던데"
   - 해결: 기능-성능-디자인의 균형점 찾기
   ```

3. **라이브러리 선택이 프로젝트에 미치는 영향**
   - 현재: 기술적 기준만 언급
   - 개선: 비즈니스 임팩트와 사용자 경험 측면
   ```
   영향 분석:
   - 로딩 속도: 라이브러리 크기가 사용자 이탈률에 미치는 영향
   - 유지보수 비용: 라이브러리 업데이트에 따른 개발 공수
   - 브랜드 일관성: UI 라이브러리 선택이 브랜드 인식에 미치는 영향
   - 접근성: 라이브러리의 접근성 지원 수준 고려
   ```

4. **라이브러리 의존성의 위험성**
   - 현재: 장점만 강조
   - 개선: 라이브러리 사용의 잠재적 위험과 대응방안
   ```
   위험 요소:
   - 라이브러리 개발 중단 시 대응 방안
   - 보안 취약점 발견 시 업데이트 계획
   - 라이선스 변경으로 인한 법적 위험
   - 벤더 록인(Vendor Lock-in) 방지 전략
   ```

5. **노코드/로우코드 도구와의 관계**
   - 현재: 전통적인 개발만 다룸
   - 개선: 현대적인 개발 환경에서의 라이브러리 개념 확장
   ```
   확장된 관점:
   - Figma 플러그인도 디자이너를 위한 "라이브러리"
   - 노션 템플릿, 에어테이블 베이스도 재사용 가능한 "라이브러리"
   - 마케팅 자동화 툴의 템플릿 시스템
   - 비개발자도 라이브러리 개념을 활용할 수 있는 영역들
   ```
