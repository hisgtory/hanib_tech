# Episode 2-8: 컬러 코드로 폰트 크기까지 정해줘야 하는 이유 [Exclusive]

"이거 그냥 대충 파란색으로 해주세요." 디자이너가 개발자에게 이렇게 말했다가 나온 결과물을 보고 당황한 적 있으신가요? 아니면 개발자가 "폰트 크기 좀 키워주세요"라고 했더니 "얼마나요?"라고 되물어 본 적은요? 이런 상황들이 반복되면서 우리 팀은 깨달았어요. 디자인과 개발 사이에는 '번역기'가 필요하다는 걸 말이에요.

## 1. 왜 '파란색'은 통하지 않을까?

디자이너가 생각하는 '파란색'과 개발자가 구현하는 '파란색'은 다를 수밖에 없어요. 디자이너는 브랜드 컬러, 사용자 경험, 접근성을 고려한 특정한 블루를 염두에 두고 있지만, 개발자는 CSS의 기본 blue(#0000FF)를 떠올리죠.

실제로 우리 팀에서 있었던 일이에요:

디자이너: "버튼을 파란색으로 만들어주세요."
개발자: "네, 알겠습니다."
(결과물을 보고)
디자이너: "이건 너무 진한데요... 좀 더 연하게..."
개발자: "얼마나 연하게요?"
디자이너: "음... 좀 더 하늘색 느낌으로..."

**💡 문제의 핵심**
- 디자이너는 비주얼한 언어로 소통한다 (느낌, 이미지, 감정)
- 개발자는 정확한 수치와 사양을 필요로 한다
- 결과: 수십 번의 수정 요청과 시간 낭비

이 문제를 해결하기 위해 우리가 도입한 것이 바로 '디자인 시스템'이에요. 디자인 시스템은 단순히 예쁘게 만들기 위한 것이 아니라, 효율적인 협업을 위한 공통 언어를 만드는 것이죠.

## 2. 디자인 시스템: 우리만의 번역기

디자인 시스템이란 디자인 요소들을 체계적으로 정리하고 규칙을 정해놓은 가이드라인이에요. 단순하게 말하면 "파란색" 대신 "#2563EB"라고 정확히 지정하는 거죠.

우리 팀의 디자인 시스템 사례를 보여드릴게요:

### 🎨 컬러 시스템

기존: "파란색으로 해주세요"
개선: "Primary-500 색상으로 해주세요 (#2563EB)"

- Primary-500: #2563EB (메인 버튼, 링크)
- Primary-400: #3B82F6 (따뜨한 블루, 호버 상태)
- Primary-300: #60A5FA (연한 블루, 비활성화)
- Neutral-900: #1F2937 (메인 텍스트)
- Neutral-600: #6B7280 (서브 텍스트)

### 📝 타이포그래피 시스템

기존: "폰트 크기 좀 키워주세요"
개선: "Heading-L 사이즈로 해주세요 (24px/1.5)"

- Heading-XL: 32px/1.2 (페이지 제목)
- Heading-L: 24px/1.3 (섹션 제목)
- Body-M: 16px/1.5 (본문)
- Caption: 14px/1.4 (설명 텍스트)

### 📱 스페이싱 시스템

기존: "여백 좋게 좀 많이 주세요"
개선: "Spacing-L로 해주세요 (24px)"

- Spacing-XS: 4px (아이콘과 텍스트 사이)
- Spacing-S: 8px (엘리먼트 내부 여백)
- Spacing-M: 16px (컴포넌트 내부 여백)
- Spacing-L: 24px (컴포넌트 간 여백)
- Spacing-XL: 48px (섹션 간 여백)

**📊 디자인 시스템 도입 효과**
- 수정 요청 횟수: 80% 감소
- 개발 시간: 40% 단축
- 디자인 일관성: 90% 향상

## 3. 디자인 시스템을 개발에 연결하는 법

디자인 시스템을 만들었다고 끝이 아니에요. 이것을 개발자가 쉽게 사용할 수 있도록 연결해야 해요. 우리 팀은 이렇게 진행했어요:

### 1단계: CSS 변수로 전환

디자인 시스템의 모든 값들을 CSS 변수로 정의했어요:

```css
:root {
  /* 컬러 시스템 */
  --color-primary-500: #2563EB;
  --color-primary-400: #3B82F6;
  --color-primary-300: #60A5FA;
  --color-neutral-900: #1F2937;
  --color-neutral-600: #6B7280;
  
  /* 타이포그래피 */
  --font-size-heading-xl: 32px;
  --font-size-heading-l: 24px;
  --font-size-body-m: 16px;
  --font-size-caption: 14px;
  
  /* 스페이싱 */
  --spacing-xs: 4px;
  --spacing-s: 8px;
  --spacing-m: 16px;
  --spacing-l: 24px;
  --spacing-xl: 48px;
}
```

### 2단계: 컴포넌트 기반 개발

디자인 시스템의 요소들을 재사용 가능한 컴포넌트로 만들었어요:

```jsx
// Button 컴포넌트
const Button = ({ variant = 'primary', size = 'medium', children }) => {
  const baseStyles = {
    padding: variant === 'large' ? 'var(--spacing-m)' : 'var(--spacing-s)',
    fontSize: variant === 'large' ? 'var(--font-size-body-m)' : 'var(--font-size-caption)',
    backgroundColor: variant === 'primary' ? 'var(--color-primary-500)' : 'transparent',
    color: variant === 'primary' ? 'white' : 'var(--color-primary-500)',
  };
  
  return <button style={baseStyles}>{children}</button>;
};
```

### 3단계: 스토리북 구축

모든 컴포넌트를 시각적으로 확인할 수 있는 스토리북을 만들었어요. 이렇게 하니 디자이너와 개발자가 같은 화면을 보며 소통할 수 있었어요.

실제 대화 변화:

기존:
디자이너: "버튼을 좋게 만들어주세요."
개발자: "어떻게요?"
디자이너: "음... 대충 크고 파란색으로..."
(결과물 확인 후)
디자이너: "이건 너무 진하고, 여백도 부족하고..."

개선 후:
디자이너: "버튼을 Button-Primary-Large로 만들어주세요."
개발자: "네, 스토리북에서 확인하고 바로 적용할게요."
(결과)
디자이너: "완벽해요!"

## 4. 현실에서 마주하는 어려움들

물론 디자인 시스템 도입이 항상 순조롭기만 한 것은 아니에요. 우리가 겪은 어려움들을 솔직하게 공유해드릴게요:

### 어려움 1: "이번만 예외로..."

디자인 시스템을 만들었는데도 여전히 "이번만 예외로 다른 색상으로 해주세요"라는 요청이 들어와요. 이럴 때마다 팀 내부에서 고민이 많았어요.

해결방법: 예외 사례가 3번 이상 발생하면 디자인 시스템에 추가하기로 규칙을 정했어요. '예외'가 일반적인 요구라면 시스템에 포함되어야 하니까요.

### 어려움 2: 기존 코드 리팩토링

이미 개발된 코드들을 디자인 시스템에 맞촰 수정하는 작업이 생각보다 방대했어요. 3개월 동안 조금씩 리팩토링하며 전환했어요.

해결방법: 신규 기능부터 디자인 시스템을 적용하고, 기존 코드는 점진적으로 바꿈어나갔어요. 마지막에는 '리팩토링 주간'을 정해서 집중적으로 수정했어요.

### 어려움 3: 브랜드 변경

회사의 브랜드 아이덴티티가 바뀔면서 전체 디자인 시스템을 바꿀 위기가 왔어요. 다행히 CSS 변수로 잘 정리해놓았던 덕분에 몇 개 값만 바꿀어도 전체가 일관되게 바뀐어요.

학습: 디자인 시스템의 진짜 가치를 느낀 순간이었어요. 만약 예전처럼 했다면 수백 개의 CSS 파일을 일일이 수정해야 했을 테니까요.

## 5. 마지막으로: 협업의 새로운 차원

디자인 시스템을 도입하고 나서, 우리 팀의 협업 방식이 완전히 바뀐어요. 더 이상 '느낌'으로 소통하지 않아도 되고, 정확한 사양으로 의도를 전달할 수 있게 됐어요.

무엇보다 좋았던 것은 디자이너와 개발자가 서로를 더 잘 이해하게 되었다는 거예요. 디자이너는 개발자가 왜 정확한 수치를 필요로 하는지 알게 되었고, 개발자는 디자이너가 왜 세세한 디테일에 신경쓰는지 이해하게 되었어요.
