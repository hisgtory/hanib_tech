# Episode 4-2: 이 버그는 누가 처리해야 해요?라고 물을 때

QA 과정에서 버그를 발견했을 때, "이 버그는 누가 처리해야 하나요?"라는 질문은 모든 팀원들이 한 번쯤은 던져본 질문일 거예요. 특히 신입이나 주니어라면 더욱 막막하게 느껴질 수 있죠. 오늘은 버그 리포팅의 기본부터 실제 상황에서 어떻게 대처해야 하는지까지 차근차근 알아보도록 해요.

## 🐛 버그의 종류를 먼저 이해하자

버그를 누가 처리해야 하는지 판단하려면, 먼저 버그의 성격을 파악해야 해요. 크게 다음과 같이 분류할 수 있어요:

- 프론트엔드 버그: 화면에 보이는 부분의 문제 (레이아웃 깨짐, 버튼 클릭 안됨, 텍스트 잘림 등)
- 백엔드 버그: 서버나 데이터 처리 관련 문제 (데이터 저장 안됨, API 에러, 로그인 실패 등)
- 디자인 이슈: 원래 디자인과 다른 부분 (색상 차이, 폰트 크기, 여백 등)
- 기획 이슈: 요구사항 자체의 문제나 누락 (기능 정의 모호, 예외 케이스 미처리 등)

## 💡 버그를 발견했을 때 확인해야 할 것들

### 1. 재현 가능한가?

먼저 해당 버그가 일관되게 재현되는지 확인해보세요. 가끔 발생하는 버그와 항상 발생하는 버그는 처리 우선순위가 다를 수 있어요. 재현 경로를 명확히 파악하고 기록해두는 것이 중요해요.

### 2. 어느 환경에서 발생하는가?

- 브라우저: Chrome, Safari, Firefox 등 특정 브라우저에서만 발생하는지
- 디바이스: 모바일/데스크톱, iOS/Android 등
- 서버 환경: 개발/스테이징/프로덕션 서버

### 3. 에러 메시지가 있는가?

개발자 도구의 콘솔을 열어 에러 메시지를 확인하세요. 빨간색으로 표시되는 에러 메시지는 버그의 원인을 파악하는 중요한 단서가 돼요.

## 🎯 누구에게 리포트해야 할까?

이제 본격적으로 버그를 누구에게 할당해야 하는지 알아볼게요. 다음은 일반적인 가이드라인이에요:

**프론트엔드 개발자에게:**
- 화면 레이아웃이 깨지는 경우
- 버튼 클릭이나 입력이 작동하지 않는 경우
- JavaScript 에러가 콘솔에 표시되는 경우
- 반응형 디자인 이슈

**백엔드 개발자에게:**
- 데이터가 저장되지 않거나 불러와지지 않는 경우
- API 호출 시 400, 500대 에러가 반환되는 경우
- 로그인/로그아웃 등 인증 관련 문제
- 데이터베이스 연결이나 쿼리 문제

**디자이너에게:**
- 원래 디자인 시안과 다른 화면 구현
- 색상, 폰트, 여백 등의 스타일 차이
- 아이콘이나 이미지 테크 누락
- UX 플로우의 비일관성

**기획자/PM에게:**
- 요구사항이 모호하거나 상충되는 경우
- 예외 케이스가 정의되지 않은 경우
- 비즈니스 로직 자체의 문제
- 사용자 시나리오의 비현실성

## 📝 올바른 버그 리포팅 방법

버그를 리포트할 때는 다음 정보를 포함하면 개발자가 빠르게 이해하고 해결할 수 있어요:

### 1. 버그 제목: 간결하고 명확하게

예) "[로그인] 비밀번호 5회 오류 시 에러 메시지 미노출"

### 2. 발생 환경

- OS: Windows 11 / macOS 14.0
- 브라우저: Chrome 120.0.6099.129
- 디바이스: iPhone 15 Pro
- 서버: 스테이징 환경

### 3. 재현 단계

- 로그인 페이지 접속
- 잘못된 비밀번호로 5회 연속 로그인 시도
- 에러 메시지 확인

### 4. 기대 결과 vs 실제 결과

- 기대: "5회 실패 후 계정 잠김 안내 메시지 노출"
- 실제: "아무 메시지 없음"

### 5. 스크린샷 및 에러 메시지

화면 캐시및 콘솔 에러를 함께 첨부하면 더 빠른 해결이 가능해요.

## 💡 Pro Tip: 불확실할 때는 이렇게!

버그를 누가 처리해야 할지 불확실할 때는:

- 팀 리드나 시니어 개발자에게 먼저 문의하기
- 판00에서 함께 트라이아지 설정하기
- 데일리 스크럼에서 공유하여 다같이 논의하기
- 개발자 도구로 직접 확인하면서 개발자와 함께 보기

## 🌈 마무리하며

QA 과정에서 버그를 발견하고 리포트하는 것은 제품의 품질을 높이는 매우 중요한 작업이에요. 처음에는 누구에게 할당해야 할지 헤맬 수 있지만, 위에서 소개한 가이드라인을 참고하면 더 효율적으로 협업할 수 있을 거예요.

기억해야 할 것은, 버그 리포팅은 비난이 아니라 협업의 과정이라는 거예요. 모든 팀원은 같은 목표를 향해 나아가고 있고, 우리가 발견한 버그는 더 나은 제품을 만들기 위한 소중한 피드백이니까요.

다음 에피소드에서는 배포 단계에 대해 알아볼게요. RC 버전이 무엇인지, 앱 심사가 왜 거절되는지, 그리고 성공적인 배포를 위한 팁들을 공유할 예정이에요!

- **구체적인 버그 리포팅 템플릿**: 즉시 사용 가능한 리포팅 포맷 제공으로 실무 적용성 높음

### 🎯 Areas for Improvement

#### 1. 팀 역학과 갈등 해결 시나리오 강화
현재 이상적인 상황 위주로 설명되어 있어, 실제 팀에서 발생하는 갈등이나 책임 떠넘기기 상황에 대한 대응책 부족.

**추가 제안:**
```
### 버그 담당자 배정 시 발생하는 갈등 해결법
- "이건 제 영역이 아니에요" 상황 시 대화법
- 애매한 경계의 버그에 대한 공정한 배정 원칙
- 우선순위 다툼 시 조정 방법
- 책임 회피 문화를 협력 문화로 바꾸는 방법
```

#### 2. 실제 팀 대화와 커뮤니케이션 스킬
이론적 분류는 좋지만, 실제로 개발자나 디자이너와 대화할 때의 구체적인 표현이나 어투에 대한 가이드 필요.

**개선 방안:**
```
### 버그 리포팅 시 효과적인 커뮤니케이션 예시
❌ 나쁜 예: "또 버그가 있네요. 이거 언제 고치실 건가요?"
✅ 좋은 예: "테스트 중 이슈를 발견했습니다. 함께 확인해보실 수 있을까요?"
→ 협력적이고 건설적인 대화 스타일 가이드
```

#### 3. 실패 사례와 개선 과정
성공적인 버그 리포팅 방법은 제시했지만, 잘못된 리포팅으로 인한 문제 상황과 개선 경험담이 부족.

**보완 내용:**
```
### 버그 리포팅 실패 사례와 교훈
- 불충분한 정보로 인해 개발자가 재현하지 못한 사례
- 잘못된 담당자 배정으로 인한 일정 지연 경험
- 우선순위 오판으로 인한 릴리즈 연기 사태
- 커뮤니케이션 실패로 인한 팀 갈등 상황
```

#### 4. 도구와 시스템 활용 가이드
버그 리포팅 방법은 설명했지만, 실제 사용하는 도구(Jira, GitHub Issues 등)나 체계적인 관리 방법 부족.

**추가 도구 제안:**
```
### 효율적인 버그 관리를 위한 도구 활용법
- Jira, Linear, GitHub Issues 등 도구별 리포팅 템플릿
- 버그 트래킹과 진행상황 모니터링 방법
- 자동화된 버그 분류와 알림 설정
- 팀별 버그 대시보드 구성 방법
```

#### 5. 측정과 개선 방법론
버그 처리 과정의 효율성을 어떻게 측정하고 개선해 나갈지에 대한 구체적 방법 필요.

**개선 제안:**
```
### 버그 처리 효율성 측정과 개선
- 평균 버그 해결 시간 추적
- 잘못된 담당자 배정 비율 모니터링  
- 팀별 버그 처리 만족도 조사
- 리포팅 품질 개선을 위한 피드백 시스템
```
