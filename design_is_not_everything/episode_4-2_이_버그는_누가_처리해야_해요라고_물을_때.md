# Episode 4-2: 이 버그는 누가 처리해야 해요?라고 물을 때

QA 과정에서 버그를 발견했을 때, "이 버그는 누가 처리해야 하나요?"라는 질문은 모든 팀원들이 한 번쯤은 던져본 질문일 거예요. 특히 신입이나 주니어라면 더욱 막막하게 느껴질 수 있죠. 오늘은 버그 리포팅의 기본부터 실제 상황에서 어떻게 대처해야 하는지까지 차근차근 알아보도록 해요.

## 🐛 버그의 종류를 먼저 이해하자

버그를 누가 처리해야 하는지 판단하려면, 먼저 버그의 성격을 파악해야 해요. 크게 다음과 같이 분류할 수 있어요:

- 프론트엔드 버그: 화면에 보이는 부분의 문제 (레이아웃 깨짐, 버튼 클릭 안됨, 텍스트 잘림 등)
- 백엔드 버그: 서버나 데이터 처리 관련 문제 (데이터 저장 안됨, API 에러, 로그인 실패 등)
- 디자인 이슈: 원래 디자인과 다른 부분 (색상 차이, 폰트 크기, 여백 등)
- 기획 이슈: 요구사항 자체의 문제나 누락 (기능 정의 모호, 예외 케이스 미처리 등)

## 💡 버그를 발견했을 때 확인해야 할 것들

### 1. 재현 가능한가?

먼저 해당 버그가 일관되게 재현되는지 확인해보세요. 가끔 발생하는 버그와 항상 발생하는 버그는 처리 우선순위가 다를 수 있어요. 재현 경로를 명확히 파악하고 기록해두는 것이 중요해요.

### 2. 어느 환경에서 발생하는가?

- 브라우저: Chrome, Safari, Firefox 등 특정 브라우저에서만 발생하는지
- 디바이스: 모바일/데스크톱, iOS/Android 등
- 서버 환경: 개발/스테이징/프로덕션 서버

### 3. 에러 메시지가 있는가?

개발자 도구의 콘솔을 열어 에러 메시지를 확인하세요. 빨간색으로 표시되는 에러 메시지는 버그의 원인을 파악하는 중요한 단서가 돼요.

## 🎯 누구에게 리포트해야 할까?

이제 본격적으로 버그를 누구에게 할당해야 하는지 알아볼게요. 다음은 일반적인 가이드라인이에요:

**프론트엔드 개발자에게:**
- 화면 레이아웃이 깨지는 경우
- 버튼 클릭이나 입력이 작동하지 않는 경우
- JavaScript 에러가 콘솔에 표시되는 경우
- 반응형 디자인 이슈

**백엔드 개발자에게:**
- 데이터가 저장되지 않거나 불러와지지 않는 경우
- API 호출 시 400, 500대 에러가 반환되는 경우
- 로그인/로그아웃 등 인증 관련 문제
- 데이터베이스 연결이나 쿼리 문제

**디자이너에게:**
- 원래 디자인 시안과 다른 화면 구현
- 색상, 폰트, 여백 등의 스타일 차이
- 아이콘이나 이미지 테크 누락
- UX 플로우의 비일관성

**기획자/PM에게:**
- 요구사항이 모호하거나 상충되는 경우
- 예외 케이스가 정의되지 않은 경우
- 비즈니스 로직 자체의 문제
- 사용자 시나리오의 비현실성

## 📝 올바른 버그 리포팅 방법

버그를 리포트할 때는 다음 정보를 포함하면 개발자가 빠르게 이해하고 해결할 수 있어요:

### 1. 버그 제목: 간결하고 명확하게

예) "[로그인] 비밀번호 5회 오류 시 에러 메시지 미노출"

### 2. 발생 환경

- OS: Windows 11 / macOS 14.0
- 브라우저: Chrome 120.0.6099.129
- 디바이스: iPhone 15 Pro
- 서버: 스테이징 환경

### 3. 재현 단계

- 로그인 페이지 접속
- 잘못된 비밀번호로 5회 연속 로그인 시도
- 에러 메시지 확인

### 4. 기대 결과 vs 실제 결과

- 기대: "5회 실패 후 계정 잠김 안내 메시지 노출"
- 실제: "아무 메시지 없음"

### 5. 스크린샷 및 에러 메시지

화면 캐시및 콘솔 에러를 함께 첨부하면 더 빠른 해결이 가능해요.

## 💡 Pro Tip: 불확실할 때는 이렇게!

버그를 누가 처리해야 할지 불확실할 때는:

- 팀 리드나 시니어 개발자에게 먼저 문의하기
- 판00에서 함께 트라이아지 설정하기
- 데일리 스크럼에서 공유하여 다같이 논의하기
- 개발자 도구로 직접 확인하면서 개발자와 함께 보기

## 🌈 마무리하며

QA 과정에서 버그를 발견하고 리포트하는 것은 제품의 품질을 높이는 매우 중요한 작업이에요. 처음에는 누구에게 할당해야 할지 헤맬 수 있지만, 위에서 소개한 가이드라인을 참고하면 더 효율적으로 협업할 수 있을 거예요.

기억해야 할 것은, 버그 리포팅은 비난이 아니라 협업의 과정이라는 거예요. 모든 팀원은 같은 목표를 향해 나아가고 있고, 우리가 발견한 버그는 더 나은 제품을 만들기 위한 소중한 피드백이니까요.
