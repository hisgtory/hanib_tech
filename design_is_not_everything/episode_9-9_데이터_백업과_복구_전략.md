## 📋 데이터 보험 약관

### 제1조 (목적 및 정의)

**제1항** 본 약관은 귀하의 소중한 디지털 자산(데이터)을 예기치 못한 사고로부터 보호하고, 손실 시 신속한 복구를 보장하는 데이터 백업 보험의 조건을 명시합니다.

**제2항** 용어의 정의
- **데이터**: 컴퓨터에 저장된 모든 디지털 정보 (문서, 사진, 동영상, 설정 등)
- **백업**: 데이터의 복사본을 안전한 장소에 보관하는 행위 (보험 가입과 동일)
- **복구**: 손실된 데이터를 백업본으로부터 되살리는 과정 (보험금 청구와 동일)
- **데이터 사고**: 하드디스크 고장, 랜섬웨어, 실수 삭제 등으로 인한 데이터 손실

> 💡 **hanib_tech의 친절한 해설**  
> 자동차 보험을 생각해보세요. 사고가 나기 전에 미리 보험료를 내고, 실제 사고가 나면 보험금을 받아 차를 수리하죠? 데이터 백업도 똑같아요!

---

### 제2조 (보장 범위 - 3-2-1 황금 룰)

**제1항** 완전 보장을 위해서는 반드시 "3-2-1 원칙"을 준수해야 합니다.

**보장 조건:**
1. **3개의 복사본**: 원본 1개 + 백업 2개 (마치 자동차보험, 건강보험, 여행자보험을 동시에 드는 것)
2. **2가지 다른 매체**: USB와 클라우드, 또는 외장하드와 NAS 등
3. **1개는 오프사이트**: 집과 사무실이 아닌 다른 장소 (화재, 도난 대비)

```
완전 보장 예시:
원본 데이터 (컴퓨터) ← 메인 거주지
백업본 1 (외장하드) ← 메인 거주지  
백업본 2 (클라우드) ← 데이터 센터 (다른 도시)
```

**제2항** 단, 다음의 경우 보장이 제한됩니다:
- 백업이 1개만 있는 경우: 50% 보장
- 같은 장소에만 백업이 있는 경우: 70% 보장
- 백업을 3개월 이상 하지 않은 경우: 보장 불가

### 제3조 (백업 상품 종류)

#### 가. 풀백업 (전체보험)
- **보장 내용**: 모든 데이터를 통째로 복사
- **보험료**: 높음 (저장 공간과 시간 많이 필요)
- **보상**: 100% 완전 복구 가능
- **권장 주기**: 월 1회

#### 나. 증분백업 (차등보험)
- **보장 내용**: 마지막 백업 이후 변경된 부분만 저장
- **보험료**: 낮음 (공간과 시간 절약)
- **보상**: 복구 시 여러 백업본 필요 (복잡함)
- **권장 주기**: 일 1회

#### 다. 차등백업 (혼합보험)
- **보장 내용**: 마지막 풀백업 이후 변경된 모든 부분
- **보험료**: 중간
- **보상**: 풀백업 + 차등백업만으로 복구 가능
- **권장 주기**: 주 1-2회

> 🏠 **일상 비유로 이해하기**  
> - 풀백업: 이사할 때 집안 물건을 모두 포장 (확실하지만 힘듦)
> - 증분백업: 매일 바뀐 물건만 따로 정리 (간단하지만 나중에 찾기 복잡)
> - 차등백업: 일주일마다 바뀐 물건들을 한 박스에 (적당히 확실하고 적당히 간단)

### 제4조 (보상 기준 - RPO와 RTO)

#### 가. RPO (Recovery Point Objective) - 허용 손실 범위
**"얼마까지 잃어도 괜찮은가?"**

- **RPO 1시간**: 1시간 전 상태까지 복구 보장 (프리미엄 보험)
- **RPO 1일**: 하루 전 상태까지 복구 보장 (일반 보험)
- **RPO 1주**: 일주일 전 상태까지 복구 보장 (기본 보험)

#### 나. RTO (Recovery Time Objective) - 복구 시간 목표
**"얼마나 빨리 돌려놓아 줄 것인가?"**

- **RTO 1시간**: 1시간 내 업무 재개 보장 (긴급 출동 서비스)
- **RTO 4시간**: 반나절 내 복구 완료
- **RTO 24시간**: 하루 내 정상화

```
보상 기준표:
업무 유형        | 권장 RPO | 권장 RTO | 비유
대형 쇼핑몰     | 1시간    | 1시간    | 응급실 수준
일반 사무실     | 1일      | 4시간    | 당일 진료
개인 작업       | 1주      | 24시간   | 일반 진료
```

### 제5조 (재해복구계획 - DRP)

**제1항** 데이터 사고 발생 시 신속한 대응을 위한 단계별 행동 지침

#### 1단계: 사고 접수 및 피해 조사
```
1. 침착하게 상황 파악 (패닉 금지)
2. 추가 피해 방지 (컴퓨터 끄기, 네트워크 차단 등)
3. 백업 상태 확인
4. 피해 규모 산정
```

#### 2단계: 복구 우선순위 결정
```
긴급도 상: 고객 DB, 진행 중인 프로젝트
긴급도 중: 이메일, 업무 문서
긴급도 하: 오래된 자료, 개인 파일
```

#### 3단계: 복구 실행
```
1. 가장 최근 백업 확인
2. 테스트 환경에서 복구 시도
3. 데이터 무결성 검증
4. 실제 환경에 적용
5. 정상 작동 확인
```

#### 4단계: 사후 관리
```
1. 사고 원인 분석
2. 백업 정책 개선
3. 직원 교육 실시
4. 문서 업데이트
```

> ⚡ **TMI: 실제 데이터 참사 사례**
> 
> **사례 1: 토이스토리 2 사건 (1998)**
> - 픽사에서 실수로 토이스토리 2 파일을 모두 삭제
> - 백업 서버도 고장 상태였음
> - 다행히 직원 한 명이 집에서 작업하던 복사본으로 구조
> - 교훈: 오프사이트 백업의 중요성
>
> **사례 2: GitLab 데이터베이스 삭제 (2017)**
> - 엔지니어 실수로 운영 DB 삭제
> - 5개의 백업 중 4개가 작동하지 않음
> - 6시간치 데이터 손실 발생
> - 교훈: 백업 테스트의 중요성

### 제6조 (백업 저장소 선택)

#### 가. 클라우드 백업 (외부 보관함)
**장점:**
- 24시간 자동 백업 (자동이체 같은 편리함)
- 오프사이트 저장 자동 보장
- 접근성 우수 (어디서나 복구 가능)
- 전문 업체의 보안 관리

**단점:**
- 월 사용료 발생
- 인터넷 속도에 의존
- 업체 정책 변경 위험

**추천 대상:** 소규모 팀, 재택근무자, 개인 사용자

#### 나. 온프레미스 백업 (집안 금고)
**장점:**
- 한번 구축하면 추가 비용 없음
- 빠른 백업/복구 속도
- 완전한 통제권
- 외부 의존성 없음

**단점:**
- 초기 투자 비용 높음
- 직접 관리 필요
- 같은 장소 위험 (화재, 도난)
- 기술 지식 필요

**추천 대상:** 대용량 데이터, 보안 중요 업체, IT 전담자 있는 조직

#### 다. 하이브리드 백업 (분산 투자)
```
로컬 NAS (빠른 복구용) + 클라우드 (재해 대비용)
= 보통예금 + 정기예금 + 주식 분산투자 전략
```

### 제7조 (면책 사항)

**다음의 경우 보장하지 않습니다:**

1. **고의 또는 중과실**
   - 백업 없이 데이터 삭제
   - 의심스러운 파일 실행
   - 보안 업데이트 무시

2. **천재지변 (단, 오프사이트 백업 있는 경우 제외)**
   - 지진, 홍수, 화재 등
   - 단, 클라우드 백업이 있으면 보장

3. **외부 요인**
   - 해킹 (보안 조치 미흡 시)
   - 내부자 범죄
   - 정전 (UPS 미설치 시)

4. **노후화**
   - 5년 이상 된 저장 매체
   - 지원 중단된 포맷
   - 호환성 문제

### 제9조 (백업 실천 가이드)

#### 1단계: 현재 상황 진단
```
□ 내 컴퓨터에 있는 중요한 데이터 목록 작성
□ 각 데이터의 중요도 평가 (상/중/하)
□ 현재 백업 상태 점검
□ 손실 시 피해 규모 산정
```

#### 2단계: 백업 전략 수립
```
□ RPO/RTO 목표 설정
□ 백업 방식 선택 (풀/증분/차등)
□ 저장 매체 결정 (클라우드/로컬/하이브리드)
□ 백업 주기 설정
```

#### 3단계: 백업 시스템 구축
```
□ 백업 소프트웨어 설치
□ 자동 백업 스케줄 설정
□ 3-2-1 원칙 적용
□ 테스트 백업 및 복구 실행
```

#### 4단계: 운영 및 관리
```
□ 정기적인 백업 상태 점검
□ 월 1회 복구 테스트
□ 백업 정책 정기 검토
□ 직원/가족 교육
```
