# Episode 2-7: 디자인할 때 획만 그으면 개발자가 화내는 이유

디자이너분들, 혹시 이런 경험 있으시나요? 열심히 디자인해서 개발자에게 전달했는데 "이거 개발하기 어려워요"라는 답이 돌아온 적 말이에요. 개발자 입장에서는 말 못할 고민들이 있답니다.

## 1. 개발자가 진짜 싫어하는 디자인들

디자이너: "이거 그냥 선 하나 그어주시면 되는 거 아닌가요?"
개발자: "(속마음) 그냥 선 하나가 어디 있어..."

예쁘게 보이는 선 하나를 위해서는:

- CSS 스타일링 10줄
- 반응형 웹에서의 브레이크포인트 고려
- 다양한 브라우저 호환성 테스트
- 다크모드 대응

하지만 이건 아무것도 아니죠. 진짜 문제는 다른 곳에 있어요.

**💡 개발자가 진짜 원하는 것**
- 명확한 기능 정의: "이 버튼을 누르면 무엇이 일어나나요?"
- 예외 상황 고려: "데이터가 없을 때는 어떻게 하죠?"
- 상태 변화: "로딩 중이거나 에러가 났을 때는?"
- 접근성: "시각장애인도 사용할 수 있나요?"

하지만 대부분의 디자인 전달은 이런식이에요:

"여기 버튼 하나 넣어주세요. 예쁘게!"

## 2. 개발자의 머릿속: 디자인 단계별 고민

디자이너가 예쁜 디자인을 전달하면, 개발자는 이런 생각을 해요:

### 단계 1: "이거 어떻게 만들지?"

디자인을 보자마자 개발자는 기술적 구현 방법을 고민해요. HTML 구조, CSS 스타일링, JavaScript 인터렉션을 어떻게 조합할지 말이죠.

### 단계 2: "모든 경우를 고려했나?"

버튼 하나만 답도 이런 상황들을 고려해야 해요:
- 기본 상태: 일반적인 모습
- 호버 상태: 마우스를 올렸을 때
- 클릭 상태: 눌렀을 때 빠른 피드백
- 비활성 상태: 사용할 수 없을 때
- 로딩 상태: 어떤 작업을 수행 중일 때
- 에러 상태: 문제가 발생했을 때

하지만 대부분의 디자인에서는 '기본 상태'만 정의되어 있어요. 그럼 개발자가 나머지 5가지 상태를 모두 직접 결정해야 합니다.

### 단계 3: "이건 그냥 디자이너가 결정해도 되는 거니?"

개발자는 이런 상황에서 다시 디자이너에게 물어보고 싶어요. 하지만 "그냥 어떻게 해도 돼"라는 답이 돌아오는 경우가 많죠.

그러면 개발자는 혹시 자신이 잘못 결정해서 나중에 다시 수정하게 될까봐 걱정해요. 그래서 정말로 '어렵다'고 말하는 거예요.

## 3. 해결책: 개발자가 좋아하는 디자인 전달법

그럼 어떻게 해야 개발자가 좋아할까요?

### ✨ 해결법 1: 모든 상태를 디자인하기

버튼 하나를 디자인할 때:
- ✅ 기본 상태
- ✅ 호버 상태 (마우스 올렸을 때)
- ✅ 클릭 상태 (누르는 순간)
- ✅ 비활성 상태 (사용할 수 없을 때)
- ✅ 로딩 상태 (작업 중일 때)

### 📝 해결법 2: 기능 명세서 작성하기

"이 버튼을 누르면 사용자 로그인 API를 호출하고, 성공하면 대시보드로 이동합니다. 실패하면 에러 메시지를 보여주세요."

이러게 구체적으로 설명해주시면 개발자는 훈씬 더 수월하게 개발할 수 있어요!

### 🤝 해결법 3: 예외 상황도 고려하기

디자인할 때 이런 상황들도 미리 생각해주세요:
- 데이터가 없을 때는 어떻게 보여줄까요?
- 로딩이 오래 걸리면 사용자에게 어떻게 알려줄까요?
- 에러가 발생했을 때 어떤 메시지를 보여줄까요?
- 모바일에서는 어떻게 다르게 보여야 할까요?

**💡 실제 사례: 다음과 같이 전달하면 개발자가 박수를 칠 거예요!**

"로그인 버튼 디자인입니다.
- 클릭 시: API 호출 후 대시보드로 이동
- 로딩 중: 버튼 비활성 + 스피너 표시
- 에러 시: '로그인에 실패했습니다. 다시 시도해주세요.' 메시지
- 모바일: 버튼 크기 48px, 터치에 적합한 크기"

이런 식으로 전달하면 개발자는 "와, 이제 어떻게 만들어야 할지 완전히 이해했어요!"라는 반응을 보이죠.

## 4. 마무리: 더 나은 협업을 위해

개발자가 "이거 어려워요"라고 말하는 건 어려움을 피하려는 게 아니에요. 오히려 더 좋은 결과물을 만들고 싶어서 그러는 거죠.

디자이너와 개발자가 서로의 관점을 이해하고 더 자세히 소통한다면, 훨씬 더 멋진 제품을 만들 수 있을 거예요.

**💬 실전 팁: 디자인 학입 체크리스트**

<callout>
📋 디자인 전달 체크리스트

- [ ] 모든 상태 디자인 (기본, 호버, 클릭, 비활성, 로딩, 에러)
- [ ] 기능 설명 작성 (무엇을 하는 버튼인지)
- [ ] 예외 상황 고려 (데이터 없음, 에러, 모바일)
- [ ] 개발자와 사전 논의 (기술적 제약사항 확인)
</callout>

마지막으로, 개발자들도 디자이너의 창의적인 아이디어를 존중하고 지지합니다. 단지 더 좋은 사용자 경험을 만들기 위해 더 명확한 정보가 필요할 뿐이죠. 함께 최고의 제품을 만들어가요! 🚀
