# Episode 2-7: 디자인할 때 획만 그으면 개발자가 화내는 이유

## Hook: 건축가가 "대충 멋있게 지어주세요"라고 했다면?

상상해보세요. 건축가가 시공업체에게 이런 설계도를 건넸다고 해봅시다:

"여기에 벽을 대충 그어놓고... 창문은 적당히 크게... 문은 여기쯤에 멋있게..."

시공업체 사장님이 어떤 반응을 보일까요? 아마 이런 말을 할 겁니다:

"벽 두께가 몇 센티인데요? 창문 크기는 정확히 얼마고, 문 높이는요? 전기선은 어디로 빼야 하죠?"

**이게 바로 디자이너가 "획만 그어놓고" 개발자에게 넘겼을 때 일어나는 일입니다.**

---

## Storyline: GPS가 "적당히 동쪽으로 가세요"라고 안내한다면?

### 🗺️ 내비게이션 시나리오

새로운 프로젝트를 시작하는 것은 처음 가는 목적지로 출발하는 것과 같습니다.

**디자이너의 가이드:**
- "적당히 동쪽으로 가다가..."
- "큰 건물이 보이면 좌회전하고..."
- "예쁜 카페 근처에서 우회전..."

**개발자가 필요한 것:**
- "3.2km 직진 후 우회전"
- "두 번째 신호등에서 좌회전"
- "목적지까지 정확히 847m"

개발자는 GPS처럼 정확한 좌표가 필요한 사람들입니다. "적당히", "대충", "멋있게"라는 표현으로는 코드를 작성할 수 없어요.

---

## Escalation: "획만 그은" 디자인의 실제 모습들

### 😵‍💫 실제로 개발자가 받는 디자인들

**Case 1: 버튼 디자인**
```
디자이너 설명: "버튼을 둥글게 해주세요"
개발자 질문: "라운드 값이 몇 픽셀인가요?"
디자이너 답변: "보기 좋게요..."
```

**Case 2: 여백 설정**
```
디자이너 설명: "여기 간격을 적당히 띄워주세요"
개발자 질문: "정확히 몇 픽셀 간격인가요?"
디자이너 답변: "음... 너무 붙어있지도 않고 너무 떨어져있지도 않게..."
```

**Case 3: 색상 지정**
```
디자이너 설명: "파란색으로 해주세요"
개발자 질문: "hex 코드나 RGB 값이 어떻게 되나요?"
디자이너 답변: "그냥... 파란색이요"
```

### 🎵 음악가 비유로 이해하기

이것은 마치 작곡가가 연주자에게:
- "첫 번째 음을 높게 해주세요" (정확히 몇 옥타브인지 모름)
- "빠르게 연주해주세요" (BPM을 모름)
- "감정적으로 연주해주세요" (어떤 감정인지 모름)

라고 말하는 것과 같습니다.

연주자는 정확한 악보가 필요하듯, 개발자도 정확한 디자인 스펙이 필요해요.

---

## 개발자가 화내는 진짜 이유

### ⏰ 시간이 2배로 늘어나는 문제

**불명확한 디자인이 개발에 미치는 영향:**

1. **추측하며 개발**: 30분 → 1시간
2. **디자이너와 소통**: +30분
3. **수정 작업**: +1시간
4. **재수정**: +30분

**총 3시간 = 원래 예상의 6배**

### 🎯 정확도 문제

개발자가 추측해서 만든 결과물:
- "이게 아니에요... 더 파랗게요"
- "간격이 너무 넓어요"
- "버튼이 너무 네모나요"

결국 몇 번씩 수정하게 되면서 시간과 에너지가 소모됩니다.

### 💸 비용 증가 문제

**불명확한 디자인의 숨겨진 비용:**
- 개발 시간 지연
- 반복 수정으로 인한 추가 비용
- 품질 저하로 인한 사용자 불만
- 팀 내 갈등으로 인한 생산성 저하

---

## TMI Box: 개발자의 뇌구조 이해하기

개발자는 컴퓨터와 대화하는 사람들입니다. 컴퓨터는:
- "적당히"를 이해하지 못해요
- "보기 좋게"를 해석할 수 없어요
- "멋있게"를 계산할 수 없어요

개발자가 컴퓨터에게 전달할 수 있는 것은 오직 **숫자**뿐입니다.
- 색상: #3498db
- 크기: 24px
- 여백: 16px 24px
- 라운드: border-radius 8px

## Payoff: 개발자 친화적인 디자인 만들기

### 🛠️ 정확한 스펙 제공하는 방법

**1. 색상 관리**
```
❌ "파란색으로 해주세요"
✅ Primary Blue: #3498db
   Secondary Blue: #2980b9
   Light Blue: #ebf3fd
```

**2. 간격 시스템**
```
❌ "적당한 여백으로..."
✅ 기본 여백 체계:
   - XS: 4px
   - S: 8px
   - M: 16px
   - L: 24px
   - XL: 32px
```

**3. 타이포그래피**
```
❌ "제목을 크게..."
✅ 제목 스타일:
   - Font: Pretendard Bold
   - Size: 24px
   - Line-height: 32px
   - Letter-spacing: -0.02em
```

### 🎨 디자인 도구 활용법

**Figma에서 개발자 친화적 파일 만들기:**

1. **Design System 구축**
   - 색상 팔레트 정의
   - 컴포넌트 라이브러리 생성
   - 일관된 간격 시스템 적용

2. **Inspect 패널 활용**
   - CSS 값 자동 생성
   - 정확한 수치 확인 가능
   - Export 설정 미리 정의

3. **프로토타이핑 상세화**
   - 애니메이션 Duration 명시
   - Easing 함수 지정
   - 상호작용 상태별 스펙 정리

### 📋 체크리스트: 개발자에게 전달하기 전

- [ ] 모든 색상에 Hex 코드가 있는가?
- [ ] 간격이 픽셀 단위로 명시되어 있는가?
- [ ] 폰트 사이즈와 line-height가 정확한가?
- [ ] 버튼, 카드 등의 라운드 값이 있는가?
- [ ] 호버, 포커스 상태가 디자인되어 있는가?
- [ ] 반응형 브레이크포인트별 스펙이 있는가?

## 마무리: 정확한 소통이 만드는 기적

디자이너가 "획만 그어놓는" 것은 창의성 때문이 아닙니다. 
개발자가 어떤 정보를 필요로 하는지 몰랐기 때문이에요.

**정확한 디자인 스펙이 가져오는 변화:**
- ⏰ 개발 시간 50% 단축
- 🔄 수정 횟수 80% 감소
- 😊 팀 만족도 크게 향상
- 🚀 제품 퀄리티 개선

**Remember:** 
개발자는 까다로워서 정확한 수치를 요구하는 게 아닙니다. 
컴퓨터가 이해할 수 있는 언어로 번역해야 하기 때문이에요.

디자이너의 창의적 아이디어 + 개발자가 이해할 수 있는 정확한 스펙 = 완벽한 제품

이제 여러분도 개발자와 훨씬 매끄럽게 협업할 수 있을 거예요!

## 📝 Editor's Feedback

### 🌟 Current Strengths
- **효과적인 비유 활용**: 건축가, GPS, 음악가 비유가 매우 직관적이고 이해하기 쉬움
- **실용적인 Before/After 예시**: 구체적인 대화 스크립트와 개선 사례가 실무에 바로 적용 가능함
- **체계적인 해결책 제시**: 색상, 간격, 타이포그래피의 구체적인 스펙 가이드가 유용함

### 🎯 Areas for Improvement

#### 1. 디자이너 관점과 창의적 우려 다루기
현재 디자이너의 "부정확함"에만 초점이 맞춰져 있어, 디자이너들이 느낄 수 있는 창의성 제약이나 업무 부담 증가에 대한 우려가 부족함.

**추가 제안:**
```
### 디자이너의 우려사항과 해결책
- "스펙 정리가 너무 번거로워요" → 효율적인 작업 플로우 제안
- "창의성이 숫자에 갇히는 느낌이에요" → 시스템 안에서의 창의적 활용법
- "시간이 더 오래 걸릴 것 같아요" → 초기 투자의 장기적 이익 설명
```

#### 2. 실제 팀 갈등과 해결 과정 시나리오
이상적인 대화 예시는 좋지만, 실제로 팀에서 발생하는 갈등 상황과 그 해결 과정이 부족함.

**개선 방안:**
```
### 실제 갈등 상황과 해결 과정
상황: "시간이 촉박한데 스펙까지 다 정리해야 하나요?"
디자이너 vs 개발자 vs PM의 입장 차이
→ 우선순위를 정하고 단계적으로 개선하는 현실적 해결책
```

#### 3. 도구 활용과 자동화 방안 확장
Figma 활용법은 언급했지만, 더 많은 도구와 자동화 방안에 대한 내용이 부족함.

**보완 내용:**
```
### 효율적인 디자인-개발 협업 도구
- Figma Plugin: Design Tokens, Dev Mode
- Zeplin, Abstract 등 핸드오프 전용 도구
- CSS-in-JS와 디자인 토큰 연동 방법
- 자동화된 스펙 문서 생성 도구
```

#### 4. 실패 사례와 교훈 부족
성공 사례는 제시했지만, 실제로 스펙이 부정확해서 실패한 프로젝트의 구체적 사례와 교훈이 필요함.

**추가 사례:**
```
### 스펙 부족으로 인한 실제 실패 사례
- 모바일 앱에서 버튼 크기가 애매해서 사용성 문제 발생
- 색상 코드 불일치로 브랜딩 혼란 야기
- 애니메이션 스펙 부족으로 성능 문제 발생
```

#### 5. 측정 가능한 성과와 개선 방법론
개선 효과를 어떻게 측정하고 지속적으로 발전시킬지에 대한 구체적 방법론 부족.

**개선 제안:**
```
### 협업 개선 성과 측정 방법
- 디자인-개발 핸드오프 시간 추적
- 수정 요청 횟수 변화 모니터링
- 팀 만족도 정기 서베이
- 프로젝트 완료 시간 단축 정도 측정
```
