# Episode 5-3. 앱 심사에서 리젝당하면 2주가 날아간다

앱 개발의 마지막 단계인 앱 스토어 심사. 개발자들이 가장 긴장하는 순간이죠. 특히 iOS 앱 스토어는 까다로운 심사 기준으로 유명합니다. 리젝(거절)당하면 수정하고 다시 제출하는 데 최소 1-2주가 걸리기 때문에, 미리 대비하는 것이 중요합니다.

이번 에피소드에서는 앱 심사에서 자주 리젝당하는 이유와 대응 방법, 그리고 심사를 원활하게 통과하기 위한 실무 팁들을 공유하려고 합니다.

## 📱 앱 심사의 기본 프로세스

먼저 앱 심사 프로세스를 이해해보겠습니다. iOS와 Android는 조금 다르지만, 기본적인 흐름은 비슷합니다:

- 앱 빌드 업로드
- 메타데이터 입력 (앱 설명, 스크린샷, 카테고리 등)
- 심사 제출
- 심사 기간 (iOS: 24-48시간, Android: 2-3시간)
- 승인 또는 리젝

## 🚫 가장 흔한 리젝 사유 TOP 5

제 경험상 가장 자주 마주하는 리젝 사유들을 순위별로 정리해봤습니다:

### 1️⃣ 약관 및 개인정보처리방침 누락

앱 내에 접근 가능한 약관과 개인정보처리방침이 없으면 바로 리젝됩니다. 특히 개인정보를 수집하는 경우(로그인, 회원가입 등) 필수입니다.

- 해결방법: 앱 내 설정 메뉴에 '약관' 및 '개인정보처리방침' 링크 추가
- 로그인/회원가입 화면에도 링크 배치
- URL은 실제 웹페이지여야 함 (임시 URL 불가)

### 2️⃣ 신고 기능 누락 (UGC 앱)

사용자가 컨텐츠를 생성할 수 있는 앱(SNS, 커뮤니티, 리뷰 등)에는 반드시 신고 기능이 있어야 합니다.

- 해결방법: 각 컨텐츠에 '신고하기' 버튼 추가
- 신고 사유 선택 등록 (스팸, 부적절한 컨텐츠 등)
- 사용자 차단 기능도 필요

### 3️⃣ 불분명한 비즈니스 모델

무료 앱인데 수익 모델이 불분명하거나, 유료 기능에 대한 설명이 부족하면 리젝됩니다.

- 해결방법: 앱 설명에 비즈니스 모델 명시
- 인앱 결제를 사용하는 경우 Apple 가이드라인 준수
- 유료 기능에 대한 명확한 안내

### 4️⃣ 크래시 및 버그

심사 중 앱이 크래시하거나 주요 기능이 작동하지 않으면 즉시 리젝됩니다.

- 해결방법: 철저한 QA 진행
- 다양한 기기와 OS 버전에서 테스트
- 크래시 리포팅 도구 통합 (Firebase Crashlytics 등)

### 5️⃣ 권한 설명 부족

카메라, 위치, 마이크 등의 권한을 요청하면서 그 이유를 명확히 설명하지 않으면 리젝됩니다.

- 해결방법: Info.plist에 권한 사용 이유 상세 기술
- 권한 요청 시 팝업에도 명확한 설명 표시
- 불필요한 권한은 요청하지 않기

## 📝 심사를 원활하게 통과하기 위한 체크리스트

심사 제출 전 꼭 확인해야 할 사항들을 체크리스트로 정리했습니다:

### ✅ 기본 체크사항

- 약관 및 개인정보처리방침 링크 확인
- UGC 앱의 경우 신고/차단 기능 확인
- 모든 화면에서 크래시 테스트
- 다양한 기기 및 OS 버전 테스트
- 권한 설명 문구 확인

### ✅ 메타데이터 체크

- 앱 설명 명확하고 정확한지 확인
- 스크린샷 품질 및 가이드라인 준수
- 카테고리 선택 적절성
- 키워드 최적화 (ASO)
- 예상 통과 시간 고려한 일정 계획

## 💡 실무에서 바로 쓸 수 있는 팁

제가 실무에서 경험한 노하우를 공유합니다:

### 1. 테스트 계정 제공하기

심사자가 로그인이 필요한 기능을 테스트할 수 있도록 테스트 계정을 제공하세요. App Store Connect의 'App Review Information'에 테스트 계정 정보를 명시하면 됩니다.

### 2. 심사 노트 활용하기

복잡한 기능이나 특수한 상황에 대한 설명을 심사 노트에 상세히 작성하세요. 예를 들어:

- "GPS 기능은 특정 지역에서만 활성화됩니다"
- "A/B 테스트 중이므로 특정 기능이 표시되지 않을 수 있습니다"
- "베타 기능은 설정 > 고급에서 활성화할 수 있습니다"

### 3. 예외 처리 강화하기

심사 환경은 예측할 수 없으므로, 다양한 예외 상황에 대비하세요:

- 네트워크 연결 없음
- 서버 연결 실패
- 빈 데이터 상태
- 권한 거부 상태

### 4. 서버 환경 안정화

심사 기간 동안 서버가 다운되면 바로 리젝됩니다. 다음 사항을 체크하세요:

- 심사 기간 동안 서버 점검 금지
- 임시 서버가 아닌 정식 서버 사용
- 서버 모니터링 강화

## 🔄 리젝 당했을 때 대처법

리젝은 누구에게나 일어날 수 있습니다. 중요한 것은 빠르고 정확하게 대응하는 것입니다:

### 1. 리젝 사유 분석

- Apple에서 제공한 리젝 사유를 꼼꼼히 읽어보세요
- 규정 번호를 확인하고 가이드라인을 상세히 읽어보세요
- 필요하면 Resolution Center를 통해 구체적인 설명을 요청하세요

### 2. 빠른 수정 및 재제출

- 간단한 문제라면 즉시 수정하여 재제출
- 빌드 수정이 필요한 경우 빠르게 업데이트
- Resolution Center에 수정 내용을 상세히 설명

### 3. 필요한 경우 Appeal(이의 제기)

- 규정 해석에 이견이 있다면 Appeal 가능
- 비슷한 사례를 찾아 근거 제시
- 전문적이고 정중한 태도로 소통

## 🎯 앞으로의 과제

앱 심사는 계속 진화하고 있습니다. 최근 트렌드를 보면:

- 개인정보 보호 정책 더욱 엄격
- 어린이 보호 기능 필수화
