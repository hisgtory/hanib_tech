# Episode 3-11: 개발자는 서로 작성한 코드를 어떻게 합쳐요?

## Hook: 10명의 개발자가 동시에 코딩한다면?

💻 상상해보세요. 쇼핑몰을 만드는데 10명의 개발자가 각각 다른 기능을 만들고 있어요. 한 명은 장바구니, 한 명은 결제, 한 명은 검색 기능을... 그런데 이걸 어떻게 하나의 완성된 서비스로 합칠까요?

## 개발자들의 협업 도구: Git과 GitHub

### Git이란?
Git은 코드의 **버전 관리 시스템**입니다. 쉽게 말해서 "코드의 변경사항을 기록하고 관리하는 도구"예요.

**📝 Git이 하는 일**:
- 코드가 언제, 누가, 무엇을 바꿨는지 기록
- 이전 버전으로 되돌리기 가능
- 여러 명이 동시에 작업해도 충돌 해결
- 브랜치를 만들어서 독립적으로 작업

### GitHub란?
GitHub는 Git을 웹에서 쉽게 사용할 수 있게 해주는 **플랫폼**입니다. 코드를 저장하고, 다른 개발자들과 협업할 수 있는 온라인 공간이죠.

**🌐 GitHub가 제공하는 것**:
- 코드 저장소 (Repository)
- 협업 도구 (Pull Request, Issues)
- 코드 리뷰 기능
- 프로젝트 관리 도구

## 협업의 핵심: Repository(저장소)

### Repository란?
Repository는 프로젝트의 모든 파일과 변경 이력이 저장된 **디지털 폴더**입니다.

```
쇼핑몰 프로젝트 Repository
├── 프론트엔드/
│   ├── 메인 페이지
│   ├── 상품 상세 페이지
│   └── 장바구니 페이지
├── 백엔드/
│   ├── 사용자 관리
│   ├── 상품 관리
│   └── 결제 처리
└── 데이터베이스 설정
```

### 중앙 저장소의 중요성
모든 개발자가 **하나의 중앙 저장소**를 공유합니다. 마치 구글 드라이브에서 문서를 공유하는 것과 비슷해요.

## Branch: 독립적인 작업 공간

### Branch란?
Branch는 **독립적인 작업 공간**입니다. 메인 코드에 영향을 주지 않고 새로운 기능을 개발할 수 있어요.

**🌳 Branch 구조 예시**:
```
main (메인 브랜치)
├── feature/장바구니 (김개발자)
├── feature/결제시스템 (이개발자)
└── feature/상품검색 (박개발자)
```

### 실제 예시: 쇼핑몰 프로젝트
**상황**: 3명의 개발자가 각각 다른 기능을 만들어야 함

**👨‍💻 김개발자**: 장바구니 기능
- `feature/cart` 브랜치 생성
- 장바구니 추가/삭제 기능 개발
- 메인 코드에 영향 없이 독립 작업

**👩‍💻 이개발자**: 결제 시스템
- `feature/payment` 브랜치 생성
- 카드 결제, 무통장입금 기능 개발
- 다른 개발자 작업과 완전 분리

**👨‍💻 박개발자**: 검색 기능
- `feature/search` 브랜치 생성
- 상품명, 카테고리 검색 기능 개발
- 독립적인 환경에서 안전하게 작업

## Commit: 변경사항 저장하기

### Commit이란?
Commit은 **코드 변경사항을 저장하는 단위**입니다. 마치 게임의 세이브 포인트 같아요.

**💾 Commit 예시**:
```
커밋 1: "장바구니 페이지 UI 추가"
커밋 2: "장바구니에 상품 추가 기능 구현"
커밋 3: "장바구니 삭제 기능 추가"
커밋 4: "장바구니 수량 변경 기능 구현"
```

### 좋은 Commit 메시지 작성법
**✅ 좋은 예시**:
- "로그인 버튼 클릭 이벤트 추가"
- "사용자 정보 수정 API 연동"
- "결제 실패 시 에러 메시지 표시 기능 추가"

**❌ 나쁜 예시**:
- "수정함"
- "버그 고침"
- "이것저것 바꿈"

## Pull Request: 코드 합치기 신청

### Pull Request란?
Pull Request(PR)는 **"내가 만든 코드를 메인 프로젝트에 합쳐달라"는 요청**입니다.

### Pull Request 과정
**1단계: 개발 완료**
```
김개발자가 장바구니 기능 개발 완료
feature/cart 브랜치에서 작업 마무리
```

**2단계: Pull Request 생성**
```
제목: "장바구니 기능 추가"
설명: 
- 상품을 장바구니에 추가/삭제하는 기능 구현
- 장바구니 내 상품 수량 변경 기능 추가
- 장바구니 총 금액 자동 계산 기능 포함
```

**3단계: 코드 리뷰**
동료 개발자들이 코드를 검토합니다:

**👨‍💻 리뷰어 A**: "장바구니가 비어있을 때 메시지 표시는 어떻게 되나요?"
**👩‍💻 김개발자**: "빈 장바구니일 때 '장바구니가 비어있습니다' 메시지를 보여줍니다."

**👩‍💻 리뷰어 B**: "상품 수량이 재고보다 많을 때는 어떻게 처리하나요?"
**👨‍💻 김개발자**: "재고 부족 알림을 표시하고 최대 가능 수량으로 자동 조정됩니다."

**4단계: 승인 및 Merge**
리뷰가 완료되면 **Merge**(병합)를 통해 메인 브랜치에 합쳐집니다.

## Merge Conflict: 충돌 상황과 해결

### Conflict가 발생하는 상황
**상황**: 김개발자와 이개발자가 **같은 파일의 같은 부분**을 수정했을 때

**김개발자 수정**:
```javascript
// 할인율 계산
const discount = price * 0.1; // 10% 할인
```

**이개발자 수정**:
```javascript
// 할인율 계산  
const discount = price * 0.15; // 15% 할인
```

### Conflict 해결 과정
**1. Git이 자동으로 충돌 감지**
```
<<<<<<< HEAD (김개발자 버전)
const discount = price * 0.1; // 10% 할인
=======
const discount = price * 0.15; // 15% 할인
>>>>>>> feature/payment (이개발자 버전)
```

**2. 팀에서 논의 후 결정**
팀 리더: "마케팅팀과 협의 결과 15% 할인이 맞습니다."

**3. 충돌 해결**
```javascript
// 할인율 계산
const discount = price * 0.15; // 15% 할인 (마케팅팀 확인)
```

## 실제 협업 시나리오: 하루의 개발 과정

### 오전 9시: 작업 시작
**👨‍💻 김개발자**:
```bash
# 최신 코드 받아오기
git pull origin main

# 새로운 기능 브랜치 생성
git checkout -b feature/wishlist

# 위시리스트 기능 개발 시작
```

### 오전 11시: 다른 개발자 작업 진행
**👩‍💻 이개발자**:
```bash
# 결제 기능 코드 커밋
git add .
git commit -m "신용카드 결제 기능 추가"
git push origin feature/payment
```

### 오후 2시: Pull Request 생성
**👨‍💻 박개발자**: 검색 기능 완료 후 PR 생성
```
제목: "상품 검색 및 필터링 기능 추가"
변경사항:
- 키워드 기반 상품 검색
- 가격대별 필터링  
- 카테고리별 정렬
- 검색 결과 페이지네이션
```

### 오후 4시: 코드 리뷰 시간
팀 전체가 박개발자의 PR을 리뷰:

**👨‍💻 김개발자**: "검색 속도는 어떤가요? 상품이 많아지면 느려질 수 있을 것 같은데요."
**👨‍💻 박개발자**: "인덱싱을 적용했고, 페이지당 20개씩만 로드해서 성능 최적화했습니다."

**👩‍💻 이개발자**: "검색어가 없을 때는 어떻게 처리되나요?"
**👨‍💻 박개발자**: "전체 상품을 인기순으로 보여주도록 했습니다."

### 오후 5시: Merge 완료
리뷰 완료 후 메인 브랜치에 병합:
```
✅ feature/search → main 브랜치 Merge 완료
📝 검색 기능이 프로덕션에 반영되었습니다
```

## 협업의 장점들

### ✅ 동시 작업 가능
- 여러 명이 같은 프로젝트를 동시에 개발
- 각자 독립적인 기능을 담당
- 작업 효율성 극대화

### ✅ 코드 품질 향상
- 동료들의 코드 리뷰로 버그 사전 발견
- 다양한 관점에서의 피드백
- 베스트 프랙티스 공유

### ✅ 지식 공유
- 다른 개발자의 코드를 보며 학습
- 새로운 기술과 패턴 습득
- 팀 전체의 기술 수준 향상

### ✅ 안정성 보장
- 메인 코드는 항상 안정적 상태 유지
- 문제 있는 코드는 별도 브랜치에서 수정
- 언제든 이전 버전으로 롤백 가능

## 협업 시 주의사항

### ⚠️ 커뮤니케이션이 핵심
**중요한 변경사항은 미리 공유**:
```
팀 채널에서:
"DB 스키마를 변경해야 할 것 같습니다. 
사용자 테이블에 '마케팅 수신 동의' 필드를 추가하려고 하는데, 
다른 분들 작업에 영향 있을까요?"
```

### ⚠️ 작은 단위로 자주 커밋
**좋은 커밋 패턴**:
```
매일 저녁 퇴근 전:
- "오늘 작업한 로그인 폼 UI 완료"
- "비밀번호 유효성 검사 로직 추가"
- "로그인 실패시 에러 메시지 표시 기능"
```

### ⚠️ 충돌을 두려워하지 말기
충돌은 자연스러운 현상입니다. 중요한 것은:
- 빠른 소통으로 해결
- 팀원들과 협의하여 최선의 결정
- 충돌 해결 과정에서 배우기

## 마무리: 혼자서는 불가능한 일들

개발자들이 Git과 GitHub를 통해 협업하는 모습은 마치 **오케스트라**와 같습니다. 각자 다른 악기(기능)를 연주하지만, 지휘자(프로젝트 매니저)의 지휘 하에 하나의 아름다운 음악(완성된 서비스)을 만들어내죠.

**🎯 핵심 포인트**:
- **Repository**: 모든 코드가 저장되는 중앙 저장소
- **Branch**: 독립적인 작업 공간
- **Commit**: 변경사항을 저장하는 단위
- **Pull Request**: 코드 합치기 신청서
- **Merge**: 브랜치를 메인 코드에 합치는 과정

이런 협업 시스템 덕분에 수백 명의 개발자가 함께 구글, 페이스북, 넷플릭스 같은 거대한 서비스를 만들 수 있는 거예요. 혼자서는 절대 불가능한 일들이 팀워크로는 가능해지는 것이죠!
