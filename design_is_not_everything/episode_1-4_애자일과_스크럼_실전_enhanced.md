## 🌊 워터폴에서 애자일로

### 개발 방법론의 진화
```
📖 소프트웨어 개발 방법론 역사

1970년대: Waterfall (폭포수)
"완벽한 계획 → 순차적 실행"
문제: 변경 불가, 늦은 피드백

1990년대: Iterative (반복)
"작은 단위로 나누어 개발"
개선: 중간 검증 가능

2001년: Agile Manifesto
"변화 수용, 협업 중시"
혁신: 유연성과 속도

2011년: Lean Startup
"빠른 실험과 학습"
확장: 비즈니스 전체로

2020년대: Hybrid
"상황에 맞는 방법론 선택"
현실: 애자일 + 워터폴 조합
```

### 애자일 선언문
```
📜 Agile Manifesto (2001)

우리는 다음을 더 가치있게 여긴다:

1. 프로세스와 도구보다 → 개인과 상호작용
2. 포괄적인 문서보다 → 작동하는 소프트웨어  
3. 계약 협상보다 → 고객과의 협력
4. 계획 따르기보다 → 변화에 대응

* 왼쪽도 가치가 있지만, 오른쪽을 더 중요시한다
```

## 🏃 스크럼 프레임워크

### 스크럼의 구조
```
🏈 스크럼 = 럭비에서 유래

3-3-5 구조:
• 3개 역할 (Roles)
• 3개 산출물 (Artifacts)
• 5개 이벤트 (Events)

스크럼의 핵심:
"짧은 주기로 빠르게 반복하며
 지속적으로 개선한다"
```

### 3가지 역할
```
👥 스크럼 팀 구성

1. Product Owner (PO)
   책임: 제품의 가치 극대화
   • 백로그 관리
   • 우선순위 결정
   • 스테이크홀더 소통
   • 인수 조건 정의

2. Scrum Master (SM)
   책임: 프로세스 촉진
   • 스크럼 코칭
   • 장애물 제거
   • 팀 보호
   • 지속적 개선

3. Development Team
   책임: 제품 개발
   • 자기 조직화
   • 다기능 팀
   • 5-9명 규모
   • 공동 책임
```

### 3가지 산출물
```
📦 스크럼 산출물 (Artifacts)

1. Product Backlog
   정의: 제품에 필요한 모든 기능 목록
   관리: Product Owner
   특징: 우선순위 정렬, 지속 업데이트

2. Sprint Backlog
   정의: 현재 스프린트에서 할 일
   관리: Development Team
   특징: 스프린트 목표 달성 위한 태스크

3. Increment
   정의: 완료된 제품 증분
   기준: Definition of Done 충족
   특징: 배포 가능한 상태
```

### 5가지 이벤트
```
📅 스크럼 이벤트 (Ceremonies)

1. Sprint (스프린트)
   기간: 1-4주 (보통 2주)
   목적: 배포 가능한 증분 생산

2. Sprint Planning (계획)
   시간: 스프린트당 8시간 이내
   결과: 스프린트 목표, 백로그 선정

3. Daily Scrum (데일리)
   시간: 매일 15분
   목적: 동기화와 계획 조정

4. Sprint Review (리뷰)
   시간: 스프린트당 4시간 이내
   목적: 증분 시연, 피드백 수집

5. Sprint Retrospective (회고)
   시간: 스프린트당 3시간 이내
   목적: 프로세스 개선
```

## 📊 스프린트 실전 운영

### 2주 스프린트 일정표
```
📅 표준 2주 스프린트 스케줄

Week 1:
월: Sprint Planning (4시간)
   - 스프린트 목표 설정
   - 백로그 선정
   - 태스크 분해

화-금: 개발 작업
   - Daily Scrum (매일 15분)
   - 페어 프로그래밍
   - 코드 리뷰

Week 2:
월-목: 개발 + 테스트
   - 기능 완성
   - 버그 수정
   - 통합 테스트

금: Sprint Review & Retro
   AM: Review (2시간)
   PM: Retrospective (2시간)
   
   → 다음 스프린트 준비
```

### 스토리 포인트 추정
```
🎯 스토리 포인트 시스템

피보나치 수열 사용:
1, 2, 3, 5, 8, 13, 21, ...

포인트별 기준:
• 1점: 반나절 작업
• 2점: 1일 작업
• 3점: 2-3일 작업
• 5점: 3-5일 작업
• 8점: 1주 작업
• 13점: 너무 큼 (쪼개기)

Planning Poker:
1. 스토리 설명
2. 각자 카드 선택
3. 동시에 공개
4. 차이가 크면 토론
5. 재투표 → 합의
```

### 속도(Velocity) 계산
```
📈 팀 속도 측정과 활용

Velocity = 완료한 스토리 포인트 합계

예시:
Sprint 1: 24 points
Sprint 2: 28 points
Sprint 3: 26 points
평균 Velocity: 26 points

활용:
• 다음 스프린트 계획: 26점 목표
• 릴리즈 예측: 100점 = 4스프린트
• 팀 성장 측정: 속도 증가 추이
```

## 📝 백로그 관리

### 사용자 스토리 작성
```
✍️ User Story 템플릿

기본 형식:
"As a [사용자 유형],
 I want [기능/목표],
 So that [가치/이유]"

예시:
"As a 쇼핑몰 고객,
 I want 원클릭 구매 기능,
 So that 빠르게 결제할 수 있다"

INVEST 기준:
• Independent: 독립적
• Negotiable: 협상 가능
• Valuable: 가치 있는
• Estimable: 추정 가능
• Small: 작은 크기
• Testable: 테스트 가능
```

### 인수 조건 정의
```
✅ Acceptance Criteria

스토리: 로그인 기능
인수 조건:
□ 이메일/비밀번호로 로그인
□ 잘못된 정보 시 에러 메시지
□ 3회 실패 시 계정 잠금
□ 비밀번호 찾기 링크 제공
□ 로그인 유지 옵션
□ 소셜 로그인 (구글, 카카오)

Definition of Done:
□ 코드 작성 완료
□ 단위 테스트 통과
□ 코드 리뷰 완료
□ 통합 테스트 통과
□ 문서 업데이트
□ 배포 준비 완료
```

## 💡 애자일 전환 전략

### 단계별 도입 계획
```
🚀 애자일 전환 로드맵

Phase 1: 파일럿 (3개월)
• 작은 팀 1개 선정

Phase 2: 확산 (6개월)
• 2-3개 팀 추가
• 스크럼 마스터 양성
• 도구 도입 (Jira)

Phase 3: 정착 (9개월)
• 전사 확대
• 프로세스 최적화
• 성과 측정

Phase 4: 성숙 (12개월+)
• 지속적 개선
• 커스터마이징
• 조직 문화화
```

### 흔한 실패 패턴
```
⚠️ 애자일 도입 실패 사례

1. "애자일 하는 척"
   증상: 이름만 스프린트, 실제는 워터폴
   해결: 진정한 반복과 피드백

2. "문서 없는 개발"
   증상: 애자일 = 문서 불필요 오해
   해결: 필요한 문서는 작성

3. "계획 없는 개발"
   증상: 그때그때 대응
   해결: 스프린트 계획 철저히

4. "스크럼 마스터 = 관리자"
   증상: 지시하고 통제
   해결: 촉진자 역할 이해

5. "회고 생략"
   증상: 개선 없는 반복
   해결: 회고 필수 진행
```
