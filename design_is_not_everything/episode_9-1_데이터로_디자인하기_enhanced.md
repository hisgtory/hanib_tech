## 📈 Part 1: 디자이너가 꼭 알아야 할 데이터 지표

### 핵심 지표 완벽 이해

#### 🎯 사용자 지표 (User Metrics)

```yaml
DAU (Daily Active Users):
  정의: 하루 동안 서비스를 사용한 순 사용자 수
  왜 중요?: 서비스의 실제 활력도
  디자인 연관성: UI/UX가 사용 빈도에 직접 영향
  목표: 꾸준한 상승 또는 유지

MAU (Monthly Active Users):
  정의: 한 달 동안 서비스를 사용한 순 사용자 수
  계산: DAU와 별개로 계산 (DAU × 30 아님!)
  Stickiness: DAU/MAU 비율 (높을수록 좋음)
  벤치마크: 소셜 앱 50%, 일반 앱 20%

Retention Rate (리텐션):
  정의: 특정 기간 후 돌아온 사용자 비율
  종류:
    - D1: 다음날 리텐션 (40-60% 목표)
    - D7: 7일 후 (20-30% 목표)
    - D30: 30일 후 (10-20% 목표)
  디자인 영향: 온보딩 UX가 D1에 결정적
```

#### 💰 비즈니스 지표 (Business Metrics)

```yaml
Conversion Rate (전환율):
  정의: 목표 행동을 완료한 비율
  계산: (전환 수 / 전체 방문자) × 100
  종류:
    - Macro: 구매, 가입 (2-3%)
    - Micro: 장바구니, 클릭 (10-20%)
  디자인 개선 포인트:
    - CTA 버튼 최적화
    - 단계 축소
    - 신뢰 신호 추가

ARPU (Average Revenue Per User):
  정의: 사용자당 평균 수익
  계산: 총 수익 / 활성 사용자 수
  디자인 역할: 
    - 프리미엄 기능 노출
    - 업셀링 UI 최적화
    - 가격 표시 방식

LTV (Lifetime Value):
  정의: 고객 생애 가치
  계산: ARPU × 평균 이용 기간
  중요성: CAC보다 높아야 수익
  디자인 기여: 리텐션 개선으로 LTV 증가
```

#### 🔄 행동 지표 (Engagement Metrics)

```yaml
Bounce Rate (이탈률):
  정의: 한 페이지만 보고 나간 비율
  정상 범위: 40-60%
  높은 이탈률 원인:
    - 느린 로딩 (3초 넘으면 53% 이탈)
    - 복잡한 UI
    - 기대와 다른 콘텐츠
  개선 방법:
    - 로딩 최적화
    - 명확한 가치 제안
    - 직관적 네비게이션

Session Duration (세션 시간):
  정의: 한 번 방문 시 머무는 시간
  좋은 지표?: 
    - SNS: 길수록 좋음
    - 유틸리티: 짧을수록 좋음 (효율성)
  측정 방법: 평균값보다 중간값 사용

Click-Through Rate (CTR):
  정의: 노출 대비 클릭 비율
  벤치마크:
    - 이메일: 2-3%
    - 배너: 0.05%
    - 검색 광고: 2-5%
  개선 팁:
    - A/B 테스트 필수
    - 카피와 디자인 일치
    - 명확한 CTA
```

## 🔬 Part 2: A/B 테스트 마스터하기

### A/B 테스트의 과학

#### 📐 테스트 설계

```markdown
## 가설 수립 프레임워크

### Bad Hypothesis ❌
"녹색 버튼이 더 나을 것 같아"

### Good Hypothesis ✅
"CTA 버튼을 파란색(#3B82F6)에서 녹색(#10B981)으로 변경하면,
색상 심리학(녹색=진행)에 따라 클릭률이 10% 이상 증가할 것이다"

구성 요소:
1. 변경 사항 (무엇을)
2. 예상 결과 (얼마나)
3. 근거 (왜)
```

#### 📊 샘플 크기 계산

```markdown
## 필요 샘플 수 계산

현재 전환율: 2%
목표 개선: 0.5%p (25% 상승)
신뢰 수준: 95%
검정력: 80%

필요 샘플: 각 그룹당 6,200명
테스트 기간: 일 트래픽 1,000명 → 약 2주

💡 Tip: 온라인 계산기 활용
- Optimizely Sample Size Calculator
- AB Test Guide Calculator
```

#### 🎲 실제 A/B 테스트 사례

**Case: 이커머스 체크아웃 최적화**

```yaml
테스트 설정:
  Control: 5단계 체크아웃
  Variant A: 3단계 통합
  Variant B: 원페이지 체크아웃
  
  트래픽 분배: 34% / 33% / 33%
  테스트 기간: 3주
  주요 지표: 구매 전환율

결과:
  Control: 2.3% 전환율
  Variant A: 3.1% 전환율 (↑34.8%)
  Variant B: 2.8% 전환율 (↑21.7%)
  
  통계적 유의성: p-value = 0.02 ✅
  
선택: Variant A
이유: 최고 전환율 + 구현 복잡도 적절

후속 조치:
  1. 전체 트래픽에 적용
  2. 모바일/데스크톱 분리 분석
  3. 다음 테스트: 단계 내 최적화
```

### 멀티배리언트 테스트

```markdown
## 여러 요소 동시 테스트

요소 1: 헤드라인
- A: "무료 배송"
- B: "30일 환불 보장"

요소 2: CTA 버튼
- A: "구매하기"
- B: "장바구니 담기"

조합 매트릭스:
| | 구매하기 | 장바구니 |
|---|---------|----------|
| 무료배송 | 2.1% | 2.4% |
| 환불보장 | 2.8% | 3.2% 🏆 |

Winner: 환불보장 + 장바구니 담기
```

## 💡 Part 6: 데이터 기반 디자인 프로세스

### 데이터 드리븐 디자인 워크플로우

```markdown
## 6단계 프로세스

### 1. Define (정의)
- 비즈니스 목표 설정
- 핵심 지표 선정
- 베이스라인 측정

### 2. Discover (발견)
- 정량 데이터 수집
- 정성 리서치 진행
- 패턴과 문제점 파악

### 3. Design (디자인)
- 데이터 기반 가설 수립
- 솔루션 디자인
- 프로토타입 제작

### 4. Deliver (전달)
- 개발 협업
- 단계적 출시
- 측정 준비

### 5. Measure (측정)
- 데이터 수집
- 목표 대비 성과
- 예상치 못한 영향

### 6. Learn (학습)
- 인사이트 도출
- 다음 반복 계획
- 지식 공유
```

### 실제 프로젝트 사례

```markdown
## Case: 온보딩 개선 프로젝트

### Phase 1: 문제 발견
데이터: D1 리텐션 25% (업계 평균 40%)
분석: 온보딩 완료율 30%
인사이트: 70%가 첫 단계에서 이탈

### Phase 2: 가설과 디자인
가설: "단계가 너무 많고 복잡하다"
솔루션:
- 5단계 → 2단계로 축소

예상 연간 가치: +2억원

### Phase 5: 학습과 확산

- 다른 플로우에 적용
```

## 🚨 Part 8: 함정과 실수 피하기

### 데이터의 함정

#### ❌ 함정 1: 상관관계 ≠ 인과관계

```markdown
관찰: "아이스크림 판매량과 익사 사고가 비례"
잘못된 결론: "아이스크림이 익사를 유발"
실제: 여름이라는 공통 원인

디자인 예시:
관찰: "다크모드 사용자의 구매율이 높다"
잘못된 결론: "다크모드가 구매를 증가시킨다"
실제: 파워유저가 다크모드 선호
```

#### ❌ 함정 2: 평균의 함정

```markdown
평균 세션 시간: 5분

실제 분포:
1분 미만: 60% (대부분)
1-3분: 20%
3-10분: 15%
30분 이상: 5% (헤비 유저)

올바른 접근:
- 중앙값 사용: 1.5분
- 세그먼트별 분석
- 분포 확인 필수
```

#### ❌ 함정 3: 샘플 편향

```markdown
"베타 테스터 피드백으로 전체 UX 변경"

문제:
- 베타 테스터 = 얼리어답터
- 일반 사용자와 다른 행동
- 기술 친숙도 높음

해결:
- 다양한 그룹 테스트
- 실제 사용자 데이터 확인
- 단계적 롤아웃
```
