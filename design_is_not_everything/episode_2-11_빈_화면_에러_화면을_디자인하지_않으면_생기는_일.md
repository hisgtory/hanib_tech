# Episode 2-11: 빈 화면, 에러 화면을 디자인하지 않으면 생기는 일

디자이너가 행복한 상태(Happy Path)만 디자인하면 어떤 일이 벌어질까요? 실제 서비스에서는 데이터가 없거나, 네트워크가 끊기거나, 서버가 다운되는 등 다양한 예외 상황이 발생합니다. 이런 상황을 고려하지 않으면 사용자 경험은 급격히 나빠지고, 개발자는 급하게 임시방편으로 처리하게 됩니다.

- 🎯 핵심 개념: Empty State와 Error State는 사용자 경험의 중요한 터치포인트입니다. 이 순간이 사용자가 서비스를 떠나거나 계속 사용하는 분기점이 될 수 있습니다.

## 빈 화면(Empty State)이란?

Empty State는 표시할 데이터가 없을 때 보여주는 화면입니다. 이는 버그가 아니라 정상적인 상황일 수 있습니다:

- 신규 사용자가 처음 서비스를 사용할 때
- 검색 결과가 없을 때
- 사용자가 모든 항목을 삭제했을 때
- 필터링 결과가 없을 때

### 좋은 Empty State의 구성 요소

- 시각적 요소: 상황을 설명하는 일러스트나 아이콘
- 명확한 제목: 현재 상황을 간단히 설명
- 설명 텍스트: 왜 비어있는지, 어떻게 채울 수 있는지
- 행동 유도(CTA): 다음 단계로 안내하는 버튼

## 에러 화면(Error State)의 중요성

에러는 예상치 못한 상황에서 발생합니다. 사용자는 당황하고 불안해합니다. 이때 어떻게 대응하느냐가 서비스의 신뢰도를 결정합니다.

### 자주 놓치는 에러 상황들

- 네트워크 연결 끊김
- 서버 응답 시간 초과(Timeout)
- 권한 없음(403 Forbidden)
- 페이지를 찾을 수 없음(404 Not Found)
- 서버 에러(500 Internal Server Error)
- 업로드 실패
- 결제 실패

### 효과적인 에러 화면의 원칙

- 사용자 친화적인 언어: 기술 용어 대신 쉽게 이해할 수 있는 표현 사용
- 문제 설명: 무엇이 잘못되었는지 명확히 전달
- 해결 방법 제시: 사용자가 취할 수 있는 행동 안내
- 심각도 표현: 아이콘이나 색상으로 중요도 구분
- 도움 요청 방법: 고객센터나 지원팀 연락처

## 실제 사례: 빈 화면과 에러 화면을 놓쳤을 때

### 1. 개발자의 임시방편

디자인이 없는 상태에서 개발자들은 다음과 같은 처리를 하게 됩니다:
- 빈 하얀 화면만 보여주기
- "No data" 같은 간단한 텍스트만 표시
- 기술적인 에러 코드 그대로 노출
- 아무런 안내 없이 무한 로딩만 보여주기

### 2. 사용자 경험의 악화

- 혼란과 불안: 무엇이 잘못된 건지, 내가 잘못한 건지 파악 불가
- 신뢰도 하락: 서비스가 안정적이지 않다는 인상
- 이탈률 증가: 다시 접속하지 않을 가능성 높음
- 네가티브 후기: SNS나 리뷰에 부정적인 경험 공유

### 3. 비즈니스 손실

- 전환율 감소: 혹은한 사용자들이 가입/구매 포기
- 고객 지원 비용 증가: 혼란스러운 사용자들의 문의 쇄도
- 브랜드 이미지 손상: 전문성 부족으로 비쳐짐

## 베스트 프랙티스: 좋은 Empty State와 Error State 디자인

### Empty State 디자인 체크리스트

- ☑️ 상황에 맞는 일러스트레이션 준비
- ☑️ 명확하고 긍정적인 메시지 작성
- ☑️ 다음 행동을 유도하는 CTA 버튼 배치
- ☑️ 검색/필터 결과가 없을 때 키워드 표시
- ☑️ 온보딩 시 기능 설명 포함

### Error State 디자인 체크리스트

- ☑️ 에러 유형별 메시지 구성
- ☑️ 기술 용어 지양, 사용자 친화적 언어 사용
- ☑️ 재시도 버튼 제공
- ☑️ 고객 지원 연락처 표시
- ☑️ 상태 코드별 적절한 비주얼 아이콘

## 실무 팁: 개발자와 협업하기

- 프로젝트 초기에 Empty/Error State 디자인 시스템 구축
- 가능한 모든 예외 상황 리스트 작성
- HTTP 상태 코드별 화면 매핑
- 로딩 상태, 스켈레튨, 새로고침 상태도 고려
- 디자인 시스템에 컴포넌트로 추가

## 마무리

Empty State와 Error State는 사용자가 예상치 못한 상황에 마주하는 순간입니다. 이 순간을 얼마나 잘 디자인하느냐에 따라 사용자는 서비스를 계속 사용할지, 또는 떠날지를 결정합니다.

개발자와 디자이너가 함께 이러한 예외 상황을 미리 고려하고 준비한다면, 사용자는 더 안정적이고 신뢰할 수 있는 서비스를 경험하게 될 것입니다. Happy Path만큼이나 Unhappy Path도 중요합니다!

- 💡 기억해야 할 포인트: "디자인이 없는 예외 상황은 사용자 경험의 블랙홀이다."
