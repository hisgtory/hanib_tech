# Episode 2-11: 빈 화면, 에러 화면을 디자인하지 않으면 생기는 일

디자이너가 행복한 상태(Happy Path)만 디자인하면 어떤 일이 벌어질까요? 실제 서비스에서는 데이터가 없거나, 네트워크가 끊기거나, 서버가 다운되는 등 다양한 예외 상황이 발생합니다. 이런 상황을 고려하지 않으면 사용자 경험은 급격히 나빠지고, 개발자는 급하게 임시방편으로 처리하게 됩니다.

- 🎯 핵심 개념: Empty State와 Error State는 사용자 경험의 중요한 터치포인트입니다. 이 순간이 사용자가 서비스를 떠나거나 계속 사용하는 분기점이 될 수 있습니다.

## 빈 화면(Empty State)이란?

Empty State는 표시할 데이터가 없을 때 보여주는 화면입니다. 이는 버그가 아니라 정상적인 상황일 수 있습니다:

- 신규 사용자가 처음 서비스를 사용할 때
- 검색 결과가 없을 때
- 사용자가 모든 항목을 삭제했을 때
- 필터링 결과가 없을 때

### 좋은 Empty State의 구성 요소

- 시각적 요소: 상황을 설명하는 일러스트나 아이콘
- 명확한 제목: 현재 상황을 간단히 설명
- 설명 텍스트: 왜 비어있는지, 어떻게 채울 수 있는지
- 행동 유도(CTA): 다음 단계로 안내하는 버튼

## 에러 화면(Error State)의 중요성

에러는 예상치 못한 상황에서 발생합니다. 사용자는 당황하고 불안해합니다. 이때 어떻게 대응하느냐가 서비스의 신뢰도를 결정합니다.

### 자주 놓치는 에러 상황들

- 네트워크 연결 끊김
- 서버 응답 시간 초과(Timeout)
- 권한 없음(403 Forbidden)
- 페이지를 찾을 수 없음(404 Not Found)
- 서버 에러(500 Internal Server Error)
- 업로드 실패
- 결제 실패

### 효과적인 에러 화면의 원칙

- 사용자 친화적인 언어: 기술 용어 대신 쉽게 이해할 수 있는 표현 사용
- 문제 설명: 무엇이 잘못되었는지 명확히 전달
- 해결 방법 제시: 사용자가 취할 수 있는 행동 안내
- 심각도 표현: 아이콘이나 색상으로 중요도 구분
- 도움 요청 방법: 고객센터나 지원팀 연락처

## 실제 사례: 빈 화면과 에러 화면을 놓쳤을 때

### 1. 개발자의 임시방편

디자인이 없는 상태에서 개발자들은 다음과 같은 처리를 하게 됩니다:
- 빈 하얀 화면만 보여주기
- "No data" 같은 간단한 텍스트만 표시
- 기술적인 에러 코드 그대로 노출
- 아무런 안내 없이 무한 로딩만 보여주기

### 2. 사용자 경험의 악화

- 혼란과 불안: 무엇이 잘못된 건지, 내가 잘못한 건지 파악 불가
- 신뢰도 하락: 서비스가 안정적이지 않다는 인상
- 이탈률 증가: 다시 접속하지 않을 가능성 높음
- 네가티브 후기: SNS나 리뷰에 부정적인 경험 공유

### 3. 비즈니스 손실

- 전환율 감소: 혹은한 사용자들이 가입/구매 포기
- 고객 지원 비용 증가: 혼란스러운 사용자들의 문의 쇄도
- 브랜드 이미지 손상: 전문성 부족으로 비쳐짐

## 베스트 프랙티스: 좋은 Empty State와 Error State 디자인

### Empty State 디자인 체크리스트

- ☑️ 상황에 맞는 일러스트레이션 준비
- ☑️ 명확하고 긍정적인 메시지 작성
- ☑️ 다음 행동을 유도하는 CTA 버튼 배치
- ☑️ 검색/필터 결과가 없을 때 키워드 표시
- ☑️ 온보딩 시 기능 설명 포함

### Error State 디자인 체크리스트

- ☑️ 에러 유형별 메시지 구성
- ☑️ 기술 용어 지양, 사용자 친화적 언어 사용
- ☑️ 재시도 버튼 제공
- ☑️ 고객 지원 연락처 표시
- ☑️ 상태 코드별 적절한 비주얼 아이콘

## 실무 팁: 개발자와 협업하기

- 프로젝트 초기에 Empty/Error State 디자인 시스템 구축
- 가능한 모든 예외 상황 리스트 작성
- HTTP 상태 코드별 화면 매핑
- 로딩 상태, 스켈레튨, 새로고침 상태도 고려
- 디자인 시스템에 컴포넌트로 추가

## 마무리

Empty State와 Error State는 사용자가 예상치 못한 상황에 마주하는 순간입니다. 이 순간을 얼마나 잘 디자인하느냐에 따라 사용자는 서비스를 계속 사용할지, 또는 떠날지를 결정합니다.

개발자와 디자이너가 함께 이러한 예외 상황을 미리 고려하고 준비한다면, 사용자는 더 안정적이고 신뢰할 수 있는 서비스를 경험하게 될 것입니다. Happy Path만큼이나 Unhappy Path도 중요합니다!

- 💡 기억해야 할 포인트: "디자인이 없는 예외 상황은 사용자 경험의 블랙홀이다."

## 피드백 및 개선 제안사항

### 강점
- Happy Path가 아닌 예외 상황의 중요성을 명확히 제시
- Empty State와 Error State의 차이점을 체계적으로 설명
- 비즈니스 임팩트까지 고려한 포괄적 접근
- 실무에서 바로 적용할 수 있는 체크리스트 제공

### 개선 제안사항

#### 1. 디자이너의 실제 경험담 추가
**예외 상황 디자인의 중요성을 깨달은 과정:**
- "처음에는 '데이터가 있을 때'의 화면만 예쁘게 만드는 게 중요하다고 생각했어요"
- "실제 서비스 런칭 후 사용자 피드백을 보니, 에러 화면에서 이탈하는 사용자가 많다는 걸 알았어요"
- "이제는 Empty State와 Error State를 먼저 디자인한 다음에 Happy Path를 디자인해요"

#### 2. 실제 팀 대화 시나리오
**디자이너-개발자 간 예외 상황 논의:**

```
👨‍💻 개발자: "API 응답이 없을 때는 어떻게 처리할까요?"
👩‍🎨 디자이너: "음... 그냥 '데이터가 없습니다'라고 텍스트만 보여주면 안 되나요?"
👨‍💻 개발자: "사용자가 혼란스러워할 것 같은데요. 어떻게 해결해야 하는지 모르잖아요"

[개선 후]
👩‍🎨 디자이너: "데이터가 없을 때 시나리오를 정리해봤어요. 신규 사용자용, 검색 결과 없음용, 네트워크 에러용으로 나눠서 각각 다른 메시지와 CTA를 준비했어요"
👨‍💻 개발자: "좋네요! 그럼 상황별로 컴포넌트를 만들어서 재사용할 수 있겠어요"
```

#### 3. 상황별 세부 가이드
**구체적인 Empty State 및 Error State 시나리오:**

**Empty State 세분화:**
```
1. 첫 방문자 (First-time user)
   - 메시지: "아직 [콘텐츠]가 없어요"
   - CTA: "첫 [콘텐츠] 만들기"
   - 이미지: 안내하는 캐릭터

2. 검색 결과 없음 (No search results)
   - 메시지: "'[검색어]'에 대한 결과가 없어요"
   - CTA: "다른 키워드로 검색" / "전체 보기"
   - 이미지: 돋보기 아이콘

3. 필터 결과 없음 (No filter results)
   - 메시지: "조건에 맞는 결과가 없어요"
   - CTA: "필터 초기화" / "조건 변경"
   - 이미지: 필터 아이콘
```

**Error State 심각도별 분류:**
```
경미한 오류 (Warning)
- 노란색 계열, ⚠️ 아이콘
- 예: "일시적으로 접속이 원활하지 않아요"

보통 오류 (Error)  
- 주황색 계열, ❌ 아이콘
- 예: "요청을 처리할 수 없어요"

심각한 오류 (Critical)
- 빨간색 계열, 🚨 아이콘
- 예: "시스템에 문제가 발생했어요"
```

#### 4. 실패 사례와 개선 과정
**실제 프로젝트에서 겪은 문제들:**

**사례 1: 네트워크 에러 처리 부재**
- 문제: 네트워크 끊김 시 무한 로딩만 표시
- 결과: 사용자들이 앱 강제 종료 후 재시작
- 개선: 타임아웃 후 "네트워크 확인 후 다시 시도해주세요" 메시지와 재시도 버튼

**사례 2: 검색 결과 없을 때 대응 부재**
- 문제: "검색 결과가 없습니다"만 표시
- 결과: 사용자들이 다른 검색을 시도하지 않고 이탈
- 개선: 유사한 키워드 제안, 인기 검색어 표시, 전체 카테고리 보기 제공

#### 5. 실무 도구 및 템플릿
**바로 사용할 수 있는 실무 자료:**

**예외 상황 매핑 템플릿:**
```
| 상황 | 사용자 감정 | 메시지 톤 | 권장 액션 | 우선순위 |
|------|------------|----------|-----------|----------|
| 첫 방문 | 호기심 | 환영하는 | 시작하기 | High |
| 검색 실패 | 실망 | 도움되는 | 대안 제시 | High |  
| 네트워크 에러 | 불안 | 안심시키는 | 재시도 | Critical |
| 서버 에러 | 분노 | 사과하는 | 지원 연결 | Critical |
```

**컴포넌트 디자인 가이드:**
```
Empty State 컴포넌트
├── 이미지 영역 (120x120px)
├── 제목 (Heading-M, 진한 회색)
├── 설명 (Body-S, 연한 회색)
└── 액션 버튼 (Primary 또는 Secondary)

Error State 컴포넌트  
├── 아이콘 영역 (48x48px, 색상별)
├── 제목 (Heading-S, 상태별 색상)
├── 설명 (Body-S, 해결 방법 포함)
├── 재시도 버튼 (Primary)
└── 지원 링크 (Text Link)
```

#### 6. 사용자 테스트 관점
**예외 상황 사용자 테스트 방법:**
- A/B 테스트: 다양한 메시지 톤과 CTA 효과 비교
- 사용자 인터뷰: 에러 상황에서의 감정과 기대사항 파악
- 행동 분석: 예외 상황 후 이탈률과 재방문율 측정
- 에러 로그 분석: 실제 발생하는 에러 패턴 파악

#### 7. 접근성과 다국어 고려사항
**포용적 디자인 접근:**
- 색상에만 의존하지 않는 상태 표현 (아이콘, 텍스트 병행)
- 스크린 리더를 위한 대체 텍스트
- 문화별 감정 표현 차이 고려
- 간결하고 명확한 언어로 번역 용이성 확보
