## 🎬 Scene 1: 디자이너의 첫 출근날

### 09:00 AM - 엘리베이터 앞에서

"안녕하세요! 오늘 첫 출근하신 디자이너분이시죠? 저는 프론트엔드 개발자 제이크입니다."

첫 만남부터 '프론트엔드'라는 단어에 살짝 긴장하는 당신. 하지만 걱정 마세요. 오늘 하루가 끝나면 이 모든 용어들이 친구처럼 느껴질 거예요.

---

## 📊 우리 팀의 진짜 모습 (현실 버전)

### 스쿼드(Squad)란? 
> **카페 창업에 비유하자면:**
> - PO = 카페 사장 (무슨 카페를 만들지 결정)
> - Designer = 인테리어 디자이너 (카페 분위기와 동선 설계)
> - Frontend = 바리스타 (손님이 직접 만나는 사람)
> - Backend = 로스터 & 재고관리자 (뒤에서 원두 공급과 시스템 관리)

소프트웨어 제품을 만들기 위한 최소 인원으로 구성된 팀은 다음과 같아요.

---

## 👥 팀원별 역할 상세 가이드

### 1. Product Owner (PO/PM)
**"무엇을 만들지 정하는 사람"**

#### 주요 업무:
- 문제 정의와 가설 수립
- 우선순위 결정
- 성공 지표 설정

#### 주 사용 툴:
- Notion, Confluence (문서화)
- Jira, Monday (일정 관리)
- Google Analytics (데이터 분석)

#### 디자이너와의 협업 포인트:
```
💬 실제 대화 예시:
PO: "이번 스프린트에서 결제 플로우 개선이 목표예요"
Designer: "사용자 리서치 결과 보면 3단계가 너무 복잡해요"
PO: "그럼 2단계로 줄이면 전환율이 얼마나 오를까요?"
Designer: "A/B 테스트해볼 수 있게 두 버전 준비할게요"
```

---

### 2. Product Designer (PD)
**"어떻게 보일지 만드는 사람 (바로 당신!)"**

#### 주요 업무:
- UX 리서치와 와이어프레임
- UI 디자인과 프로토타이핑
- 디자인 시스템 관리

#### 주 사용 툴:
- Figma (메인 디자인 툴)
- Principle, ProtoPie (프로토타이핑)
- Zeplin (개발 핸드오프)

#### 개발자와의 협업 꿀팁:
```
✅ DO:
- 8px 그리드 시스템 사용
- 컴포넌트 일관성 유지
- 엣지케이스 디자인 준비

❌ DON'T:
- 픽셀 단위 애매하게 하기
- 갑작스러운 디자인 변경
- 개발 중간에 "조금만 수정"
```

---

### 3. Frontend Developer
**"사용자가 보는 화면을 만드는 사람"**

#### 웹 프론트엔드 개발자:
- **역할**: 웹사이트 UI 구현
- **기술 스택**: HTML, CSS, JavaScript, React/Vue
- **협업 포인트**: 반응형 디자인, 브라우저 호환성

#### 앱 개발자 (iOS/Android):
- **역할**: 모바일 앱 UI 구현
- **기술 스택**: 
  - iOS: Swift, SwiftUI
  - Android: Kotlin, Jetpack Compose
- **협업 포인트**: 플랫폼별 가이드라인 준수

#### 실무 대화 예시:
```
💬 디자인 리뷰 미팅:
Designer: "이 애니메이션 300ms로 해주세요"
Frontend: "성능상 200ms가 적절할 것 같은데요"
Designer: "사용성 테스트에서 너무 빠르다는 피드백이..."
Frontend: "그럼 250ms로 중간 지점 찾아볼까요?"
```

---

### 4. Backend Developer
**"데이터와 로직을 관리하는 사람"**

#### 주요 업무:
- API 설계와 구현
- 데이터베이스 관리
- 서버 인프라 운영

#### 디자이너가 알아야 할 포인트:
```
🎯 "이건 서버에서 처리해야 해요"라고 할 때:
- 대용량 데이터 처리
- 복잡한 계산이나 알고리즘
- 보안이 중요한 정보
- 여러 사용자 간 동기화
```

## 🚨 실패 사례로 배우는 교훈

### Case 1: "디자인이 다 끝났는데요?"
**상황**: 디자인 완성 후 개발자에게 전달
**문제**: 기술적으로 구현 불가능한 인터랙션
**결과**: 2주 작업 폐기, 처음부터 다시

**교훈**: 
> 💡 디자인 초기 단계부터 개발자와 기술 검토 필수!

### Case 2: "간단한 수정인데..."
**상황**: 출시 직전 "버튼 색만 바꿔주세요"
**문제**: 전체 디자인 시스템과 충돌
**결과**: 3일 지연, QA 재진행

**교훈**:
> 💡 작은 변경도 시스템 전체에 영향을 줄 수 있음

### Case 3: "PO가 원래 그렇게 말했나?"
**상황**: 각자 다르게 이해한 요구사항
**문제**: 3주 후 완전히 다른 결과물
**결과**: 스프린트 실패

**교훈**:
> 💡 문서화와 동기화 미팅의 중요성

## 📈 스쿼드 진화 과정

### Phase 1: 초기 스타트업 (5-10명)
```
CEO(=PO) + Designer + Developer(풀스택)
모두가 모든 일을 조금씩
```

### Phase 2: 성장기 (20-50명)
```
PO + Designer + Frontend + Backend + QA
첫 번째 스쿼드 완성
```

### Phase 3: 확장기 (50-200명)
```
검색 스쿼드 | 결제 스쿼드 | 회원 스쿼드
각 스쿼드별 독립적 운영
```

### Phase 4: 대기업 (200명+)
```
트라이브(Tribe) > 스쿼드(Squad) > 챕터(Chapter)
Spotify 모델 적용
```

## 💬 팀 미팅에서 자주 나오는 대화

### 스프린트 플래닝
```
PO: "이번 스프린트 목표는 온보딩 개선입니다"
Designer: "유저 테스트 결과 3단계가 이탈 포인트예요"
Frontend: "애니메이션 추가하면 2일 더 필요해요"
Backend: "API는 이미 준비되어 있어요"
QA: "테스트 시나리오 5개 정도 나올 것 같아요"
```

### 데일리 스크럼
```
Designer: "상세 페이지 디자인 80% 완료했어요"
Frontend: "메인 페이지 퍼블리싱 끝났는데 리뷰 부탁드려요"
Backend: "API 문서 업데이트했으니 확인해주세요"
```

### 회고 미팅
```
"잘한 점: 디자인-개발 싱크가 잘 맞았다"
"아쉬운 점: 기획 변경이 너무 잦았다"
"개선할 점: 주간 싱크 미팅 추가하자"
```

## 🎁 보너스: 용어 사전

### 디자이너가 자주 듣는 개발 용어
- **PR (Pull Request)**: 코드 리뷰 요청
- **머지 (Merge)**: 코드 합치기
- **빌드 (Build)**: 앱 만들기
- **배포 (Deploy)**: 사용자에게 공개
- **롤백 (Rollback)**: 이전 버전으로 되돌리기
- **핫픽스 (Hotfix)**: 긴급 수정

### 개발자가 자주 듣는 디자인 용어
- **와이어프레임**: 뼈대 스케치
- **목업**: 실제와 유사한 디자인
- **프로토타입**: 인터랙션 포함 시연
- **디자인 시스템**: 디자인 규칙 모음
- **핸드오프**: 개발자에게 전달
