# Episode 2-4. 와이어프레임 대충 그리면 개발자가 울어요

와이어프레임은 제품의 뼈대를 그리는 첫 단계입니다. 디자이너가 머릿속에 있는 아이디어를 처음으로 시각화하는 순간이죠. 하지만 이 와이어프레임을 '대충' 그리면 어떤 일이 벌어질까요? 개발자는 물론이고, 팀 전체가 혼란에 빠질 수 있습니다.

이번 에피소드에서는 와이어프레임을 제대로 그리는 방법과 개발자와의 원활한 협업을 위한 실무 팁을 알아보겠습니다.

## 💡 와이어프레임이란 무엇인가요?

와이어프레임(Wireframe)은 화면의 기본 구조와 요소들의 배치를 보여주는 설계도입니다. 건축가가 집을 짓기 전에 평면도를 그리듯이, 디자이너도 UI를 만들기 전에 와이어프레임을 그립니다. 이는 색상이나 디테일한 디자인 요소 없이 오직 기능과 구조에만 집중한 가장 기본적인 형태의 시각화입니다.

## 🤔 왜 와이어프레임이 중요할까요?

- 시간과 비용 절약: 고퀄리티 디자인을 만들기 전에 기본 구조를 검증할 수 있습니다
- 커뮤니케이션 향상: 팀원들과 아이디어를 명확히 공유할 수 있습니다
- 빠른 수정: 디자인 단계로 넘어가기 전에 비즈니스 요구사항을 쉽게 반영할 수 있습니다
- 개발 가능성 확인: 기술적 제약사항이나 구현 가능성을 미리 확인할 수 있습니다

## 😱 '대충' 그린 와이어프레임의 문제점

**1. 모호한 레이아웃**

"여기에 버튼 하나 넣어주세요"라고만 적혀있다면, 개발자는 몇 가지 질문을 하게 됩니다:
- 버튼의 크기는 얼마나 되나요?
- 다른 요소들과의 간격은 어떻게 되나요?
- 반응형에서는 어떻게 표시되나요?

**2. 상태 표현의 부재**

버튼이나 입력 필드의 다양한 상태(호버, 클릭, 비활성화, 에러 등)를 표현하지 않으면 개발자는 세부 동작을 예측할 수 없습니다.

**3. 기능적 설명 부족**

예를 들어, "여기서 검색할 수 있음"이라고만 적혀있다면:
- 검색 결과는 어떻게 표시되나요?
- 자동완성 기능이 있나요?
- 필터링 기능도 필요한가요?

## 🎯 효과적인 와이어프레임 작성법

**1. 명확한 레이아웃 그리드 사용**

8포인트 그리드를 사용하면 요소들의 정확한 배치와 크기를 표현할 수 있습니다. 예를 들어:
- 헤더: 64px 높이
- 컨텐츠 영역: 좌우 16px 패딩
- 버튼 간격: 8px

**2. 의미 있는 레이블링**

모든 요소에는 기능을 설명하는 레이블을 붙여야 합니다:
- "로그인 버튼 (이메일/비밀번호 입력 후)"
- "검색 결과 영역 (10개씩 페이지네이션)"
- "프로필 이미지 (클릭 시 프로필 편집 페이지로 이동)"

**3. 인터랙션 표현**

화살표와 주석을 통해 사용자의 행동 흐름을 설명합니다:
- 버튼 클릭 → 모달 팝업
- 스크롤 다운 → 무한 스크롤
- 탭 전환 → 컨텐츠 변경

**4. 반응형 고려**

데스크탑, 태블릿, 모바일 버전을 모두 고려해야 합니다:
- 데스크탑: 3열 레이아웃
- 태블릿: 2열 레이아웃
- 모바일: 1열 레이아웃

## 🛠️ 개발자와의 협업을 위한 팁

**1. 기술적 제약사항 확인**

와이어프레임을 완성한 후 개발자와 함께 리뷰하면서 다음을 확인합니다:
- 기술적으로 구현 가능한지
- 성능 이슈가 예상되는 부분이 있는지
- 서드파티 API나 라이브러리가 필요한지

**2. 컴포넌트 단위로 생각하기**

개발자는 컴포넌트 단위로 생각합니다. 와이어프레임에서도 재사용 가능한 요소들을 표시해주면 좋습니다:
- "이 카드 컴포넌트는 여러 곳에서 사용됩니다"
- "이 버튼은 전역 컴포넌트로 사용합니다"

**3. 예외 상황 고려**

다음과 같은 예외 상황도 함께 표현해야 합니다:
- 데이터가 없을 때 (Empty State)
- 로딩 중일 때
- 에러가 발생했을 때
- 권한이 없을 때

## 📚 실제 사례: 잘된 와이어프레임 vs 문제가 있는 와이어프레임

**❌ 문제가 있는 예시:**

- "여기 리스트"
- "버튼"
- "텍스트 입력"

**✅ 잘된 예시:**

- "상품 목록 (10개씩, 무한 스크롤, 필터 기능 포함)"
- "구매하기 버튼 (Primary, 56px 높이, 비활성화 상태 표현)"
- "검색어 입력 필드 (40px 높이, 자동완성 기능, 최근 검색어 5개 표시)"

## 🖍️ 와이어프레임 도구 추천

- 사용자 흐름과 인터랙션이 표현되어 있는가?
- 반응형 디자인이 고려되어 있는가?

- 예외 상황이 표현되어 있는가?

## 💡 마무리

와이어프레임은 단순한 화면 설계도가 아닙니다. 디자이너와 개발자, 그리고 전체 팀이 함께 제품을 만들어가는 첫 번째 단계입니다. 세심하고 명확한 와이어프레임은 개발 과정에서의 커뮤니케이션 비용을 크게 줄이고, 모두가 같은 목표를 향해 나아가도록 도와줍니다.

## 피드백 및 개선 제안사항

### 강점
- 와이어프레임의 중요성과 구체적인 작성 방법을 체계적으로 설명
- 문제가 있는 예시와 올바른 예시를 명확히 대비하여 제시
- 개발자와의 협업을 위한 실용적인 팁 포함
- 체크리스트를 통한 실행 가능한 가이드라인 제공

### 개선 제안사항

#### 1. 디자이너 관점 강화
**실제 디자이너들의 어려움과 고민 반영:**
- "처음에는 와이어프레임이 단순해 보여서 대충 그렸는데, 나중에 개발자가 질문을 받고 나서야 얼마나 중요한지 깨달았어요" - 주니어 UX 디자이너 경험담
- 디자이너가 와이어프레임 단계에서 겪는 실제 딜레마와 의사결정 과정 추가

#### 2. 실제 팀 대화 예시
**디자이너-개발자 간 실제 대화 시나리오 추가:**

```
👩‍🎨 디자이너: "이 부분에 검색 기능을 넣고 싶어요"
👨‍💻 개발자: "검색 결과는 어떻게 표시하실 건가요?"
👩‍🎨 디자이너: "음... 그냥 리스트로요?"
👨‍💻 개발자: "검색어가 없을 때는요? 결과가 없을 때는요? 로딩 중일 때는요?"
👩‍🎨 디자이너: "아, 그런 것까지 생각해야 하는군요!"

→ 개선된 대화:
👩‍🎨 디자이너: "검색 기능을 추가하려고 하는데, 이런 상태들을 고려해봤어요:
- 기본 상태: placeholder '상품명을 입력하세요'
- 검색 중: 로딩 스피너
- 결과 있음: 리스트 형태로 10개씩 표시
- 결과 없음: '검색 결과가 없습니다' 메시지
어떻게 보시나요?"
```

#### 3. 실패 사례와 학습 경험
**실제로 발생한 문제 사례들:**
- "와이어프레임에 반응형을 고려하지 않아서 모바일에서 레이아웃이 완전히 깨진 사례"
- "상태 변화를 표현하지 않아서 개발 중간에 전체 UI가 바뀐 경험"
- "컴포넌트 재사용성을 고려하지 않아서 개발 시간이 2배로 늘어난 사례"

#### 4. 실용적 도구 및 템플릿
