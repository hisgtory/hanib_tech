## Part 1: 암호화, 왜 디자이너가 알아야 하나?

```

❌ domain.com/user/주민번호
```

## Part 2: 암호화 기초 개념 (디자이너 버전)

### 🔑 암호화란?

```
일상 비유:
암호화 = 자물쇠
키 = 열쇠
평문 = 원본
암호문 = 잠긴 상태

"안녕하세요" 
    ↓ (암호화)
"x7@9kL#mP"
    ↓ (복호화)
"안녕하세요"
```

### 🎨 디자이너가 꼭 알아야 할 암호화 용어

```
1. HTTPS (Hypertext Transfer Protocol Secure)
= 웹사이트 자물쇠 🔒
디자인 포인트: 브라우저 주소창에 자물쇠 아이콘

2. SSL/TLS
= 데이터 전송 보안
디자인 포인트: "보안 연결" 표시

3. 해싱 (Hashing)
= 단방향 변환 (되돌릴 수 없음)
용도: 비밀번호 저장
디자인 포인트: "비밀번호 찾기" 대신 "재설정"

4. 암호화 (Encryption)
= 양방향 변환 (되돌릴 수 있음)
용도: 메시지, 파일
디자인 포인트: 잠금/해제 아이콘

5. End-to-End (종단간)
= 중간에 아무도 못 봄
예시: WhatsApp, Signal
디자인 포인트: "종단간 암호화" 뱃지
```

## Part 4: 실수하기 쉬운 보안 이슈들

### 💥 디자이너가 자주 하는 실수

#### 실수 1: URL에 민감정보 노출

```
❌ 잘못된 설계:
mysite.com/order?card=1234567812345678&cvv=123
→ 브라우저 히스토리에 저장됨
→ 서버 로그에 기록됨
→ 공유시 노출됨

✅ 올바른 설계:
mysite.com/order/a7x9k2m
→ 세션이나 쿠키로 처리
→ POST 방식 사용
```

#### 실수 2: 자동완성 무분별 사용

```
❌ 위험한 자동완성:
<input type="text" autocomplete="on" name="credit-card">
<input type="text" autocomplete="on" name="ssn">

✅ 안전한 설정:
<input type="password" autocomplete="new-password">
<input type="text" autocomplete="off" name="verification-code">
```

#### 실수 3: 클립보드 보안 무시

```
❌ 문제 상황:
- 비밀번호 복사 가능
- OTP 코드 복사 가능
- 카드번호 전체 복사

✅ 해결 방법:
CSS: user-select: none;
JS: 복사 이벤트 차단
UX: 복사 필요시 부분만 허용
```

## Part 5: 보안과 사용성의 균형

### ⚖️ Trade-off 관리

```
보안 🔒 ←→ 🎨 편의성

너무 엄격하면:
- 사용자 이탈
- 고객 불만
- 서비스 포기

너무 느슨하면:
- 보안 사고
- 법적 책임
- 신뢰 상실

황금 균형점:
1. 리스크 레벨별 차등 적용
2. 컨텍스트 기반 보안
3. 점진적 보안 강화
```

### 🎯 리스크 레벨별 UX 전략

```
Level 1 - Low Risk (뉴스레터 구독):
- 이메일만 요구
- 인증 선택적
- 소셜 로그인 OK

Level 2 - Medium Risk (일반 회원가입):
- 이메일 인증 필수
- 기본 비밀번호 규칙
- 캡차 적용

Level 3 - High Risk (결제/금융):
- 2단계 인증
- 생체 인증
- 세션 타임아웃
- 거래 알림

Level 4 - Critical (관리자/의료):
- 다단계 인증
- IP 제한
- 접속 기록
- 정기 비밀번호 변경
```
