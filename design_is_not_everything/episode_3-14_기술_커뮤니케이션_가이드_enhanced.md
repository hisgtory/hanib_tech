## Part 1: 개발자 언어 사전 - 자주 듣는 용어 100선

### 🔤 일상 대화편

#### "이거 하드코딩 되어 있어요"
```
뜻: 값을 직접 코드에 박아놨다
예: 버튼 색상을 #FF5733으로 직접 적음
문제: 나중에 바꾸려면 코드 다 찾아서 수정해야 함
대안: "변수로 빼서 관리하면 어때요?"
```

#### "레거시 코드가 너무 많아서..."
```
뜻: 오래된 코드가 많다
의미: 수정하기 무섭고 위험하다
대응: "리팩토링 일정 잡아야 하나요?"
```

#### "의존성 지옥에 빠졌어요"
```
뜻: A를 바꾸려면 B도 바꿔야 하고, B를 바꾸려면 C도...
```

## Part 2: 효과적인 요청 방법

### ❌ 나쁜 요청 vs ✅ 좋은 요청

#### Case 1: 기능 추가 요청

```markdown
❌ 나쁜 예:
"여기에 애니메이션 넣어주세요."

문제점:
- 어떤 애니메이션?
- 언제 실행?
- 성능 영향은?

✅ 좋은 예:
"사용자가 버튼을 클릭하면 0.3초 동안 페이드 인 애니메이션을 
보여주고 싶어요. 모바일에서도 부드럽게 동작해야 합니다.
참고 레퍼런스는 이거예요: [링크]"

좋은 점:
- 구체적인 트리거
- 명확한 duration
- 성능 고려
- 레퍼런스 제공
```

#### Case 2: 버그 리포트

```markdown
❌ 나쁜 예:
"로그인이 안 돼요."

문제점:
- 어떤 상황에서?
- 어떤 계정으로?
- 에러 메시지는?

✅ 좋은 예:
"재현 경로:
1. Chrome 120버전, macOS
2. test@example.com 계정
3. 비밀번호 3번 틀린 후 재시도
4. 올바른 비밀번호 입력해도 '잘못된 비밀번호' 에러
5. 콘솔 에러: 'rate limit exceeded'

예상 동작: 비밀번호 재설정 안내
실제 동작: 계속 로그인 시도 가능"
```

#### Case 3: 디자인 변경 요청

```markdown
❌ 나쁜 예:
"이 화면 좀 이쁘게 바꿔주세요."

✅ 좋은 예:
"현재 전환율이 2%인데, 업계 평균 5%를 목표로 
CTA 버튼을 더 눈에 띄게 수정하고 싶습니다.

변경사항:
- 버튼 크기: 44px → 56px
- 색상: #gray → #blue (AA 기준 충족)
- 위치: 스크롤 없이 보이는 영역으로

A/B 테스트로 검증 예정입니다."
```
