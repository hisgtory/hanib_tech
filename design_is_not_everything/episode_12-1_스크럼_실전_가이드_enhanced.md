# Episode 12-1: 스크럼, 이제는 필수다 - 실전 가이드

## 🎬 Scene: 디자이너가 처음 스크럼을 경험하던 날

"스프린트? 백로그? 번다운 차트? 이게 다 뭐예요?"

첫 스크럼 미팅에 참석한 주니어 디자이너 민지의 표정은 당황 그 자체였다. 개발자들은 익숙한 듯 스토리 포인트를 논의하고 있었지만, 민지에게는 외계어처럼 들릴 뿐이었다.

"걱정 마세요. 처음엔 다 그래요. 3개월만 지나면 스크럼 없이는 일 못한다고 할 거예요."

시니어 디자이너 현주의 말이 과연 맞을까? 

**스포일러: 맞습니다. 그리고 이 에피소드를 읽고 나면 당신도 스크럼 전도사가 될 거예요.**

## Part 1: 스크럼, 디자이너 관점에서 이해하기

### 🎯 스크럼의 핵심 개념 (디자이너 버전)

#### 1. 스프린트 (Sprint)
**개발자 설명**: "2주 단위로 개발하는 거예요"
**디자이너 이해**: "2주 안에 완성 가능한 디자인을 준비하고 구현까지 함께 보는 것"

```
📅 2주 스프린트 - 디자이너의 실제 일정

Week 1:
월: 스프린트 계획 → 이번 스프린트 작업 확정
화-수: 디자인 작업 → 와이어프레임, 목업
목: 디자인 리뷰 → 개발팀과 구현 가능성 논의
금: 디자인 핸드오프 → 개발 시작

Week 2:
월-화: 개발 진행 체크 → 디자인 QA
수: 중간 점검 → 필요시 수정
목: 최종 QA → 디자인 검수
금: 스프린트 리뷰 & 회고
```

#### 2. 백로그 (Backlog)
**일반적 정의**: "해야 할 일의 목록"
**디자이너 관점**: "우선순위가 정해진 디자인 작업 리스트"

### 💬 실제 스크럼 미팅 대화

#### Daily Standup (매일 아침 10분)

**❌ 잘못된 예시:**
```
스크럼 마스터: "민지님, 어제 뭐 하셨어요?"
민지: "음... 디자인했어요."
스크럼 마스터: "오늘은요?"
민지: "계속 디자인할 거예요."
스크럼 마스터: "블로커는요?"
민지: "없어요."
```

**✅ 올바른 예시:**
```
민지: "어제는 결제 플로우 3개 화면 중 2개를 완성했습니다.
      오늘은 마지막 화면 완성하고 프로토타입 만들 예정이에요.
      블로커는 결제 실패 케이스 시나리오가 명확하지 않아서,
      PM님께 확인 필요합니다."

PM: "아, 그거 제가 오전에 정리해서 드릴게요!"
개발자: "프로토타입 나오면 바로 개발 시작할 수 있을 것 같네요."
```

## Part 2: 스프린트 계획, 디자이너는 뭘 준비해야 하나?

### 📋 스프린트 계획 회의 준비 체크리스트

```markdown
## 디자이너의 스프린트 계획 준비물

### 필수 준비
- [ ] 지난 스프린트 미완성 작업 정리
- [ ] 이번 스프린트 디자인 작업 목록
- [ ] 각 작업의 예상 소요 시간
- [ ] 디자인 디펜던시 체크 (선행 작업 필요 여부)
- [ ] 리소스 준비 상태 (아이콘, 이미지 등)

### 스토리 포인트 산정 기준 (디자인)
- 1 point: 간단한 수정 (색상, 텍스트 변경)
- 2 points: 기존 컴포넌트 활용한 새 화면
- 3 points: 새로운 플로우 설계 필요
- 5 points: 복잡한 인터랙션 설계 포함
- 8 points: 전체 서비스 리디자인 수준

### 질문 준비
- [ ] 이번 스프린트 목표는 무엇인가?
- [ ] 디자인 리뷰는 언제 진행하나?
- [ ] 개발 핸드오프 데드라인은?
- [ ] QA 일정은 어떻게 되나?
```

### 🎲 스토리 포인트 산정 실습

**케이스 1: "로그인 화면에 소셜 로그인 버튼 추가"**
```
디자이너 생각: "버튼 하나 추가? 1포인트!"
실제 필요 작업:
- 소셜 로그인 3종 버튼 디자인
- 각 상태별 디자인 (normal, hover, pressed, disabled)
- 에러 케이스 화면 디자인
- 약관 동의 플로우 추가
- 기존 화면과의 일관성 체크

결론: 3 포인트
```

## Part 3: 실패 사례로 배우는 스크럼

### 💥 실패 사례 1: "디자인이 늦어서 스프린트 실패"

**상황**: 2주 스프린트, 디자이너는 첫 주 금요일에 디자인 전달
**결과**: 개발팀 3일만에 구현 불가능, 스프린트 목표 미달성

**문제 분석**:
```
Week 1:
월: "복잡해서 시간이 좀 걸릴 것 같아요"
화-수: 완벽한 디자인 추구
목: "거의 다 됐어요, 내일까지는..."
금: 오후 6시 디자인 전달

Week 2:
월: 개발자 "이제 시작해야 하는데 시간이..."
금: 50% 구현으로 스프린트 종료
```

**교훈과 해결책**:
1. **Iteration over Perfection**: 70% 완성도로 빠르게 전달
2. **Progressive Enhancement**: 기본 디자인 먼저, 디테일은 순차적으로
3. **Daily Sync**: 매일 진행 상황 공유

### 💥 실패 사례 2: "스프린트 중간에 디자인 대폭 변경"

**대화 재현**:
```
Day 3: 
디자이너: "어제 밤에 더 좋은 아이디어가 떠올랐어요!"
개발자: "이미 50% 구현했는데요..."
PM: "일정은 못 미루는데..."

결과: 야근 → 번아웃 → 팀 갈등
```

**해결 방법**:
- **Design Freeze**: 개발 시작 후 핵심 변경 금지
- **Minor Changes Only**: 색상, 간격 등 마이너 수정만 허용
- **Next Sprint Queue**: 큰 변경은 다음 스프린트로

### 💥 실패 사례 3: "회고 시간 = 비난 시간"

**잘못된 회고**:
```
"디자인이 늦게 나와서 문제였어요"
"개발이 느려서 그래요"
"기획이 자꾸 바뀌어서..."
```

**올바른 회고 - Start/Stop/Continue**:
```
Start (시작할 것):
- 디자인 중간 리뷰 도입
- 컴포넌트 라이브러리 활용

Stop (멈출 것):
- 스프린트 중간 스펙 변경
- 구두로만 요청하기

Continue (계속할 것):
- 매일 스탠드업 미팅
- 금요일 오후 배포 금지
```

## Part 4: 스크럼 도구 완벽 활용법

```markdown
### 자동화 룰
- 디자인 완료 → 자동으로 개발 티켓 생성
- QA 발견 이슈 → 디자이너 자동 태그
```

### 📊 번다운 차트 읽기

```
이상적인 번다운 차트:
     ↘
       ↘
         ↘
           ↘
             ↘

실제 디자인팀 번다운 차트:
     →→↘
         ↓ (갑자기 일 추가)
        ↗↘
           →→ (마지막에 몰아치기)
              ↓
```

**번다운 차트가 평평할 때 (→→→)**:
- 원인: 디자인 작업이 '진행 중'에만 머물러 있음
- 해결: 작업을 더 작은 단위로 분할

## Part 5: 애자일 세레모니 실전 가이드

### 🎭 각 세레모니별 디자이너 역할

#### 1. Sprint Planning (스프린트 계획)
**시간**: 2-4시간
**디자이너 역할**:
- 디자인 작업 소개 및 스토리 포인트 협의
- 기술적 제약사항 확인
- 디자인 핸드오프 일정 확정

#### 2. Daily Standup (데일리 스탠드업)
**시간**: 15분 이내
**디자이너 발언 구조**:
```
1. Yesterday: "어제 완성한 것"
2. Today: "오늘 할 것"
3. Blockers: "도움 필요한 것"
```

#### 3. Sprint Review (스프린트 리뷰)
**시간**: 1-2시간
**디자이너 역할**:
- 구현된 디자인 시연
- 디자인 의도 설명
- 피드백 수집 및 기록

#### 4. Retrospective (회고)
**시간**: 1-1.5시간
**디자이너 기여**:
- 디자인 프로세스 개선점 제안
- 협업 이슈 공유
- 액션 아이템 도출

### 📈 스크럼 성숙도 레벨

```
Level 1 - Chaos (혼돈)
- 스프린트가 뭔지 모름
- 마감 직전 디자인 전달
- 회고 없음

Level 2 - Learning (학습)
- 스프린트 사이클 이해
- 가끔 제시간 전달
- 형식적 회고

Level 3 - Practicing (실천)
- 스토리 포인트 산정 참여
- 80% 제시간 전달
- 의미있는 회고

Level 4 - Optimizing (최적화)
- 베로시티 예측 가능
- 95% 제시간 전달
- 개선 액션 실행

Level 5 - Master (마스터)
- 스프린트 리듬 체득
- 팀 전체 생산성 향상
- 지속적 개선
```

## 🎁 Bonus: 스크럼 마스터가 되고 싶다면?

### 디자이너 출신 스크럼 마스터의 장점

1. **시각적 소통 능력**
   - 복잡한 프로세스를 다이어그램으로
   - 보드를 예쁘고 직관적으로

2. **사용자 중심 사고**
   - 팀원 = 사용자
   - 프로세스 개선 = UX 개선

3. **균형잡힌 시각**
   - 디자인-개발 갈등 중재
   - 양쪽 입장 이해

### 스크럼 마스터 인증 준비

```markdown
## CSM (Certified ScrumMaster) 준비

### 학습 자료
- Scrum Guide (공식 가이드)
- "Scrum: The Art of Doing Twice the Work in Half the Time"
- Mountain Goat Software 블로그

### 실습 방법
1. 팀내 임시 스크럼 마스터 역할
2. 회고 진행 연습
3. 퍼실리테이션 스킬 향상

### 예상 질문
Q: "스프린트 중간에 우선순위가 바뀐다면?"
A: 프로덕트 오너와 협의 후 다음 스프린트 반영

Q: "팀원이 스탠드업을 거부한다면?"
A: 1:1로 이유 파악, 심리적 안전감 조성
```

## 📊 스크럼 성과 측정 지표

### 디자인팀 KPI in 스크럼

```
측정 지표:
1. Design Velocity: 완료한 디자인 스토리 포인트
2. Handoff Timing: 개발 시작 전 디자인 완료율
3. Design Debt: 개선 필요 디자인 백로그
4. Rework Rate: 디자인 수정 요청률

월별 추적:
Sprint 1: Velocity 15, Handoff 60%, Rework 30%
Sprint 2: Velocity 18, Handoff 75%, Rework 25%
Sprint 3: Velocity 21, Handoff 85%, Rework 20%
Sprint 4: Velocity 24, Handoff 95%, Rework 15%

개선 포인트:
- Handoff 타이밍 개선으로 개발 효율 상승
- Rework 감소로 전체 생산성 향상
```

## 💡 핵심 메시지

> "스크럼은 단순한 프로세스가 아니라 '함께 성장하는 문화'입니다.
> 디자이너로서 스크럼을 제대로 이해하고 실천하면,
> 단순히 그림 그리는 사람이 아닌 '제품을 만드는 팀원'이 됩니다."

**기억하세요:**
- 완벽한 디자인 < 시간 내 전달되는 디자인
- 혼자 빛나는 것 < 팀이 함께 성공하는 것
- 프로세스는 도구일 뿐, 목적은 좋은 제품
