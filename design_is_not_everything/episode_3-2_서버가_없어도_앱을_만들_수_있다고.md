# Episode 3-2: 서버가 없어도 앱을 만들 수 있다고?

"서버가 없다고? 그럼 앱은 어떻게 동작하는거지?" 처음 서버리스(Serverless)라는 말을 들었을 때 든 생각이에요. 사실 서버가 진짜 없는 건 아니거든요. 그럼 대체 뭐가 다른 걸까요?

## 1. 전통적인 방식 vs 서버리스: 뭐가 다른가?

먼저 우리가 알던 전통적인 방식부터 살펴볼게요.

**🏢 전통적인 서버 운영 방식**
여러분이 카페를 운영한다고 생각해보세요. 하루 종일 카페를 열어두고, 직원들이 대기하고 있어야 하죠. 손님이 몰리든 안 몰리든 상관없이 말이에요.

- 24시간 서버 가동 (전기세 💸)
- 서버 관리 인력 필요 (인건비 💸)
- 갑작스런 트래픽 증가? 서버 다운 💥
- 사용자가 적어도 고정비용 발생

**☁️ 서버리스 방식**
이번에는 푸드트럭을 생각해보세요. 손님이 주문할 때만 트럭이 나타나서 음식을 만들어 주고 다시 사라져요. 효율적이죠?

- 요청이 있을 때만 작동 🔄
- 사용한 만큼만 반납 (요청 당 과금) 💰
- 자동 스케일링 (트래픽 증가에 자동 대응) 🚀
- 서버 관리 부담 없음 (클라우드에서 알아서) 🎈

**📊 비용 비교 (스타트업 기준)**
- 전통 서버: 월 $200-500 (사용자 없어도)
- 서버리스: 월 $5-50 (사용자 수에 비례)

이 선의 차이가 스타트업들이 서버리스를 선택하는 이유예요. 특히 초기에는 사용자가 적으니까 서버리스를 쓰면 비용이 거의 들지 않아요.

## 2. 서버리스의 핵심: 함수(Functions)

서버리스의 핵심은 **함수(Functions)**에요. 간단히 말하면, 하나의 작업만 수행하는 작은 코드 조각이라고 생각하면 돼요.

**🏨 전통적인 모놀리식 앱**
한 건물에 대형마트가 있다고 생각해보세요. 오직 하나의 거대한 코드덧어리가 모든 기능을 처리하죠.

- 로그인, 결제, 배송 모두 하나의 서버에서
- 하나의 서버가 다운되면 전체 서비스 중단
- 업데이트 시 전체 서비스 재시작 필요

**🎆 서버리스 마이크로서비스**
이번에는 상가길을 생각해보세요. 각각의 전문점들이 따로 있어요.

- 함수 1: 로그인 전담 (건에 대한 전문가)
- 함수 2: 결제 전담 (장난감에 대한 전문가)
- 함수 3: 배송 전담 (럩셔리에 대한 전문가)

하나가 문제가 생겨도 다른 것들은 정상 작동하고, 필요한 것만 업데이트할 수 있어요.

**💫 실제 서버리스 함수 예시**

이메일 전송 함수를 만든다고 해보세요:

- 누군가 회원가입을 하면 → 자동으로 함수 실행
- 함수가 반갑 이메일 전송 → 작업 완료 후 종료
- 비용: 회원가입 1건당 0.0001달러

만약 하루에 회원가입이 10건이면 0.001달러, 100건이면 0.01달러만 내면 돼요. 전통 서버라면 사용자가 0명이어도 월 최소 $50는 내야 하죠.

## 3. 서버리스로 앱 만들기: 실제 사례들

"그래서 실제로 어떤 앱들이 서버리스로 만들어지는데?" 생각보다 많은 서비스가 서버리스로 돌아가고 있어요.

**📱 인스타그램 스토리**
스토리에 사진을 올리면 24시간 후 자동으로 삭제되죠? 이것도 서버리스 함수예요!

- 사진 업로드 시 → 24시간 타이머 함수 설정
- 24시간 후 → 자동 삭제 함수 실행
- 전 세계 수억 사용자의 스토리를 이렇게 처리

**🚗 우버 라이드 매칭**
우버에서 타기 버튼을 눌렀을 때 가장 가까운 운전자를 찾는 기능도 서버리스예요.

- 타기 요청 → 위치 기반 매칭 함수 실행
- 가장 가까운 운전자 3명에게 알림 전송
- 매칭 성공 → 함수 종료, 실패 → 다시 시도

**📸 이미지 자동 압축**
패이스북에 사진을 올리면 자동으로 여러 크기로 만들어지죠? 이것도 서버리스!

- 사진 업로드 → 이미지 처리 함수 실행
- 자동으로 여러 해상도 버전 생성 (150x150, 300x300, 1200x1200)
- 섬네일, 타임라인, 앱에서 각각 다른 크기 사용

**📊 실제 서버리스 도입 효과**
- 넷플릭스: 서버리스 도입 후 운영비용 70% 절감
- 코카콜라: 전 세계 자판기 데이터 처리를 서버리스로
- 에어비앤비: 검색 및 예약 시스템의 90%가 서버리스

## 4. 서버리스를 선택할 때 vs 안 할 때

"서버리스가 이렇게 좋다면 모든 앱을 서버리스로 만들면 되는 거 아니야?" 아니에요. 서버리스도 만능은 아니에요.

**✅ 서버리스가 좋은 경우**

- **스타트업 초기 단계**: 사용자가 적고 비용 절약이 중요
- **불규칙한 트래픽**: 가끔 버즈가 되고 가끔 조용한 서비스
- **이벤트 기반 작업**: 사용자가 어떤 액션을 할 때만 동작
- **마이크로서비스 아키텍처**: 작은 기능들로 나누어 개발

**❌ 서버리스를 피해야 하는 경우**

- **실시간 처리**: 게임, 웹 소켓, 라이브 스트리밍 처럼 지연시간 0이 중요
- **대용량 데이터 처리**: 빅데이터 분석, 머신러닝 모델 학습
- **지속적인 고트래픽**: 항상 사용자가 많은 서비스 (페이스북, 인스타그램 수준)
- **레거시 시스템**: 이미 서버 인프라가 많이 구축된 대기업

**💰 비용 비교 예시**

하루 만 명의 사용자가 있는 서비스를 예로 들어보세요:

- **전통 서버**: 월 $2,000 (항상 가동 상태)
- **서버리스**: 월 $8,000 (요청당 과금)

이런 경우는 전통 서버가 더 저렴해요. 그래서 "사용자가 많을지 적을지"를 예측해서 선택하는 게 중요해요.
