# Episode 6-5: 100명 조직에서도 스크럼을 하는 방법
### 스케일드 스크럼의 예술: 오케스트라처럼 조화시키기

---

## 🎼 오프닝: 100명의 연주자가 하나의 곡을 연주하려면

베를린 필하모닉 오케스트라를 상상해보세요. 100명이 넘는 연주자들이 각각 다른 악기를 연주하지만, 지휘자의 지휘봉 아래 하나의 웅장한 교향곡을 만들어냅니다. 

우리 회사도 마찬가지예요. 처음엔 5명이었던 개발팀이 어느새 100명이 되었다면? "우리도 스크럼을 해야 하는데..."라고 고민하시겠죠. 혹시 "100명이 다 같이 데일리 스탠드업을 해야 하나?"라고 생각하고 계신가요?

**걱정 마세요. 오케스트라도 바이올린, 첼로, 플루트 섹션으로 나누어 연습하듯이, 스크럼도 규모에 맞게 확장할 수 있어요.**

---

## 🎻 Part 1: 파트 구성 - 팀을 나누는 법

### 오케스트라의 섹션 나누기

베토벤의 9번 교향곡을 연주할 때, 지휘자는 악기별로 섹션을 나눕니다:
- 현악기 섹션 (바이올린, 비올라, 첼로)
- 관악기 섹션 (플루트, 오보에, 호른)
- 타악기 섹션 (팀파니, 심벌즈)

### 스크럼 팀의 섹션 나누기

마찬가지로 100명 조직에서는 **기능별로 스크럼 팀을 구성**해야 합니다:

**🏢 전자상거래 회사의 예시:**
- **사용자 경험 팀** (로그인, 회원가입, 마이페이지)
- **상품 관리 팀** (카탈로그, 검색, 추천)
- **주문 결제 팀** (장바구니, 결제, 주문관리)
- **물류 배송 팀** (재고, 배송, 반품)

각 팀은 **5-9명**으로 구성하고, 각각의 **제품 책임자(Product Owner)**와 **스크럼 마스터(Scrum Master)**를 둡니다.

### 🎯 TMI Box: 왜 5-9명일까요?
아마존의 창립자 제프 베조스가 만든 "Two Pizza Rule"을 아시나요? 피자 두 판으로 배를 채울 수 있는 크기의 팀이 가장 효율적이라는 법칙이에요. 사람이 많을수록 의사소통 복잡도가 기하급수적으로 증가하거든요.

---

## 🎯 Part 2: 지휘자의 역할 - 스크럼 마스터들의 스크럼

### 오케스트라의 지휘자

지휘자는 각 섹션의 수석 연주자들과 리허설 전 미팅을 갖습니다. "오늘 2악장에서 현악기가 좀 더 부드럽게, 3악장에서 관악기가 조금 더 힘 있게 연주해주세요."

### 스크럼 오브 스크럼즈(Scrum of Scrums)

**각 팀의 스크럼 마스터들이 모여 더 큰 스크럼을 진행**하는 것이에요.

**📅 매일 아침 9:00 - SoS 미팅 (15분)**
- 각 팀 대표 1명씩 참석
- 어제 우리 팀이 완료한 것
- 오늘 우리 팀이 할 것
- 다른 팀과의 의존성 이슈

**🗣️ 실제 대화 예시:**
```
사용자경험팀: "로그인 API 완료했어요. 내일 상품팀에서 테스트 가능합니다."
상품팀: "검색 기능에서 사용자 권한 체크가 필요한데, 언제쯤 가능할까요?"
사용자경험팀: "오늘 오후 배포 예정이니, 내일부터 연동 테스트 하시면 됩니다."
```

### 🎼 TMI Box: 실리콘밸리의 비밀
Spotify는 이런 방식으로 1000명이 넘는 개발자들을 관리해요. "Squad(팀) - Tribe(부족) - Guild(길드)" 구조로, 마치 바이킹 시대의 조직 구조를 차용했죠. 각 Squad는 스크럼 팀처럼 운영되고, 여러 Squad가 모여 Tribe를 이룹니다.

## 🎪 Part 3: 리허설 - PI Planning의 중요성

### 오케스트라의 총 리허설

시즌 첫 공연 전, 모든 연주자들이 모여 전체 곡을 처음부터 끝까지 연주해보는 시간이에요. 각 섹션이 어떻게 조화를 이루는지, 타이밍은 맞는지 확인하죠.

### PI Planning (Program Increment Planning)

**분기별로 모든 팀이 모여 3개월간의 계획을 수립하는 시간**이에요. SAFe(Scaled Agile Framework)의 핵심 이벤트죠.

**📋 PI Planning 진행 순서:**
1. **비전 공유** (CEO/CTO 발표)
2. **각 팀별 계획 발표** (30분씩)
3. **의존성 매핑** (팀 간 연결점 확인)
4. **리스크 식별** (잠재적 문제점)
5. **최종 커밋** (각 팀의 약속)

**🏢 실제 사례 - 핀테크 회사:**
```
Q1 목표: 새로운 대출 상품 출시

인증팀: "본인인증 강화 (2주)"
심사팀: "신용평가 알고리즘 개선 (6주)"
상품팀: "대출 상품 UI/UX 개발 (4주)"
결제팀: "자동이체 시스템 연동 (3주)"

의존성: 인증팀 → 심사팀 → 상품팀 → 결제팀
리스크: 외부 신용평가 API 변경 가능성
```

## 🎭 Part 4: 공연 - 실제 스프린트 운영

### 오케스트라의 공연

지휘자는 각 악장마다 다른 섹션에게 신호를 보냅니다. 때로는 현악기만, 때로는 관악기와 타악기가 함께, 클라이맥스에서는 모든 섹션이 하나가 되어 연주하죠.

### 스케일드 스크럼의 실제 운영

**🔄 2주 스프린트 사이클:**

**Week 1:**
- 각 팀: 개별 개발 작업
- SoS 미팅: 매일 진행상황 공유
- 문제 발생시: 즉시 관련 팀들과 협의

**Week 2:**
- 통합 테스트: 팀 간 협업 작업
- 데모 준비: 각 팀별 + 전체 통합 버전
- 스프린트 리뷰: 전체 참석 (각 팀 5분씩 발표)

**📊 실제 대시보드 예시:**
```
전체 진행률: 73%

팀별 현황:
✅ 사용자경험팀: 95% (예정보다 1일 빠름)
⚠️ 상품팀: 60% (API 응답 속도 이슈)
✅ 결제팀: 80% (정상 진행)
🔴 배송팀: 40% (외부 업체 연동 지연)
```

### 🚨 위기 상황 대응

오케스트라에서 첼로 섹션의 현이 끊어졌을 때를 상상해보세요. 지휘자는 즉시 다른 섹션으로 멜로디를 넘기고, 첼로는 조용히 현을 교체하죠.

**스크럼에서도 마찬가지:**
- 핵심 기능에 블로커 발생 → 다른 팀에서 대체 방안 검토
- 긴급 버그 → 해당 팀의 스프린트 목표 조정
- 외부 의존성 지연 → 전체 일정 재조정

## 🎵 Part 5: 프레임워크 선택 - SAFe vs LeSS

### SAFe (Scaled Agile Framework)

마치 **클래식 오케스트라**처럼 체계적이고 정형화된 구조예요.

**👥 적합한 조직:**
- 대기업 (100명 이상)
- 규제가 많은 업계 (금융, 의료)
- 명확한 계층 구조 선호

**🏗️ 구조:**
- Portfolio → Program → Team 레벨
- 다양한 역할과 이벤트가 정의됨
- 상세한 가이드라인 제공

### LeSS (Large-Scale Scrum)

**재즈 앙상블**처럼 자유롭고 유연한 접근법이에요.

**👥 적합한 조직:**
- 스타트업/중견기업 (50-500명)
- 빠른 변화가 필요한 업계
- 자율성과 창의성 중시

**🎭 특징:**
- 기본 스크럼 원칙 유지
- 최소한의 추가 규칙
- 팀들의 자율적 조정 중시

## 💡 Try This at Work: 우리 팀 스케일드 스크럼 체크리스트

**📋 1단계: 현재 상황 파악**
- [ ] 우리 조직의 개발자는 몇 명인가요?
- [ ] 현재 몇 개의 제품/서비스를 운영하나요?
- [ ] 팀 간 의존성이 얼마나 복잡한가요?

**📋 2단계: 팀 구성 검토**
- [ ] 각 팀이 5-9명으로 구성되어 있나요?
- [ ] 각 팀이 독립적으로 배포할 수 있나요?
- [ ] 팀별로 명확한 책임 영역이 있나요?

**📋 3단계: 조율 메커니즘 점검**
- [ ] 팀 간 의사소통 채널이 있나요?
- [ ] 의존성을 추적하는 시스템이 있나요?
- [ ] 전체 목표와 우선순위가 공유되나요?

## 🎼 마무리: 조화의 완성

오케스트라에서 가장 아름다운 순간은 100명의 연주자가 하나의 목소리로 연주할 때예요. 각자의 파트를 완벽히 연주하면서도, 전체의 조화를 놓치지 않는 거죠.

스케일드 스크럼도 마찬가지예요. **각 팀이 자신의 목표를 달성하면서도, 회사 전체의 비전을 향해 나아가는 것**. 그게 바로 스케일드 스크럼의 예술입니다.

다음 주 월요일, 개발팀 전체 회의에서 이렇게 제안해보세요:
"우리도 오케스트라처럼 각자의 파트를 연주하면서 하나의 곡을 완성해볼까요?"

---

**💬 Common Questions:**

**Q: 100명이 다 같이 회의하면 안 되나요?**
A: 100명이 한 번에 회의하면 소통보다는 발표가 되어버려요. 오케스트라도 섹션별로 연습하고, 필요할 때만 전체가 모이잖아요.

**Q: 팀별 우선순위가 다르면 어떻게 하나요?**
A: PI Planning에서 전체 비전을 먼저 정렬하고, 각 팀의 목표를 회사 목표와 연결해야 해요. 지휘자가 각 섹션에게 역할을 주는 것처럼요.

**Q: 의존성 관리가 너무 복잡해요.**
A: 처음엔 복잡하지만, 시각화 도구(Dependency Board)를 사용하고 정기적으로 점검하면 자연스러워져요. 마치 악보를 보고 연주하는 것처럼요.

## 📝 Feedback

### 현재 잘된 점
- 핵심 개념을 명확하게 설명
- 실무 예시와 사례 제공
- 단계별 가이드 포함

### 개선이 필요한 부분

1. **디자이너 관점 추가**
   - 디자이너가 이해해야 할 핵심 포인트
   - 디자인 작업과의 연관성
   - 협업 시 주의사항

2. **실무 대화 예시**
   - 팀 미팅에서의 실제 대화
   - 갈등 상황과 해결 방법
   - 효과적인 커뮤니케이션 팁

3. **실패 사례와 교훈**
   - 실제 프로젝트 실패 경험
   - 피해야 할 함정들
   - 배운 교훈과 개선 방안

4. **실용적 도구/템플릿**
   - 바로 사용 가능한 템플릿
   - 추천 도구와 서비스
   - 체크리스트와 가이드라인

5. **시각적 요소 강화**
   - 프로세스 다이어그램
   - 비교 표와 차트
   - 인포그래픽 추가 제안
