# Episode 2-6. 와이어프레임이 못생기면 개발이 어려워지는 이유

개발자에게 와이어프레임을 보여주면서 "이거 개발할 수 있어요?" 하고 물었는데, 개발자가 한숨을 쉬며 "음... 이거 좀 복잡할 것 같은데요..."라고 답했던 경험, 있으신가요? 분명 단순해 보이는 화면인데 왜 개발자는 어려워할까요?

사실 와이어프레임의 '못생김'은 시각적 디자인의 문제가 아니에요. 진짜 문제는 구조적 설계에 있죠. 개발자가 보기에 '못생긴' 와이어프레임은 기술적으로 구현하기 어려운 구조를 가지고 있거든요.

## 1. 개발자의 눈으로 본 '못생긴' 와이어프레임

디자이너와 개발자는 같은 화면을 보지만 전혀 다른 것을 봅니다. 디자이너는 사용자 경험과 시각적 흐름을 보지만, 개발자는 데이터 구조와 로직 흐름을 보거든요.

예를 들어, 이런 간단해 보이는 화면을 생각해보세요:

해당 제품의 '추천 리스트'가 표시되는 영역
- 5개의 상품 카드가 가로로 나열
- '더보기' 버튼이 오른쪽 끝에 위치

디자이너 눈에는 단순한 리스트로 보이지만, 개발자의 머릿속에는 이런 질문들이 손스바닥 침니다:

**🤔 개발자의 내적 독백**

- '추천 리스트'를 어디서 가져오지? API 따로 만들어야 하나?
- 5개를 고정으로 보여주나? 데이터가 없으면 빈 카드를 보여주나?
- 추천 알고리즘은 어떻게 동작하지? 사용자별로 달라지나?
- '더보기' 버튼을 누르면 어디로 가지? 전체 목록인가, 다음 5개를 더 로드하나?
- 각 상품 카드에는 어떤 정보를 표시해야 하지? 이미지, 가격, 별점, 좋아요 버튼?
- 모바일에서는 스와이프로 동작하나? PC에서는 버튼으로 동작하나?

이처럼 간단해 보이는 한 줄의 리스트 안에 실제로는 수십 가지의 결정 사항이 숨어있어요. 개발자가 '이거 복잡할 것 같은데요'라고 말하는 이유는 이 질문들에 대한 답이 뒤따르지 않기 때문이죠.

특히 쫀치는 것은 대부분의 디자이너들이 이런 질문들이 있다는 사실 자체를 모른다는 거예요. 그래서 와이어프레임에 'UI 요소만' 디자인하고 '데이터와 로직'은 빠뜨리게 되죠.

## 2. '좋은' 와이어프레임이 개발을 빠르게 만드는 이유

그러면 개발자가 보기에 '좋은' 와이어프레임은 어떤 모습일까요? 단순히 예쁜 것이 아니라, 개발자가 바로 코드를 쓸 수 있도록 충분한 정보를 담고 있는 와이어프레임이에요.

저희 팀의 예를 들어보죠. 이전에는 이런 식으로 상황이 전개됐어요:

디자이너: '이 페이지에 관련 상품 리스트 넣어주세요!'
개발자: '그... 관련 상품이 뭐죠? 어떻게 추출하나요?'
디자이너: '음... 비슷한 거요!'
개발자: (속마음에 질문만 50개)

하지만 지금은 완전히 달라졌어요. 디자이너가 와이어프레임에 이런 주석을 달아주거든요:

**📝 개선된 와이어프레임 예시**

[UI 요소: 가로 스크롤 리스트]

- 데이터 소스: GET /api/products/{product_id}/related
- 로직: 현재 상품과 같은 카테고리 + 비슷한 가격대 + 높은 별점 순
- 개수: 최대 10개, 최소 3개 (데이터 부족 시 빈 상태 메시지)
- 모바일: 스와이프, PC: 좌우 화살표 버튼
- 더보기 동작: 카테고리 페이지로 이동 (category_id 파라미터 전달)

이런 정보가 있으면 개발자는 즉시 코드를 쓸 수 있어요. '아, 이거라면 상품 데이터베이스에서 category_id로 필터링하고, price 범위로 정렬하면 되겠네'라고 바로 알 수 있거든요.

이전에는 '관련 상품'이란 모호한 설명 때문에 PM과 먰번의 미팅을 거쳐야 했었는데, 이제는 와이어프레임만 보고도 바로 개발에 들어갈 수 있게 됐어요.

**📈 개발 속도 비교**

- 이전: 와이어프레임 확인 → 질문 → 미팅 → 상세 설명 → 다시 미팅 → 개발 시작 (3-4일)
- 지금: 와이어프레임 확인 → 바로 개발 시작 (30분)

## 3. 개발자를 위한 와이어프레임 자체 체크리스트

그러면 어떻게 하면 개발자가 보기에 '좋은' 와이어프레임을 만들 수 있을까요? 저희 팀이 사용하는 체크리스트를 공유해드릴게요.

**✅ 데이터 역영 체크리스트**

- [ ] 데이터 소스를 명시했나? (API 엔드포인트, DB 테이블 등)
- [ ] 데이터 정렬/필터링 로직을 설명했나?
- [ ] 데이터가 비어있거나 로딩 중일 때의 상태를 정의했나?
- [ ] 데이터 개수의 최대/최소값을 정했나?

**✅ 사용자 인터에션 체크리스트**

- [ ] 각 버튼/링크의 동작을 구체적으로 설명했나?
- [ ] 모바일/PC에서의 인터에션 차이를 내킬나?
- [ ] 페이지 이동 시 전달할 파라미터를 명시했나?

**✅ 예외 상황 체크리스트**

- [ ] 에러 발생 시 사용자에게 보여줄 메시지를 정했나?
- [ ] 버튼 비활성화 상태를 고려했나? (로딩 중, 권한 없음 등)
- [ ] 네트워크 연결 실패 시 대응을 정의했나?

이 체크리스트만 지켜도 개발 시간이 히조로 달라져요. 디자이너가 이러한 정보들을 미리 고민해주면, 개발자는 '아 이거 복잡할 것 같은데요' 대신 '아 이거라면 금요일까지 완성 가능할 것 같아요'라고 대답할 수 있게 돼요.

## 4. 실제 사례: 버그 리포트와 수정 요청의 90%가 줄어든 이유

저희 팀이 이 방식을 도입한 후 가장 드라마틱한 변화는 개발 완료 후에 일어났어요. QA 단계에서 나오는 버그 리포트와 수정 요청이 90% 가까이 줄어들었거든요.

이전에는 이런 식으로 일이 진행됐어요:

개발 완료 후:
"이 리스트에서 데이터가 5개만 나와요. 10개로 늘려주세요"
"더보기 버튼을 누르니까 이상한 페이지로 가요"
"이 상품들이 왜 추천되는지 모르겠어요"

이런 요청들은 모두 '버그'가 아니라 '사양' 변경이었어요. 그런데 개발자 입장에서는 이미 구현이 완료된 기능을 다시 수정해야 하니까 가장 비효율적인 작업이죠.

하지만 지금은 완전히 달라졌어요. 와이어프레임에 모든 사양이 정의되어 있기 때문에, 개발자는 그 사양대로 정확히 구현해요. 그리고 QA 단계에서는 '와이어프레임대로 구현됐는지'만 확인하면 되죠.

**📉 비효율 감소 지표**

- 버그 리포트 수: 주평균 25개 → 3개 (88% 감소)
- 수정 요청 수: 주평균 15개 → 1-2개 (90% 감소)
- 개발 완료 후 수정 시간: 평균 3일 → 0.5일 (83% 감소)

가장 중요한 것은 팀 분위기의 변화였어요. 이전에는 개발 완료 후에 '이거 왜 이렇게 돼있어요?' 식의 대화가 잦았는데, 지금은 '와이어프레임대로 잘 만들어졌네요!' 같은 긍정적인 피드백이 대부분이에요.

**📝 핵심은 결국 소통이에요**

와이어프레임이 '못생기면' 개발이 어려워지는 진짜 이유는 기술적 복잡성이 아니라 소통의 부재였어요. 디자이너와 개발자가 같은 언어로 대화할 수 있도록 하는 것, 그것이 바로 '좋은' 와이어프레임의 핵심이죠.

결국 좋은 와이어프레임은 단순히 '예쁜' 것이 아니라, 개발자가 바로 코드를 쓸 수 있도록 충분한 정보를 담고 있는 것이에요. 처음에는 이런 주석들을 달는 게 번거롭게 느껴졌지만, 막상 해보니 개발 속도도 빨라지고 팀 분위기도 좋아지더라고요.

다른 팀에서도 비슷한 경험을 하셨다면, 어떤 방식을 사용하고 있는지 궁금해요. 다음에는 '기획서를 디자인하는 법'에 대해서 이야기해볼게요!
