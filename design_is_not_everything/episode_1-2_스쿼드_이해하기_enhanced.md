# 에피소드 1-2: 스쿼드(Squad)가 뭐예요? - 작은 팀이 큰 일을 하는 비밀

## 🎯 이 글을 읽으면 얻게 되는 것
- 스쿼드 조직 구조의 핵심 이해
- 전통적 팀과 스쿼드의 차이점
- 스쿼드에서 각 역할의 책임과 협업 방식
- 성공적인 스쿼드 운영 노하우

## 🏃 스쿼드, 특공대처럼 일하는 팀

### 스쿼드의 탄생 배경
```
📖 스쿼드의 역사

2012년 Spotify:
"300명이 한 제품을 만들면 너무 느려요"
→ 8-10명 단위 자율 팀으로 분할
→ 각 팀이 독립적으로 의사결정
→ 결과: 개발 속도 3배 향상

이후 확산:
• Netflix: 마이크로서비스 팀
• Amazon: Two-Pizza Teams
• 카카오: 크루(Crew)
• 토스: 사일로(Silo)
```

### 스쿼드 vs 전통적 팀
```
🏢 조직 구조 비교
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
구분        | 전통적 팀      | 스쿼드
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
인원       | 20-50명       | 5-10명
의사결정    | 위계적        | 자율적
책임       | 기능별        | 제품/서비스별
협업       | 부서 간 요청   | 팀 내 즉시
속도       | 2-3주         | 2-3일
실패       | 큰 리스크      | 빠른 학습
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 각 역할의 실제 하루
```

15:00 - 피그마 프로토타입 제작

09:00 - 코드 리뷰
```

## 🔄 스쿼드 간 협업

### 디펜던시 관리
```
🔗 스쿼드 간 의존성 해결

상황: 결제 스쿼드가 회원 API 필요

나쁜 예 ❌:
결제팀 → 팀장 → 팀장 → 회원팀
(2주 소요)

좋은 예 ✅:
결제 개발자 ↔️ 회원 개발자
(2일 소요)

협업 도구:
• Slack 채널: #squad-결제-회원
• API 문서: 자동 생성/공유
• 정기 Sync: 주 1회 15분
```

### 길드(Guild) 시스템
```
🏛️ 전문성 공유를 위한 길드

길드 = 같은 전문 분야 사람들의 모임

프론트엔드 길드:
• 모든 스쿼드의 프론트엔드 개발자
• 월 1회 기술 공유
• 공통 컴포넌트 개발
• 베스트 프랙티스 정립

디자인 길드:
• 모든 스쿼드의 디자이너
• 디자인 시스템 관리
• 사용성 테스트 방법론
• 트렌드 공유

데이터 길드:
• 분석가 + 관심 있는 개발자
• 데이터 파이프라인 구축
• 대시보드 표준화
• 인사이트 공유
```

## 📊 스쿼드 성과 측정

### OKR (Objectives and Key Results)
```
🎯 스쿼드 OKR 예시

Q1 온보딩 스쿼드 OKR:

Objective: 
"신규 사용자가 우리 서비스를 사랑하게 만들기"

Key Results:
1. 가입 완료율 60% → 80%
2. 첫 구매까지 시간 7일 → 3일
3. 튜토리얼 완료율 30% → 70%
4. NPS 점수 20 → 50

측정 방법:
• 주간 대시보드 체크
• 격주 사용자 인터뷰
• 월간 회고 미팅
• 분기 OKR 리뷰
```

### 속도 vs 품질 균형
```
⚖️ 스쿼드의 딜레마

속도 중시 스쿼드:
• 빠른 실험과 학습
• MVP 우선 출시
• 버그 허용도 높음
• 스타트업 초기

품질 중시 스쿼드:
• 안정성 최우선
• 충분한 테스트
• 버그 제로 목표
• 금융, 의료 서비스

균형 잡기:
1. 핵심 기능 = 품질
2. 실험 기능 = 속도
3. 기술 부채 = 20% 시간 할당
4. 회고를 통한 지속 개선
```

```
👥 리더 없는 리더십

전통적 팀장 ❌

• 스프린트마다 스크럼 마스터 교체

```

### 심리적 안전감
```
🛡️ 실패해도 안전한 환경

심리적 안전감 체크리스트:
□ 실수를 공유할 수 있다
□ 모르는 것을 물어볼 수 있다
□ 다른 의견을 제시할 수 있다
□ 리스크를 감수할 수 있다
□ 도움을 요청할 수 있다

구축 방법:
1. 실패 공유 세션
2. "멍청한 질문" 시간
3. 아이디어 브레인스토밍
4. 1:1 정기 면담
5. 팀 회식/워크샵
```

## 💡 스쿼드 도입 시 주의사항

### 흔한 실패 패턴
```
⚠️ 스쿼드 도입 실패 사례

1. 이름만 스쿼드
   • 기존 팀에 이름만 변경
   • 자율성 없이 지시만
   • 해결책: 진정한 권한 위임

2. 너무 많은 스쿼드
   • 2-3명 미니 스쿼드
   • 전문성 부족
   • 해결책: 적정 규모 유지

3. 고립된 스쿼드
   • 다른 팀과 소통 없음
   • 중복 개발
   • 해결책: 길드/챕터 운영

4. 불명확한 미션
   • 목표가 모호함
   • 성과 측정 불가
   • 해결책: 명확한 OKR
```

## 💡 핵심 정리

> "최고의 제품은 작고 자율적인 팀에서 나온다"

1. **작은 팀의 힘**: 5-10명이 최적, 의사결정 빠름
2. **자율과 책임**: 스스로 결정하고 책임지는 문화
3. **미션 중심**: 명확한 목표와 측정 가능한 성과
4. **지속적 개선**: 정기 회고와 실험 문화
5. **협업의 균형**: 독립성과 협력의 조화

스쿼드는 단순한 조직 구조가 아니라 일하는 방식의 혁신입니다. 작은 팀이 큰 임팩트를 만들 수 있는 비결이 바로 여기에 있습니다!
