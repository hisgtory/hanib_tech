# 리팩터링 - hanib_tech 콘텐츠 시리즈

> *"이번 스프린트 동안은 리팩터링을 좀 할게요"*

## 🎯 콘텐츠 개요

이 시리즈는 개발자라면 누구나 들어봤지만 정확히 언제, 왜 해야 하는지 애매한 '리팩터링'에 대해 실무적 관점에서 설명한 한입씨 테크의 교육 콘텐츠입니다. 총 8개의 카드로 구성되어 있으며, 현실적인 대화로 시작해서 실무에서 적용할 수 있는 구체적인 가이드라인을 제시합니다.

## 📚 콘텐츠 구성 (8개 카드)

### 1️⃣ 인트로: 현실적인 개발팀 대화
**대화 시나리오**:
- "네! 근데 저희 그런 기능이 있었나요?"
- "..? 리팩터링 한다니까요..?"
- ".. 네??"

**스토리텔링 특징**:
- 검은 배경에 말풍선으로 실제 회사 대화 재현
- 리팩터링에 대한 인식 차이를 유머러스하게 표현
- 독자의 공감대 형성과 호기심 유발

**핵심 메시지**: 
- 리팩터링이 뭔지 모르는 것은 당연함
- 개발팀 내에서도 소통이 어려운 주제

---

### 2️⃣ 리팩터링의 정의와 시각적 설명
**메인 메시지**: "리팩터링은 코드를 개선하는 작업이에요"

**핵심 내용**:
- 그동안 누적된 **"레거시"를 정신하는 작업이죠.**
- **기능을 변경하지 않고** 코드의 **내부 구조를 개선**하는 것이 목표예요

**시각적 스토리텔링**:
```
[Before - 복잡하고 어수선한 블록들] → [After - 깔끔하게 정리된 블록들]
개발자들이 정리 작업을 하는 모습
```

**교육적 포인트**:
- 추상적 개념을 시각적으로 구체화
- "정리 작업"이라는 친근한 비유 사용
- 기능 변경 없이 구조만 개선한다는 핵심 강조

**추가 설명**:
- "이전 사용 방식을 지원하지 않는 구버전 라이브러리를 최신화하는 것도 리팩터링이라고 할 수 있어요."

---

### 3️⃣ 리팩터링이 필요한 이유
**메인 메시지**: "Q. 리팩터링 왜 해야 하는 거예요?"

**핵심 내용**:
- 누적된 레거시는 개발자들의 **생산성***과 서비스의 **안정성***을 떨어뜨려요
- 코드의 퀄리티를 다시 일정 수준까지 올려 생산성과 안정성을 확보하기 위해 리팩터링이 필요해요

**구체적 예시**:
#### *생산성이 떨어진 예시
- 하나를 바꿔먼 다 같이 바뀔 것이라 생각했던 컴포넌트들이 부분만 변경되는 현상이 발견돼요

#### *안정성이 떨어진 예시  
- 하나를 고치면 또 다른 버그가 생기는 등 버그가 쉽게 생기는 확률이 되어요

**시각적 강조**:
- 노란색 박스로 핵심 키워드 강조
- 구체적인 실무 상황 예시 제공

---

### 4️⃣ 리팩터링 시기에 대한 현실적 조언
**메인 메시지**: "Q. 신규 기능 개발할 때마다 조금씩 개선하면 안되는 건가요?"

**핵심 답변**: "네! 맞아요!"

**상세 설명**:
- 보통 개발자들은 **코드를 수정할 때마다** 아쉬운 부분들을 **조금씩 같이 수정**해요
- **하지만** 수정해야하는 규모가 크면 시간도 오래 걸리고 리팩터링 후 별도 테스트가 필요해서 이런 경우에는 일정을 따로 할애해서 작업해요

**실무적 통찰**:
- 소규모 개선과 대규모 리팩터링의 구분
- 프로젝트 스케줄링과의 연관성
- 테스트의 중요성 언급

---

### 5️⃣ 대규모 리팩터링 발생 상황
**대화 시나리오**:
- "아 잠, 대규모 공사이다보니 리팩터링 후 처음보는 버그들이 발견될 수 있어요."
- 고양이: "버그가 수정되는게 아니라 더 생길 수도 있다고요?"

**현실적 답변**:
- "네, 그렇기 때문에 QA를 꼭 진행해야 하고, 여유가 있는 팀일 경우 테스트 코드를 작성해 본 다면 리팩터링 전후의 안정성 문제를 해결할 수 있어요."

**중요한 교육 포인트**:
- 리팩터링의 위험성을 솔직히 인정
- QA의 필수성 강조
- 테스트 코드의 역할과 가치 설명
- 팀 상황에 따른 현실적 접근법 제시

---

### 6️⃣ 리팩터링 시기 결정 가이드라인
**메인 메시지**: "Q. 리팩터링은 언제 해야하고 얼마나 걸리나요?"

**핵심 가이드라인**:
1. **버그가 많거나 개발 속도가 느려진 것 같다면**
2. **최근에 신규 기능을 급하게 출시했다면**

**결론**: "리팩터링이 필요한 시점일지도 몰라요!"

**실무적 조언**:
- 리팩터링은 여러 요인에 따라 기간이 달라져요
- 작은 프로젝트의 경우 한 주에서 **n주** 단위로 수행되고
- 대규모 프로젝트의 경우 **분기** 단위로 진행되기도 해요

**현실적 관점**:
- 정확한 기간 예측의 어려움 인정
- 프로젝트 규모별 현실적인 시간 프레임 제시
- 상황에 따른 유연한 접근 필요성 강조

---

### 7️⃣ 개발자의 자세와 팀 문화
**대화 상황**:
- 고양이: "개발자분들이 리팩터링 하는 동안 다른 업무를 먼저 진행하고 있으면 되는건가요?"

**현실적 답변**:
- "네! 리팩터링 기간동안 다른 개발 작업은 활동되기 때문에 디자이너나 PM은 다음 스프린트나 OKR, 유저 리서치 등등 각자의 업무를 하고 있으면 돼요."

**팀워크 관점**:
- 리팩터링이 개발팀만의 일이 아님을 명확히
- 다른 직군과의 협업 방식 제시
- 각자 역할에 집중하는 것의 중요성

---

### 8️⃣ 마무리: 격려와 실천 독려
**메인 메시지**: "우리가 방을 몰아서 치우듯 코드도 정리하는 시간이 필요해요"

**핵심 메시지**:
- 당장의 성과가 부족해보여도 **장기적인 관점에서 조직을 위해 꼭 필요**해요
- **동료가 정확한 동기와 목표를 가지고 리팩터링 기간을 제안한다면 적극 지지해주세요!**

**감정적 호소**:
- 일상적 비유("방 정리")로 친근함 조성
- 장기적 관점의 중요성 강조
- 팀원 간 지지와 이해의 필요성 역설

**행동 유도**:
- 구체적인 행동 지침 제공
- 개인이 아닌 팀 전체의 관점에서 접근
- 적극적 지지의 중요성 강조

## 🎨 콘텐츠 스타일 분석

### 교육적 접근법의 특징

#### 1. 현실 기반 스토리텔링
- 실제 개발팀에서 벌어질 수 있는 대화로 시작
- 비개발자도 이해할 수 있는 친근한 비유 활용
- 이상론보다는 현실적 제약 조건 고려

#### 2. 단계적 이해 구조
```
현실적 혼란 → 정의 명확화 → 필요성 설명 → 시기 판단 → 위험 인식 → 실무 가이드 → 팀 문화 → 실천 독려
```

#### 3. 균형잡힌 관점 제시
- 리팩터링의 장점뿐만 아니라 위험성도 솔직히 언급
- 이상적 상황과 현실적 제약 모두 고려
- 완벽한 해답보다는 상황별 가이드라인 제공

### 시각적 디자인 특징

#### 색상 전략
- **검은색 배경**: 진지하고 전문적인 주제임을 암시
- **노란색 배경**: 밝고 친근한 설명 공간
- **회색 배경**: 중립적이고 차분한 정보 전달

#### 캐릭터 활용
- **고양이 캐릭터**: 궁금해하는 비개발자 대변
- **개발자 캐릭터**: 전문적 지식 전달자
- **말풍선**: 자연스러운 대화 분위기 조성

### 대상 독자 고려

#### 다층적 접근
1. **개발자**: 실무적 가이드라인과 기술적 정확성
2. **비개발자**: 리팩터링의 필요성과 가치 이해
3. **팀 리더**: 프로젝트 관리와 팀 문화 관점

#### 소통 전략
- 전문 용어 사용 후 즉시 쉬운 설명 제공
- 구체적 예시를 통한 추상적 개념 구체화
- 질문-답변 형식으로 독자 관점 반영

## 🚀 실무 활용 가치

### 팀 내 소통 도구
- 리팩터링 필요성 설득 자료로 활용
- 프로젝트 계획 수립 시 참고 가이드
- 팀 문화 개선을 위한 토론 소재

### 교육 콘텐츠 확장
- 구체적 리팩터링 기법 소개
- 테스트 코드 작성 방법론
- 레거시 코드 관리 전략

### 조직 문화 개선
- 장기적 관점의 중요성 인식 제고
- 개발팀과 다른 직군 간 이해 증진
- 기술 부채 관리의 체계적 접근

## 🎯 핵심 메시지 분석

### 메인 메시지
**"리팩터링은 선택이 아니라 필수다. 하지만 언제, 어떻게 할지는 상황에 따라 달라진다."**

### 서브 메시지
1. **현실 인정**: 완벽한 타이밍은 없다
2. **균형 감각**: 기능 개발과 코드 품질의 균형
3. **팀워크**: 혼자가 아닌 팀 전체의 협력이 필요
4. **장기적 관점**: 당장의 성과보다는 지속가능성

### 감정적 메시지
- **공감**: "우리도 이런 고민을 한다"
- **안정감**: "위험을 알고 대비하면 괜찮다"
- **동기부여**: "함께 더 나은 코드를 만들어가자"

## 📊 교육적 효과

- 기술 부채의 위험성에 대한 경각심 제고
- 코드 품질이 팀 전체에 미치는 영향 이해

### 실무 적용 능력 향상
- 리팩터링 시기 판단 기준 습득
- 팀 내 소통 방법 개선
- 위험 관리 및 대응 방안 학습

### 팀 문화 개선
- 상호 이해와 지지 문화 조성
- 장기적 관점의 의사결정 문화 정착
- 품질에 대한 공동 책임 의식 향상
